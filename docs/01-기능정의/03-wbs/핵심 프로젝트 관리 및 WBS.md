# 핵심 프로젝트 관리 및 WBS (PM Core & WBS)

ZeroPM 프로젝트는 MS Project와 같은 전문적인 프로젝트 관리 기능을 웹 환경에서 구현하며, 특히 **Zero-Cost** 인프라를 활용하여 이 목표를 달성하는 것을 핵심으로 합니다. 따라서 핵심 PM 기능들은 클라이언트 측의 자체 개발된 알고리즘 엔진과 PostgreSQL 기반의 Supabase 백엔드 인프라를 중심으로 설계되었습니다.

---

## 핵심 프로젝트 관리 및 WBS (PM Core & WBS) 상세 기능 분석

WBS (Work Breakdown Structure, 작업 분류 체계)는 프로젝트의 범위를 관리 가능한 단위로 분해하는 가장 기초적인 기능이며, PM Core 기능은 이 WBS를 기반으로 프로젝트의 계획, 추적, 비교를 가능하게 하는 요소들로 구성됩니다.

### I. WBS 구조 및 계층 관리

WBS 기능은 Task를 논리적인 계층 구조로 관리하고, 상위 작업에 하위 작업의 정보를 집계하는 롤업(Roll-up) 기능을 포함합니다.

#### 1. 무한 계층형 WBS 지원
*   **기능:** 사용자는 Task를 무한히 하위로 들여쓰기(Indent)하여 **무한 계층 구조(Infinite Hierarchy)**를 만들 수 있어야 합니다.
*   **기술적 구현 (ZeroPM):** PostgreSQL 데이터베이스에서 Task의 계층 구조는 **인접 리스트(Adjacency List) 모델**을 사용하여 구현됩니다. 이는 각 Task 테이블 행에 `parent_id` 필드를 두어 부모 Task를 참조하게 하는 방식입니다.
*   **쿼리 전략:** 인접 리스트 모델은 Task를 이동(쓰기 작업)할 때 쓰기 성능이 우수하다는 장점이 있지만, 전체 트리를 조회하거나 계층 구조를 계산할 때는 PostgreSQL의 **재귀 CTE(Recursive CTE)** 구문을 활용해야 합니다.
*   **순서 관리:** 같은 `parent_id` 내의 Task 순서를 보장하기 위해 `sort_order` 필드가 사용됩니다.
*   **WBS 코드:** 구조가 변경될 때마다 1, 1.1, 1.1.1 등 계층 코드가 자동으로 재계산되어야 합니다.

#### 2. 요약 작업 (Summary Tasks) 및 자동 롤업 (Roll-up)
*   **요약 작업 정의:** 하위 Task(Subtasks)가 생성되면 상위 Task는 자동으로 **요약 작업(Summary Task)**으로 변환됩니다.
*   **자동 집계 (Roll-up):** 요약 작업의 기간(Duration), 시작일, 종료일, 완료율(`progress`), 비용 등은 하위 Task들의 데이터를 기반으로 **자동으로 집계**되어 상위 요약 작업에 반영되어야 합니다.

### II. 태스크 유형 및 시각화 확장

Gantt 차트의 유연성과 분석 결과를 효과적으로 전달하기 위해 다양한 태스크 유형과 시각적 커스터마이징이 지원됩니다.

#### 1. 기본 태스크 유형
표준 Gantt 차트에서 기본적으로 제공되는 태스크 유형은 다음과 같습니다:
*   **Task (일반 작업)**
*   **Milestone (마일스톤)**: 프로젝트의 주요 완료 시점을 나타내며, 일반적으로 기간이 0인 작업으로 표현됩니다.
*   **Summary task (요약 작업)**: 하위 작업들을 포함하는 상위 계층 작업입니다.

#### 2. 커스텀 태스크 유형 (Custom Task Type) 지원
*   **확장성:** 기본 유형 외에 **사용자 정의 Task Type**을 추가하여 시각적 확장성을 확보할 수 있습니다 (예: `urgent`, `narrow`, `progress`, `round` 등).
*   **구현 방법:** 새로운 Task Type은 `taskTypes` 배열에 새 객체를 추가하여 정의할 수 있으며, 각 유형의 **시각적인 모습(look)**은 해당 Task Type의 `id`를 활용한 **CSS 규칙**을 통해 커스터마이징할 수 있습니다.
*   **활용:** 이 기능은 CPM 분석 후 **임계 경로(Critical Path) 상의 Task**를 시각적으로 강조하거나, 자원 충돌이 발생한 Task를 구분하는 데 필수적입니다.

### III. 프로젝트 추적 및 기준선 관리 (Baselines & Interim Plans)

프로젝트의 초기 계획과 실제 진행 상황을 비교하여 편차(Variance)를 분석할 수 있는 기준점 설정 기능은 PM Core의 핵심 요소입니다.

#### 1. 프로젝트 기준선 (Baseline) 설정
*   **정의:** 기준선은 프로젝트 시작 시 승인된 **최초의 계획(approved scope, schedule, and cost plan)**을 기록한 고정된 참조점(fixed reference point)입니다.
*   **포함 정보:** 기준선은 **시작일, 종료일, 기간, 작업, 비용 추정치** 등 약 20개의 기본 참조점(primary reference points)을 저장합니다.
*   **다중 기준선:** 프로젝트가 진행됨에 따라 계획이 크게 변경될 경우, 추가 기준선을 설정할 수 있으며, **최대 11개의 기준선** 데이터셋을 저장하고 관리할 수 있습니다. 이는 장기 프로젝트나 단계별 프로젝트에 유용합니다.
*   **기준선 적용:** 기준선은 **프로젝트 전체** 또는 **선택된 Task** (하위 작업 및 요약 작업 포함)에 대해 설정할 수 있습니다.
*   **기술적 구현 (ZeroPM):** 기준선 데이터는 `baselines` 및 `baseline_tasks` 테이블에 스냅샷 형태로 저장되며, 현재 계획(`start_date`, `finish_date`, `duration`)과의 차이를 비교하는 데 사용됩니다.

#### 2. 중간 계획 (Interim Plan) 설정
*   **정의:** 중간 계획은 프로젝트가 시작된 **이후**에 저장하는 현재 프로젝트 데이터의 스냅샷으로, 기준선과 비교하여 프로젝트 진행 상황을 평가하는 데 사용됩니다.
*   **포함 정보:** 중간 계획은 오직 **현재 시작일(Current start dates)**과 **현재 종료일(Current finish dates)**의 두 가지 정보만을 저장합니다.
*   **다중 중간 계획:** **최대 10개의 중간 계획**을 설정할 수 있습니다. 예를 들어, 월별 또는 분기별로 중간 계획을 설정할 수 있습니다.

### IV. 태스크 의존성 관리 (Dependencies)

정교한 스케줄링 엔진의 근간이 되는 의존성 관리 기능입니다.

#### 1. 의존성 유형 및 제약 조건
*   **그래프 모델링:** Task와 Dependency는 **방향성 비순환 그래프(DAG)** 구조의 노드와 엣지로 모델링됩니다.
*   **4가지 의존성 유형:** 프로젝트의 복잡한 공정 논리를 표현하기 위해 4가지 유형의 의존성을 지원해야 합니다:
    *   **Finish-to-Start (FS)**: 선행 작업이 끝나야 후행 작업이 시작 (가장 일반적).
    *   **Start-to-Start (SS)**: 선행 작업이 시작되면 후행 작업도 시작 가능.
    *   **Finish-to-Finish (FF)**: 선행 작업이 끝나야 후행 작업도 끝날 수 있음.
    *   **Start-to-Finish (SF)**: 선행 작업이 시작되어야 후행 작업이 끝날 수 있음.
*   **Lag 및 Lead Time:** 의존 관계에 시간적 지연(**Lag**, 양수 값) 또는 중첩(**Lead**, 음수 값)을 적용할 수 있습니다.

#### 2. 데이터베이스 저장 전략
*   **정규화된 테이블:** 의존성 정보는 Task 테이블의 JSONB에 저장하는 대신, `links`라는 **별도의 연결 테이블(Junction Table)**로 정규화하여 저장하는 것이 데이터 무결성과 그래프 쿼리 성능에 유리합니다.
*   **핵심 필드:** `source` (선행 작업 ID)와 `target` (후행 작업 ID)이 Task 테이블을 참조하는 외래 키로 사용됩니다.

### V. CPM 계산 및 스케줄링 연동
WBS와 Dependency를 기반으로 **CPM(Critical Path Method)**을 계산하는 로직은 클라이언트 측 JavaScript/TypeScript 엔진에서 실행됩니다.

#### 1. CPM 계산의 필수 단계
*   **유효성 검증:** 스케줄링 계산을 시작하기 전에 **위상 정렬(Topological Sort)** 알고리즘(Kahn's algorithm 또는 DFS)을 사용하여 의존성 그래프에 **순환 참조(Cycle)**가 존재하는지 탐지해야 합니다. 순환이 있으면 프로젝트 완료가 불가능하므로 즉시 경고해야 합니다.
*   **전진 계산 (Forward Pass):** 각 Task의 **조기 시작일(ES)**과 **조기 종료일(EF)**을 계산하여 프로젝트의 최소 소요 기간을 결정합니다.
*   **후진 계산 (Backward Pass):** 프로젝트 최종 완료일로부터 역산하여 **늦은 시작일(LS)**과 **늦은 완료일(LF)**을 계산합니다.
*   **임계 경로 식별:** **총 여유 시간(Total Float, $LS - ES$ 또는 $LF - EF$)**이 0인 Task들을 식별하여 **임계 경로(Critical Path)**를 도출합니다.

#### 2. 업무일 기반 일정 계산
*   **비즈니스 캘린더 반영:** Task 기간 계산 시 단순한 날짜 연산이 아닌, **주말 및 사용자 정의 공휴일(Holidays)**을 제외한 **유효 작업 시간**만을 계산해야 합니다.
*   **라이브러리:** `dayjs-business-days`와 같은 라이브러리를 채택하여 사용자 정의 휴일 목록을 쉽게 반영하고 정확한 일정 계산을 수행합니다. 이 캘린더 정보는 `projects` 테이블의 `calendar_settings` 필드에 JSONB 형태로 저장될 수 있습니다.

---

**비유적 설명:**

핵심 프로젝트 관리 기능은 건물을 짓는 설계도와 같습니다. **WBS**는 건물의 층과 방을 나누는 **구조체**이며, **Dependency**는 어떤 벽이 먼저 세워져야 하는지 정하는 **공정 순서**입니다. **Baseline**은 고객과 합의한 **최초의 청사진**이며, **CPM** 엔진은 이 청사진을 기반으로 "이 건물이 무너지지 않고 가장 빨리 지어질 수 있는 최소 시간(Critical Path)"을 계산해내는 **구조 역학 계산기** 역할을 수행합니다. 모든 기능이 이처럼 구조적으로 엮여 있어, 하나의 Task를 변경하면 전체 건물의 일정에 자동으로 영향을 미치게 됩니다.
