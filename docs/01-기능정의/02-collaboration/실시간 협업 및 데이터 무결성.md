# 실시간 협업 및 데이터 무결성

ZeroPM 프로젝트의 **실시간 협업(Real-time Collaboration)**은 Google Docs와 유사한 **충돌 없는 동시 편집(Conflict-free Co-authoring)** 경험을 제공하는 것을 핵심 목표로 하며, 이는 **Zero-Cost 운영**을 유지하면서도 엔터프라이즈급 기능을 구현하기 위한 가장 중요한 기술적 도전 과제입니다.

ZeroPM은 이 목표를 달성하기 위해 **CRDTs (Conflict-free Replicated Data Types)**와 **Supabase Realtime** 인프라를 결합하는 하이브리드 전략을 사용합니다.

## 핵심 기술 전략: CRDTs (Yjs) 채택 및 원리

실시간 협업 기능을 구현하는 두 가지 주요 접근 방식인 **OT(Operational Transforms)**와 **CRDTs** 중, ZeroPM은 분산 환경과 오프라인 지원에 유리한 CRDT를 선택했습니다.

### 1. CRDTs의 필요성 및 장점

*   **충돌 없는 병합:** CRDTs는 변경 사항이 **어떤 순서로 적용되더라도** (Commutative) 모든 클라이언트가 **동일한 최종 상태에 도달함**이 수학적으로 보장되도록 설계된 데이터 구조입니다. 이는 충돌 해결 로직을 직접 작성할 필요 없이 변경 사항이 **자동으로 병합**된다는 것을 의미합니다.
*   **OT와의 비교:** OT는 중앙 서버가 모든 연산의 순서를 정렬하고 변환해야 하므로, 간트 차트처럼 복잡한 데이터 구조(트리+의존성 그래프)에 적용하기에는 **구현 난이도가 극도로 높습니다**. 반면 CRDTs는 서버 의존성이 낮아 **오프라인 우선(Offline-First)** 앱에 완벽하게 적합하며, 클라이언트가 독립적으로 작업하고 나중에 동기화할 수 있습니다.
*   **Yjs 채택:** ZeroPM은 고성능 CRDT 라이브러리인 **Yjs**를 채택하여 Task 속성, 기간, 의존성 등 프로젝트 데이터 구조를 CRDT 객체로 변환합니다.

### 2. 클라이언트 우선 UX (Optimistic UI) 및 오프라인 지원

CRDTs 전략은 **클라이언트 우선(Local-First)** 아키텍처를 지원하며, 이는 사용자 경험(UX)을 크게 향상시킵니다.

*   **즉각적인 피드백:** **Optimistic UI 패턴**을 적용하여, 사용자가 Task 드래그와 같은 액션을 수행할 때 서버 응답을 기다리지 않고 UI를 즉시 업데이트함으로써 체감 성능을 향상시킵니다. 이는 네트워크 지연이나 느린 서버 작업이 **백그라운드에서 처리**되도록 합니다.
*   **오프라인 기능:** CRDT 상태를 **IndexedDB** 또는 **RxDB**와 같은 로컬 데이터베이스에 영구 저장함으로써, 사용자는 네트워크 연결 없이도 Task 수정 및 스케줄링 계산을 수행할 수 있으며, 연결이 복구되면 Yjs를 통해 변경 사항이 Supabase에 자동으로 동기화됩니다.

## 실시간 통신 및 인프라: Supabase Realtime 활용

ZeroPM은 **Supabase Realtime** 기능을 활용하여 실시간 통신 계층을 Zero-Cost로 구축합니다.

1.  **실시간 통신 역할:** Supabase Realtime은 CRDT 업데이트 델타와 현재 사용자 상태를 클라이언트 간에 전달하는 **전송 계층(WebSockets)** 역할을 수행합니다.
2.  **Zero-Cost 규모:** Supabase 무료 티어는 **200개의 동시 연결(Concurrent Peak Connections)**과 월 **200만 건의 메시지(Messages Per Month)**를 지원하며, 이는 소규모 팀 협업 요구사항을 충족시키기에 충분한 규모입니다.
3.  **Broadcast 및 P2P:** CRDT의 동기화 상태 및 커서 위치와 같은 임시 상태 변경은 **Supabase Realtime의 Broadcast 기능**을 통해 연결된 클라이언트 간에 빠르게 교환되어 서버 부하를 줄이고 반응 속도를 높이는 데 사용됩니다. 장기적으로는 WebRTC를 통한 **P2P 통신**을 사용하여 Supabase Realtime 서버의 메시지 사용량 부하를 **극적으로 줄이는** 전략이 고려됩니다.

## 충돌 방지 및 무결성 보장

CRDT가 데이터 병합을 자동화하지만, 복잡한 프로젝트 구조의 무결성과 논리적 충돌을 방지하기 위한 추가적인 메커니즘이 필요합니다.

### 1. 프레즌스(Presence) 기반의 소프트 락 (Soft Lock)

CRDT는 데이터 충돌을 해결하지만, 두 사용자가 같은 Task를 동시에 수정하여 발생하는 **논리적 충돌**을 방지하기 위해 Supabase의 **Presence** 기능이 사용됩니다.

*   **현재 사용자 인식:** 사용자가 특정 작업을 선택하거나 편집할 때, **Supabase Presence** 기능은 다른 사용자가 **어떤 Task를 보고 있거나 편집 중인지** 실시간으로 표시합니다.
*   **UX 기반 충돌 회피:** 이는 해당 Task에 **"소프트 락(Soft Lock)"** 사용자 경험을 구현하여, 다른 사용자는 해당 작업을 편집하는 것이 비활성화되거나 경고를 받게 됩니다. 이는 복잡한 CRDT 구현 대신 **UX 차원의 충돌 회피 전략**으로, 소규모 팀 협업에 안정적인 기능을 제공합니다.

### 2. 구조적 무결성을 위한 낙관적 잠금 및 트랜잭션

CRDT가 해결하지 못하는 구조적 변경(예: Task 삭제 또는 WBS 이동) 및 CPM 계산 결과의 최종 저장에는 데이터베이스 수준의 보호가 적용됩니다.

*   **낙관적 잠금 (Optimistic Locking):** Task 테이블에 **`version`** 또는 `updated_at` 필드를 추가하여 **낙관적 잠금** 패턴을 구현합니다. 클라이언트가 업데이트를 요청할 때, 서버는 클라이언트가 읽었던 버전과 현재 버전을 비교하여 불일치 시 업데이트를 거부합니다. 이는 CRDT가 담당하지 않는 **높은 수준의 구조적 무결성**을 보장합니다.
*   **원자적 트랜잭션:** Task, Dependency, Resource 할당 등 **논리적으로 연결된** 여러 데이터 변경은 **PostgreSQL의 트랜잭션** 기능을 활용하여 **단일 원자적 단위(Atomic Unit)**로 처리되어야 합니다. Supabase 환경에서는 이 복잡한 트랜잭션 로직을 **Postgres 함수** 내에 캡슐화하고 **RPC 호출**을 통해 실행함으로써, 네트워크 오류나 클라이언트 측 실패가 발생해도 데이터가 불완전하게 업데이트되는 것을 방지합니다.
