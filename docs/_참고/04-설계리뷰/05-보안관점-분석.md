# ZeroPM 보안 및 컴플라이언스 심층 분석 보고서

**작성일**: 2026-06-11
**분석 대상**: ZeroPM 웹 기반 프로젝트 관리 도구
**분석 범위**: 기획/설계 문서 및 현행 코드베이스 기반

---

## 1. 인증/인가 설계 평가

### 현황
Supabase Auth(이메일/패스워드 + Social OAuth)와 PostgreSQL RLS를 결합한 구조다. `profiles` 테이블의 `role` 필드로 RBAC를 구현하고, `project_members` 테이블로 프로젝트별 역할(admin/planner/viewer)을 부여한다.

### 문제점

**RBAC 역할 정의 불일치**
`database.ts`의 `UserRole`은 `admin | member | observer`인데, 데이터 설계 문서의 `project_members.role`은 `admin | planner | viewer`다. 두 체계가 동기화되지 않아 RLS 정책에서 어느 값을 기준으로 삼는지 모호하다. 실제 RLS 정책이 `member`를 체크하는데 DB에 `planner`가 저장돼 있다면 의도치 않은 접근 차단 또는 허용이 발생한다.

**JWT 토큰 무효화 전략 부재**
문서 어디에도 토큰 만료(expiry) 설정, 리프레시 토큰 로테이션(refresh token rotation), 강제 로그아웃(server-side revocation) 전략이 명시되어 있지 않다. 사용자 권한이 강등되거나 탈퇴 처리된 이후에도 기존 JWT가 만료 전까지 유효한 상태로 남는다.

**OAuth 보안 설정 미명시**
Social OAuth 구현에서 PKCE(Proof Key for Code Exchange) 적용 여부, `state` 파라미터를 통한 CSRF 방어, OAuth 리다이렉트 URI 화이트리스트 관리 방침이 문서에 전혀 언급되어 있지 않다. Supabase Auth는 기본적으로 PKCE를 지원하지만, Next.js App Router 환경에서 커스텀 흐름을 구현할 때 누락 가능성이 있다.

**`observer` 역할의 권한 범위 미정의**
`UserRole`에 `observer`가 있지만 어떤 데이터를 읽을 수 있고 어떤 액션이 차단되는지 구체적인 정의가 없다. RLS 정책에서 `observer`를 명시적으로 다루지 않으면 기본 허용 또는 기본 차단 중 어느 쪽이 적용되는지 불명확하다.

### 개선 권고

- `UserRole`과 `project_members.role`을 단일 체계로 통일하고, 각 역할의 허용 액션 매트릭스를 문서화한다.
- Supabase Dashboard에서 JWT 만료 시간을 최대 1시간으로 제한하고, refresh token rotation을 활성화한다. 권한 변경 시 `auth.users`의 `confirmed_at` 갱신 또는 `supabase.auth.admin.signOut(userId)` 호출로 즉시 무효화한다.
- OAuth 리다이렉트 URI를 환경별(dev/prod)로 엄격히 화이트리스트 관리하고, `state` 파라미터 검증 코드를 `src/app/auth/callback/` 라우트에 명시적으로 작성한다.

## 2. 데이터 접근 제어(RLS) 평가

### 현황
문서상 "Basic owner access" RLS가 구현 완료로 표시되어 있고, `project_members` 테이블 조회를 통해 멤버 여부를 확인하는 방식이 설계되어 있다. pgTAP 기반의 RLS 테스트가 `supabase/tests/`에 존재한다.

### 문제점

**RLS 정책 커버리지 불명확**
구현 계획의 체크리스트에서 "Basic owner access"만 완료로 표시되고, 멤버 초대 시스템은 미완료 상태다. 즉 현재 `project_members` 테이블이 스키마만 존재하고 RLS가 이를 실제로 참조하는지 불명확하다. 쓰기/삭제 정책의 커버리지가 읽기 정책과 동일한 수준으로 적용되고 있는지 확인이 필요하다.

**Service Role Key 노출 위험**
Next.js Serverless Functions에서 원자적 트랜잭션을 처리할 때 Service Role Key를 사용하게 된다. 이 키는 RLS를 우회하는 슈퍼 권한이므로, 단 한 곳이라도 클라이언트 번들에 포함되거나 `NEXT_PUBLIC_` 접두사가 붙으면 전체 데이터베이스가 노출된다. 현재 환경 변수 관리 전략이 문서화되어 있지 않다.

**공유 링크(Read-only Report Link)의 보안 설계 부재**
보고서 공유 링크 기능이 기획에 포함되어 있으나, 토큰 기반 익명 접근 시 RLS를 어떻게 통과시키는지 설계가 없다. 예측 가능한 UUID나 짧은 토큰을 사용하면 열거 공격(enumeration attack)에 취약하다. 별도의 `shared_reports` 테이블과 `anon` 역할에 대한 한정적 RLS 정책이 필요하다.

**`links` 및 부가 테이블의 RLS 누락 가능성**
`tasks` 테이블의 RLS는 설계가 있지만, `links`, `resources`, `assignments`, `costs`, `baselines`, `baseline_tasks` 테이블 각각에 독립적인 RLS 정책이 설정되어 있는지 확인이 필요하다. `links` 테이블에 RLS가 없으면 타 프로젝트의 의존성 구조 전체가 노출된다.

### 개선 권고

- 모든 테이블(tasks, links, resources, assignments, costs, baselines, baseline_tasks, interim_plans)에 대해 SELECT/INSERT/UPDATE/DELETE 각각의 RLS 정책을 명시적으로 작성하고 pgTAP 테스트로 커버한다.
- Service Role Key는 반드시 서버 사이드 환경 변수에만 저장하고, Vercel의 Environment Variables 설정에서 "Server only"로 구분한다. 코드 리뷰 시 `NEXT_PUBLIC_SUPABASE_SERVICE` 패턴을 차단하는 ESLint 규칙을 추가한다.
- 공유 링크는 `crypto.randomUUID()` 또는 128비트 이상의 토큰으로 생성하고, `shared_reports` 테이블에 만료일(`expires_at`)과 조회 횟수 제한을 함께 저장한다.

## 3. 데이터 보호 및 프라이버시

### 현황
사용자 프로필(`profiles`)에 `display_name`, `avatar_url`이 저장되고, 프로젝트 데이터 전반이 Supabase 클라우드에 저장된다. 데이터 암호화나 개인정보 처리 방침에 대한 언급은 문서에 없다.

### 문제점

**개인정보처리방침 및 법적 근거 부재**
국내 개인정보보호법(PIPA) 기준으로 `display_name`과 이메일 주소는 개인정보에 해당한다. 서비스 이용자가 있다면 개인정보처리방침 게시, 수집 동의, 보유 기간 정의가 법적으로 요구된다. 해외 사용자가 포함되면 GDPR의 적법한 처리 근거(lawful basis) 확보와 DPA(Data Processing Agreement) 체결이 필요하다.

**데이터 거주지(Data Residency) 미결정**
Supabase Free/Pro 플랜에서 기본 리전은 `us-east-1`이다. 국내 서비스라면 `ap-northeast-2`(서울) 리전 사용이 권장되며, 일부 기업 고객은 계약상 국내 데이터 저장 요건을 요구할 수 있다. 현재 리전 선택 기준이 문서에 없다.

**민감 데이터 암호화 전략 부재**
`calendar_settings`의 공휴일 목록이나 비용(`costs.amount`) 데이터는 그 자체로 민감하지 않지만, 향후 사용자의 업무 일정 및 예산 정보가 집적되면 기업 기밀에 해당할 수 있다. 현재 저장 데이터에 대한 컬럼 수준 암호화(Column-level Encryption)나 애플리케이션 수준 암호화 계획이 없다.

**데이터 삭제 권리(Right to Erasure) 미구현**
GDPR 및 개인정보보호법상 정보 주체는 자신의 데이터 삭제를 요청할 권리가 있다. 사용자 탈퇴 시 `profiles`, `project_members`, `assignments` 등 관련 테이블의 연쇄 삭제 또는 익명화 처리 흐름이 설계되어 있지 않다.

### 개선 권고

- Supabase 프로젝트 생성 시 `ap-northeast-2`(서울) 리전을 선택하고, 설계 문서에 리전 선택 근거를 명시한다.
- 서비스 출시 전 개인정보처리방침을 작성하고 법무 검토를 받는다. 최소 수집 원칙에 따라 불필요한 프로필 필드를 제거한다.
- 사용자 삭제 API(`DELETE /api/users/me`)를 구현하고, Supabase `auth.admin.deleteUser()`와 연계하여 관련 데이터를 일괄 삭제하거나 익명화한다.

## 4. 클라이언트 측 보안

### 현황
CPM 스케줄링 엔진이 클라이언트(브라우저)에서 실행되고, Web Worker로 비동기 계산된 결과가 Supabase에 직접 저장된다. React 19 + Next.js App Router 기반이며, XSS/CSRF 방어는 프레임워크 기본 동작에 의존하는 것으로 보인다.

### 문제점

**클라이언트 계산 결과의 서버 재검증 부재**
CPM 계산 결과인 `early_start`, `late_finish`, `total_float`, `is_critical` 값이 클라이언트에서 계산된 후 서버에 그대로 저장된다. 악의적인 사용자가 이 값을 조작하여 임계 경로를 은닉하거나 허위 일정 데이터를 삽입할 수 있다. 서버(Supabase RPC 또는 Next.js API)에서 이 값을 재계산하거나 최소한 범위 검증하는 로직이 없다.

**Content Security Policy(CSP) 미설정**
`next.config.js`에 CSP 헤더 설정이 문서에 언급되어 있지 않다. SVAR Gantt가 인라인 스타일을 사용한다면 `unsafe-inline`을 허용해야 하지만, 이 경우 XSS 공격 벡터가 넓어진다. Nonce 기반 CSP 또는 hash 기반 CSP 적용이 필요하다.

**CSRF 방어 현황 불명확**
Next.js App Router의 Server Actions는 기본적으로 SameSite 쿠키와 origin 검증을 수행하지만, 커스텀 API 라우트(`/api/*`)에서 별도의 CSRF 토큰 검증을 수행하는지 문서에 명시되어 있지 않다.

**낙관적 UI 업데이트의 일관성 위험**
Optimistic UI에서 서버 저장 실패 시 UI 롤백 로직이 있어야 한다. 현재 `DBObserver`와 `UIObserver`가 병렬로 실행되므로, DB 저장이 실패해도 UI가 성공 상태를 보여주는 불일치가 발생할 수 있다.

### 개선 권고

- CPM 계산 필드(`is_critical`, `total_float`)를 RLS 정책에서 클라이언트 직접 쓰기를 차단하고, Supabase RPC 함수 내에서만 업데이트되도록 한다. 당장 서버 재계산이 어렵다면 최소한 `total_float >= 0`, `duration > 0` 같은 범위 검증을 DB `CHECK` 제약 조건으로 추가한다.
- `next.config.js`에 아래와 같은 보안 헤더를 추가한다:
  ```js
  headers: [{ source: '/(.*)', headers: [
    { key: 'X-Content-Type-Options', value: 'nosniff' },
    { key: 'X-Frame-Options', value: 'DENY' },
    { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains' },
  ]}]
  ```
- `DBObserver`의 저장 실패 시 `UIObserver`에 롤백 이벤트(`task:rollback`)를 emit하는 에러 처리 흐름을 추가한다.

## 5. API 및 인프라 보안

### 현황
Supabase Anon Key가 클라이언트에 노출되는 구조이며, Next.js 미들웨어에서 인증 검증이 이루어진다. 환경 변수 관리 전략은 `.env.local` 사용으로 추정되나 문서화되어 있지 않다.

### 문제점

**Anon Key의 올바른 이해 부족 가능성**
Supabase Anon Key는 공개 설계(public by design)이지만, RLS가 제대로 설정되지 않은 경우 이 키만으로 모든 데이터에 접근 가능하다. 개발팀 전원이 "Anon Key는 RLS 정책이 방어선이며, 키 자체가 비밀이 아님"을 명확히 인지하고 있는지 확인이 필요하다.

**Next.js 미들웨어 인증 검증의 엣지 케이스**
Next.js 미들웨어는 Edge Runtime에서 실행되므로 Supabase의 `createServerClient`를 사용하는데, 쿠키 갱신 타이밍과 미들웨어 실행 순서에 따라 인증이 누락될 수 있다. 특히 정적 자산(`/_next/static/`)과 API 라우트의 인증 우회 여부를 명시적으로 점검해야 한다.

**환경 변수 분류 체계 부재**
`NEXT_PUBLIC_*` 변수(클라이언트 번들 포함)와 서버 전용 변수의 명확한 분류 기준이 없다. Service Role Key나 webhook 시크릿이 실수로 `NEXT_PUBLIC_` 접두사로 선언될 위험이 있다.

**공개 저장소 배포 시 시크릿 노출 위험**
`.env.local`이 `.gitignore`에 포함되어 있더라도, 개발 과정에서 하드코딩된 API 키나 Supabase URL이 커밋에 포함될 수 있다. GitHub의 secret scanning이 활성화되어 있는지 확인이 필요하다.

### 개선 권고

- 환경 변수를 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`(클라이언트 허용)와 `SUPABASE_SERVICE_ROLE_KEY`(서버 전용)으로 명확히 분류하고, README에 용도를 문서화한다.
- GitHub 저장소 설정에서 "Secret scanning"과 "Push protection"을 활성화하여 시크릿 패턴이 포함된 커밋을 사전 차단한다.
- `pre-commit` 훅 또는 GitHub Actions에서 `gitleaks` 또는 `trufflehog`를 실행하여 커밋 전 시크릿 스캔을 자동화한다.
- Vercel 배포 환경에서 Service Role Key는 Production 환경에만 설정하고, Preview 환경에는 제한된 권한의 별도 키를 사용한다.

## 6. 운영 보안

### 현황
pgTAP 기반의 RLS 테스트가 `supabase/tests/`에 존재하고, Jest 기반의 단위 테스트가 비즈니스 로직을 커버한다. 보안 패치 관리나 침해 사고 대응 계획은 문서에 없다.

### 문제점

**의존성 취약점 관리 부재**
SVAR React Gantt(GPLv3), Yjs, Supabase 클라이언트 등 서드파티 라이브러리에 대한 정기적 취약점 스캔 계획이 없다. 특히 SVAR Gantt는 GPLv3 라이선스이므로 재배포 시 법적 리스크도 동반된다.

**침해 사고 대응 계획 부재**
Supabase 데이터베이스 자격증명 노출, JWT 시크릿 탈취, 또는 비정상적인 대량 데이터 조회 발생 시 어떤 절차로 대응하는지 정의되어 있지 않다. Supabase 무료 플랜은 데이터베이스 활동 로그의 보존 기간이 짧아 사후 분석에 제약이 있다.

**보안 테스트 범위 편중**
현재 보안 테스트가 pgTAP(RLS 단위 테스트)에 집중되어 있다. OWASP Top 10 기준의 통합 보안 테스트(인증 우회, 권한 상승, SQL Injection, 입력 검증)는 수행되지 않는 것으로 보인다.

### 개선 권고

- `npm audit`를 CI 파이프라인(GitHub Actions)에 추가하고, Dependabot을 활성화하여 자동 의존성 업데이트 PR을 생성한다.
- 침해 사고 대응 체크리스트를 작성한다: Supabase 자격증명 로테이션 절차, 영향 받은 사용자 알림 방법, Vercel 배포 롤백 절차.

- OWASP ZAP 또는 Burp Suite를 사용한 연 1회 이상의 침투 테스트를 계획한다.

## 종합 평가

ZeroPM의 보안 아키텍처는 Supabase Auth + RLS 조합이라는 **검증된 기반** 위에 구축되어 있어 기본 방어선은 갖추고 있다. 그러나 현 시점의 가장 큰 리스크는 **RLS 정책의 실제 커버리지 불확실성**과 **Service Role Key 관리 전략의 미명시**다. 이 두 가지는 단일 실수로 전체 데이터베이스가 노출될 수 있는 치명적 지점이므로 H-1, H-2를 최우선 조치 항목으로 권고한다.

중기적으로는 클라이언트 스케줄링 엔진의 결과값이 서버 검증 없이 DB에 저장되는 신뢰 모델의 한계를 인식하고, 최소한 DB 제약 조건 수준의 방어를 추가해야 한다. 장기적으로는 GDPR/개인정보보호법 준수를 위한 법적 기반 마련이 서비스 확장의 필수 전제 조건이 될 것이다.
