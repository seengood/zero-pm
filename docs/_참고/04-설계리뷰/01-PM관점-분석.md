# ZeroPM PM 관점 심층 분석 보고서

**작성일**: 2026-06-11
**분석 대상**: ZeroPM 기획/설계 문서 전체 (기능정의, 데이터설계, 구현계획, 아키텍처 설계)
**분석 관점**: 시니어 PM / 프로젝트 관리 전문가

---

## 총평 (Executive Summary)

ZeroPM은 기술적 설계 완성도가 높고, 아키텍처 의사결정의 근거가 명확하게 문서화되어 있다. 특히 View/Brain 분리, 이벤트 기반 아키텍처, CPM 스케줄링 엔진의 설계는 시니어 엔지니어 수준의 완성도를 보인다. 그러나 **PM 문서로서의 완성도**는 전체적으로 낮다. 핵심 이유는 기능이 사용자 관점이 아닌 기술 구현 관점에서만 정의되어 있기 때문이다. 사용자 스토리, 페르소나, 수용 기준, 기능 ID 체계, 비기능 요구사항 명세가 전무하며, 이는 팀 확장 시 커뮤니케이션 비용과 품질 리스크를 크게 높인다.

---

## 1. 요구사항 완성도 분석

### 1.1 기능 요구사항 커버리지

**현황 평가: 보통 (C+)**

핵심 PM 기능(간트 차트, 의존성, CPM, WBS)은 상세하게 정의되어 있다. 보고서/분석, 자원/비용 관리도 테이블 형태로 목록화되어 있다. 그러나 각 기능이 "무엇을 하는가"에 대한 기술은 있지만 "왜 필요한가", "어떤 조건에서 올바르게 동작한다고 볼 것인가"에 대한 정의는 없다.

**문제점**

- 기능 목록이 체크리스트 수준에 머무른다. 예를 들어 "Implement Baseline Management"는 구현 목표이지 기능 요구사항이 아니다. 베이스라인을 몇 개까지 저장할 수 있는지, 저장 후 수정이 가능한지, 삭제 시 관련 데이터는 어떻게 처리되는지 등의 세부 규칙이 없다.
- 기능 간 경계가 불명확하다. 예를 들어 EVM 분석이 "자원 및 비용 관리 섹션과 연동"된다고 기술되어 있으나, 그 연동의 구체적인 데이터 흐름과 계산식이 정의되지 않았다.
- ALAP(As Late As Possible) 제약 조건이 "아직 완전히 구현되지 않음"으로 표시되어 있으나, 이것이 기능 갭인지 의도적 제외인지 문서상 명확하지 않다.

**개선 권고**

각 기능에 대해 최소한 다음 세 가지를 명시해야 한다: (1) 기능 설명 한 줄, (2) 입력/출력 규칙, (3) 예외 처리 방침. 기능 목록을 Phase별로 정리하되, 각 항목에 고유 ID(예: `FEAT-SCH-001`)를 부여하여 추적성을 확보해야 한다.

---

### 1.2 비기능 요구사항(NFR) 정의 현황

**현황 평가: 미흡 (D)**

아키텍처 문서 내에 성능 목표가 일부 언급되어 있으나(예: "60FPS 스크롤 성능", "200개 동시 연결"), 이것이 공식적인 NFR 명세서로 문서화되지 않았다. 가용성, 신뢰성, 보안, 유지보수성에 대한 정량적 목표가 전혀 없다.

**문제점**

- 성능: "수천 개의 작업"이라는 표현이 있으나, 공식 목표가 1,000개인지 10,000개인지 정의되지 않았다. Supabase Free Tier에서 응답 시간 SLA가 없다.
- 가용성: 목표 가동률(예: 99.5%)이 정의되지 않았다. Supabase와 Vercel의 Free Tier SLA가 유료 플랜과 다르다는 점이 리스크로 식별되지 않았다.
- 보안: RLS 정책이 구현되어 있으나, 전체 보안 요구사항(예: 데이터 암호화 수준, 세션 타임아웃, 비밀번호 정책)이 문서화되지 않았다.
- 접근성(Accessibility): WCAG 준수 여부에 대한 언급이 전혀 없다.

**개선 권고**

별도의 NFR 문서를 작성하고, 각 항목에 측정 가능한 기준을 정의해야 한다. 예시: "10,000개 Task 로드 시 초기 렌더링 3초 이내", "Supabase 장애 시 로컬 캐시로 읽기 전용 모드 30분 지속" 등.

---

### 1.3 우선순위 체계 적절성

**현황 평가: 미흡 (D+)**

Phase 1~6의 단계별 구현 순서가 기술 의존성 기준으로 정의되어 있어 기술적 논리는 합리적이다. 그러나 **비즈니스 가치** 기준의 우선순위 체계가 없다.

**문제점**

- MoSCoW(Must/Should/Could/Won't) 분류가 어디에도 없다. 예를 들어 번다운 차트는 Phase 6이지만, 실제 사용자에게는 Phase 2의 Web Worker보다 더 높은 가치를 제공할 수 있다.
- 기능 ID 체계가 전무하다. 구현 계획의 체크리스트와 기능 정의 문서, 데이터 설계 문서 간의 연결고리가 없어서 특정 기능이 어느 테이블과 연결되는지 추적이 불가능하다.
- "Zero-Cost 운영" 목표가 명시되어 있으나, 이것이 Must 제약인지 Should 목표인지 불명확하다.

**개선 권고**

Phase별 목록에 MoSCoW 분류를 병행 표기하고, 각 기능에 고유 ID를 부여해야 한다. 또한 비즈니스 임팩트와 기술 복잡도를 2x2 매트릭스로 정리하여 Phase 내 세부 우선순위를 가시화해야 한다.

## 2. 제품 전략 및 로드맵 평가

### 2.1 Phase별 구현 계획의 논리적 순서

**현황 평가: 우수 (A-)**

기술 의존성 관점에서 Phase 순서는 매우 논리적이다. Auth → WBS → Scheduling Engine → UI → Resource → Reporting의 흐름은 각 계층이 하위 계층에 의존하는 아키텍처를 정확히 반영한다. "Critical Path Dependencies" 섹션에서 이 의존 관계를 명시적으로 정리한 것도 적절하다.

**문제점**

- Phase 3의 UI 구현에 "Optimistic UI updates"와 "Presence for soft locking"이 포함되어 있는데, 이 두 기능은 Phase 1의 CRDT/Yjs 통합에 의존한다. 그런데 Phase 1에서 "Integrate Yjs with Supabase Realtime"이 미체크 상태이므로, Phase 3 진입 전에 이 항목이 완료되어야 한다는 게이팅 조건이 문서상 명시되지 않았다.
- Phase별 예상 소요 기간이 전혀 없다. 6개 Phase의 구현이 언제 완료될지에 대한 예측이 없어 로드맵으로서의 기능을 하지 못한다.
- "구현 완료" 기준이 없다. 현재 체크리스트의 체크 여부가 완료 기준인데, "무엇이 되어야 완료인가"가 정의되지 않았다.

**개선 권고**

각 Phase에 진입 조건(Entry Criteria)과 완료 조건(Exit Criteria)을 명시해야 한다. 또한 대략적인 T-shirt 사이징(S/M/L/XL)이라도 Phase별로 부여하여 전체 타임라인 감각을 확보해야 한다.

---

### 2.2 MVP 범위의 적절성

**현황 평가: 주의 필요 (C)**

현재 정의된 MVP(Phase 1+2+3 완료 상태)는 인증 + WBS + CPM 스케줄링 + 간트 차트로, 기술적으로 완결성이 있으나 **사용자가 실제로 가치를 느끼기에는 범위가 과도하게 넓다**.

**문제점**

- Phase 2의 Web Worker 기반 CPM 최적화는 MVP 단계에서 필수가 아닐 수 있다. 수천 개의 Task를 처음부터 다룰 초기 사용자는 없을 가능성이 높다.
- 반면, Phase 6의 PDF 내보내기나 대시보드 같은 기능은 실제 PM 업무에서 매우 초기부터 필요한 기능이다. 이것이 Phase 6에 배치된 것은 기술 의존성 논리는 맞지만, 비즈니스 가치 관점에서는 재검토가 필요하다.
- "Zero-Cost 운영"을 목표로 하면서 동시에 "MS Project급 정밀한 스케줄링"을 MVP로 설정하는 것은 타겟 사용자와 제품 포지셔닝 간의 간극을 만들 수 있다.

**개선 권고**

MVP는 "특정 사용자가 특정 가치를 경험할 수 있는 최소 기능 집합"으로 재정의해야 한다. 예를 들어 "5명 이하 팀이 하나의 프로젝트에서 간트 차트로 일정을 관리하고 의존성을 설정할 수 있다"를 MVP 성공 기준으로 정의하고, 이를 달성하는 데 필요한 기능만 Phase 1로 압축하는 접근을 권장한다.

---

### 2.3 경쟁 제품 대비 차별화 포인트

**현황 평가: 미흡 (D+)**

기술 문서 내에 차별화 요소가 언급되어 있으나(Zero-Cost, 오픈소스, MS Project급 CPM), 이것이 경쟁 분석에 기반한 포지셔닝 전략으로 정리되지 않았다.

**문제점**

- 경쟁 제품(MS Project, Asana, Monday.com, Linear, Notion) 대비 ZeroPM의 포지셔닝이 어디에도 명시되지 않았다.
- "Zero-Cost"는 차별화 포인트이기도 하지만 제약이기도 하다. Supabase Free Tier의 200개 동시 연결 한계는 팀 규모 확장 시 즉각적인 병목이 된다.
- 오픈소스 핵심 라이브러리(SVAR Gantt GPLv3)를 사용하면서 제품 자체의 라이선스 정책이 명시되지 않았다.

**개선 권고**

경쟁 분석 섹션을 추가하고, ZeroPM이 어떤 사용자 세그먼트를 위한 제품인지 명확히 정의해야 한다. "Self-hosted, Zero-Cost PM 도구를 원하는 소규모 기술 팀"과 같이 구체적인 타겟을 정의하면 기능 우선순위 결정과 마케팅 전략에 일관성이 생긴다.

## 3. 사용자 관점 요건

### 3.1 페르소나 정의 여부

**현황 평가: 없음 (F)**

문서 전체에 사용자 페르소나가 존재하지 않는다. 데이터 설계의 `profiles` 테이블에 `role` 필드(`admin`, `member`, `observer`)가 있고, `project_members`에 `admin`, `planner`, `viewer` 역할이 있으나, 이것은 권한 체계이지 사용자 페르소나가 아니다.

**문제점**

- "누가 간트 차트를 편집하는가", "누가 보고서를 보는가", "누가 CPM 결과를 해석하는가"에 대한 사용자 정의가 없다.
- 역할(`planner`, `viewer`)이 정의되어 있으나, 각 역할의 주요 태스크와 고통 포인트(Pain Point)가 없어서 기능 우선순위 결정이 개발자 주도로 이루어질 수밖에 없다.

**개선 권고 (High Priority)**

최소 두 개의 핵심 페르소나를 정의해야 한다. 예를 들어 "프리랜서 PM(혼자 사용, 리포팅 중심)"과 "스타트업 팀 리더(5-10명 협업, 일정 관리 중심)"를 정의하고, 각 페르소나의 주요 사용 시나리오를 3개씩 도출하면 기능 우선순위 결정의 기반이 생긴다.

---

### 3.2 사용자 스토리 및 유스케이스 문서화 여부

**현황 평가: 없음 (F)**

기능 정의 문서가 모두 기술 구현 관점으로 작성되어 있고, 사용자 스토리 형식("As a [role], I want to [action], so that [benefit]")이 하나도 없다.

**문제점**

- 개발자가 "무엇을 구현할지"는 알지만 "왜 구현하는지", "어떤 맥락에서 사용되는지"를 알 수 없어서 엣지 케이스 처리나 UX 결정 시 기준이 없다.
- CPM 계산 결과를 간트 차트에 시각화하는 기능이 있으나, 사용자가 이 정보를 어떤 맥락에서, 어떤 결정을 내리기 위해 보는지가 정의되지 않았다.

**개선 권고 (High Priority)**

Phase별로 최소 5개의 핵심 사용자 스토리를 작성해야 한다. Epic 수준의 스토리만으로도 팀 내 공통 이해를 형성할 수 있다. 예시: "프로젝트 플래너로서, 임계 경로 상의 작업을 간트 차트에서 시각적으로 식별하고 싶다. 왜냐하면 지연 위험이 높은 작업에 리소스를 우선 배치해야 하기 때문이다."

---

### 3.3 수용 기준(Acceptance Criteria) 정의 여부

**현황 평가: 없음 (F)**

기능 정의와 구현 계획 모두에 수용 기준이 전혀 없다. 체크리스트의 체크가 완료 기준의 전부이다.

**문제점**

- "Implement CPM Calculation"이 체크되어 있으나, CPM이 올바르게 구현되었다는 것을 어떻게 검증하는가에 대한 기준이 없다. 테스트 코드(`src/lib/__tests__/scheduling.test.ts`)가 존재하지만, 이것이 기능 요구사항의 수용 기준을 커버하는지 확인할 방법이 없다.
- QA 팀이나 스테이크홀더가 기능을 검수할 때 사용할 수 있는 기준이 없어서 "완료"의 정의가 사람마다 달라진다.

**개선 권고 (High Priority)**

각 기능에 Given-When-Then 형식의 수용 기준을 최소 2-3개씩 정의해야 한다. 예시: "Given 선행 작업의 종료일이 변경되었을 때, When 간트 차트가 업데이트되면, Then 후행 작업의 시작일이 자동으로 재계산되어 표시되어야 한다."

---

## 4. 추적성(Traceability)

### 현황 평가: 미흡 (D)

**문제점**

- 기능 정의 → 데이터 설계 → 구현 계획 → 테스트 간의 연결 고리가 없다. 예를 들어 "번다운 차트" 기능이 어느 테이블 데이터를 사용하는지, 어느 Phase에서 구현되는지, 어떤 테스트로 검증되는지 한 눈에 파악할 수 없다.
- 구현 계획의 체크리스트 항목이 기능 정의 문서의 어느 항목과 대응되는지 명확하지 않다. "Implement Dependency Management"와 Links 테이블 설계, `e2s`/`s2s`/`e2e`/`s2e` 링크 타입 코드 간의 추적이 문서상으로 연결되지 않는다.
- Git 커밋 히스토리에 기능 ID가 없어서 코드 변경이 어느 요구사항에 대응되는지 추적이 불가능하다.

**개선 권고 (Medium Priority)**

요구사항 추적 매트릭스(RTM: Requirements Traceability Matrix)를 간단한 스프레드시트 형태로 유지해야 한다. 최소한 기능 ID, 해당 데이터 테이블, 구현 파일, 테스트 파일을 한 행에 매핑하는 수준이면 충분하다. 커밋 메시지에 기능 ID를 포함하는 컨벤션도 병행하면 코드 레벨 추적성이 확보된다.

## 5. 리스크 및 의존성 관리

### 5.1 외부 의존성 리스크

**SVAR Gantt GPLv3 라이선스**

현황: CLAUDE.md에 "SVAR React Gantt는 GPLv3이고 프로젝트 본체는 MIT"라는 경고가 있으나, 공식 문서에는 리스크 항목으로 등재되지 않았다. GPLv3 컴포넌트를 MIT 프로젝트에 포함하면 배포 시 전체 소스코드 공개 의무가 발생할 수 있다. 이는 상업적 배포나 SaaS화 시 법적 리스크가 된다.

**Supabase Free Tier 제약**

현황: 200개 동시 연결, 월 200만 건 메시지 제한이 언급되어 있으나, 이것이 언제 병목이 되는지에 대한 시뮬레이션이 없다. 100명 사용자가 동시에 접속하면 각 사용자가 Realtime 채널을 2개(tasks, links) 구독할 경우 200개 연결 한계에 즉시 도달한다.

**개선 권고 (High Priority)**

라이선스 리스크에 대해 법률 자문을 받거나, SVAR Gantt의 유료 라이선스 비용을 사전에 파악해두어야 한다. Supabase 제약에 대해서는 Pro Plan 전환 기준(동시 사용자 수, 월간 메시지 수)을 미리 정의하고, 해당 기준 도달 시 비용 예산을 문서화해야 한다.

---

### 5.2 미구현 기능의 우선순위 논거

**문제점**

ALAP 제약 조건, 자원 관리, EVM 분석이 미구현 상태이나, 이것이 의도적 범위 제한인지 기술 난이도 때문인지 문서상 불명확하다. 특히 자원 관리와 EVM은 기능 정의 문서에서 상세히 설명되어 있음에도 Phase 5, 6으로 밀려 있어, 초기 사용자가 핵심 기능을 오랫동안 사용할 수 없는 상태가 지속된다.

**개선 권고 (Medium Priority)**

미구현 기능에 대해 각각 "의도적 제외(Won't have in MVP)", "기술적 선행 조건 미충족(Blocked)", "우선순위 낮음(Backlog)" 중 하나로 명시적으로 분류해야 한다. 이를 통해 사용자 피드백 수집 후 우선순위를 조정할 근거가 생긴다.

---

### 5.3 Zero-Cost 제약의 현실성

**현황 평가: 주의 필요**

Vercel Hobby Plan + Supabase Free Tier 조합은 소규모 팀(10명 이하, 동시 접속 20명 이하)에서는 현실적이다. 그러나 다음 시나리오에서 즉각적인 비용이 발생한다.

- Supabase Realtime 200개 연결 초과
- Supabase 데이터베이스 500MB 초과 (대규모 프로젝트 데이터)
- Vercel Hobby Plan의 서버리스 함수 실행 시간 초과

**개선 권고 (Medium Priority)**

"Zero-Cost 운영 가능 범위"를 명시해야 한다. 예: "프로젝트 5개, 팀원 20명, 동시 접속 50명 이하에서 Free Tier 운영 가능". 이 범위를 초과하는 시나리오에 대한 대응 계획(Supabase Pro: $25/월, Vercel Pro: $20/월 등)도 문서화해야 한다.

## 결론

ZeroPM은 기술적 기반이 탄탄한 프로젝트이다. 아키텍처 설계, 데이터 모델링, 구현 계획의 기술적 완성도는 시장에 출시할 수 있는 수준의 제품을 만들 수 있는 기반을 갖추고 있다. 그러나 현재 문서 체계는 "엔지니어가 엔지니어를 위해 쓴 문서"에 머물러 있다.

PM 문서로서 가장 시급한 과제는 **사용자 중심 관점의 도입**이다. 페르소나, 사용자 스토리, 수용 기준 세 가지만 갖춰도 팀 커뮤니케이션 효율이 크게 향상되고, 기능 완성도에 대한 공통 기준이 생긴다. 두 번째 과제는 **추적성 체계 구축**으로, 기능 ID 도입과 요구사항 추적 매트릭스를 통해 요구사항 변경 영향도 분석과 회귀 테스트 관리가 가능해진다.

기술은 준비되어 있다. 이제 그 기술을 올바른 사용자 문제에 연결하는 PM 프레임워크를 갖추는 것이 다음 단계이다.
