# 사용자 인터페이스 및 아키텍처 (UI/Architecture)

**사용자 인터페이스 및 아키텍처 (UI/Architecture)** 기능 도출 및 분석에 대해 ZeroPM 프로젝트의 **Zero-Cost** 및 **고성능** 목표를 중심으로 상세하게 설명드립니다.

ZeroPM은 MS Project급의 기능을 구현하기 위해 UI 렌더링, 클라이언트 측 로직, 서버리스 백엔드 간의 역할을 명확히 분리한 **분리형 아키텍처(Decoupled Architecture)**를 채택하고 있습니다.

---

## 사용자 인터페이스 및 아키텍처 (UI/Architecture) 상세 기능 분석

ZeroPM의 UI/Architecture는 고성능 프론트엔드 라이브러리, 클라이언트 측 스케줄링 엔진, 그리고 비용 효율적인 서버리스 백엔드를 통합하여 구성됩니다.

### I. 프론트엔드 기술 스택 및 호스팅 전략

ZeroPM은 **Next.js**를 프론트엔드 프레임워크로, **Vercel Hobby Plan**을 호스팅 인프라로 채택하여 개발 및 운영 비용을 최소화(Zero-Cost)합니다.

| 기능 분류 | 상세 기능 | 설명 및 기술적 근거 |
| :--- | :--- | :--- |
| **핵심 스택** | **Next.js / React 19 / TypeScript** | 최신 React 동시성 모드를 지원하며, 복잡한 알고리즘 구현의 타입 안정성을 보장합니다. |
| **Zero-Cost 호스팅** | **Vercel Hobby Plan** | Next.js 기반 애플리케이션의 호스팅은 Vercel Hobby Plan을 활용합니다. 이 플랜은 개인 및 소규모 팀의 비상업적 용도에 적합한 충분한 무료 자원(월 6,000 빌드 실행 시간, 100만 Edge 요청 등)을 제공합니다. |
| **백엔드 인프라** | **Supabase Free Tier** | PostgreSQL 기반의 데이터 저장소 및 인증(Auth) 처리를 담당하며, 관계형 구조와 무제한 API 요청 수로 Firestore의 쓰기 한도 위험을 회피합니다. |

### II. 고성능 UI 렌더링 (Gantt Chart & Virtualization)

MS Project급의 사용자 경험을 제공하기 위해, 대용량 데이터를 원활하게 처리하는 **가상화(Virtualization)** 기술이 필수적입니다.

#### 1. Gantt 차트 라이브러리 선정 및 성능
*   **선정 라이브러리:** **SVAR React Gantt**는 ZeroPM의 요구사항을 충족하는 유일한 오픈소스 솔루션으로 선정되었습니다.
*   **성능 최적화:** SVAR React Gantt는 **행 가상화(Row Virtualization)**와 **타임라인 가상화**를 지원합니다. 이는 뷰포트(Viewport)에 보이는 요소만 DOM에 렌더링함으로써 수천 개의 작업을 렌더링할 때도 60FPS의 스크롤 성능을 보장하고 메모리 사용량을 최소화합니다.
*   **커스터마이징:** SVAR Gantt는 **커스텀 Task Type**을 정의하고 해당 `id`를 활용한 **CSS 규칙**을 추가하여 시각적 확장이 용이합니다. 이는 CPM 분석 후 **임계 경로(Critical Path) 상의 Task**를 강조하거나, 자원 충돌이 발생한 Task를 시각적으로 구분하는 데 활용됩니다.

#### 2. 데이터 그리드 및 보조 시각화
*   **데이터 그리드:** Gantt 차트의 좌측 Task 목록 인터페이스(데이터 그리드)에도 **가상화**가 적용되어 대용량 데이터의 편집 및 스크롤 성능을 확보해야 합니다.
*   **보조 뷰:**
    *   **Kanban View:** Task의 상태(status)를 중심으로 시각화합니다.
    *   **Calendar View:** Task의 `due_date`를 중심으로 일정을 시각화하며, **FullCalendar** 라이브러리의 오픈소스 코어를 활용하여 구현될 수 있습니다.

### III. 클라이언트 측 스케줄링 엔진 아키텍처 (Logic Layer)

ZeroPM은 서버 측의 고비용 상용 스케줄링 엔진 없이, 모든 복잡한 CPM 및 RCPSP 계산을 클라이언트 측에서 자체 알고리즘으로 처리합니다.

#### 1. 로직-뷰 분리 전략
*   **Logic Layer (Brain):** 자체 개발된 **스케줄링 엔진(Graph Engine)**이 Task의 의존성(DAG)을 기반으로 CPM 계산을 수행합니다.
*   **View Layer:** SVAR React Gantt는 오직 엔진에서 계산된 결과(날짜, 여유 시간, Criticality)를 받아 화면에 그리는 역할만 수행합니다.

#### 2. 성능 최적화 및 비동기 계산
*   **Web Worker 활용:** CPM 및 RCPSP 충돌 감지 로직은 계산 복잡도가 높기 때문에, **Web Workers**를 활용하여 **백그라운드 스레드에서 비동기적으로 실행**되어야 합니다.
*   **메인 스레드 보호:** 이 비동기 실행을 통해 계산 중에도 **UI 메인 스레드가 블로킹되는 현상**을 방지하고 사용자 응답성을 유지할 수 있습니다.
*   **그래프 라이브러리:** 스케줄링 엔진은 Task와 Dependency를 노드와 엣지로 모델링하기 위해 **@dagrejs/graphlib**과 같은 JavaScript 그래프 라이브러리를 활용합니다.

### IV. 실시간 협업 및 동시성 아키텍처

ZeroPM은 Google Docs와 유사한 실시간 협업 경험을 위해 **CRDTs(Yjs)**와 **Supabase Realtime**을 결합하는 하이브리드 아키텍처를 사용합니다.

#### 1. CRDTs 채택 및 Optimistic UI
*   **CRDT 기반 동기화:** **Yjs**를 채택하여 여러 사용자의 동시 편집 시 변경 사항이 **충돌 없이 자동으로 병합**되도록 합니다. 이는 OT(Operational Transforms) 방식보다 복잡한 Gantt 데이터 구조(트리+그래프)에 구현하기 쉽고 분산 환경에 적합합니다.
*   **클라이언트 우선 UX (Optimistic UI):** 사용자가 Task 드래그와 같은 액션을 수행할 때 서버 응답을 기다리지 않고 **UI를 즉시 업데이트**하여 체감 성능을 높입니다. 이 Optimistic UI는 오프라인 기능을 완벽하게 지원하며, 변경 사항은 RxDB/IndexedDB와 같은 로컬 데이터베이스에 저장됩니다.

#### 2. 실시간 통신 및 잠금 전략
*   **Supabase Realtime:** **WebSockets** 기반의 통신 계층 역할을 수행하며, CRDT 업데이트 델타 및 임시 상태(예: 커서 위치)를 클라이언트 간에 전달합니다. 무료 티어는 **200개의 동시 연결**과 월 **200만 건의 메시지**를 지원합니다.
*   **프레즌스 기반 소프트 락:** CRDT가 해결하지 못하는 **논리적 충돌**을 방지하기 위해 **Supabase Presence** 기능이 사용됩니다. Presence는 다른 사용자가 **현재 어떤 Task를 편집 중인지** 실시간으로 표시하여, 다른 사용자가 해당 작업을 수정하지 않도록 유도하는 **UX 기반의 충돌 회피 전략**을 제공합니다.

### V. 데이터 무결성 및 보안 아키텍처

데이터 무결성은 PostgreSQL의 기능에 전적으로 의존합니다.

#### 1. 데이터 무결성을 위한 트랜잭션 및 RPC
*   **원자적 트랜잭션:** Task 생성, Dependency 변경, CPM 계산 결과 일괄 저장 등 **논리적으로 연결된** 여러 데이터 변경은 PostgreSQL의 **트랜잭션(Transaction)** 기능을 활용하여 **단일 원자적 단위(Atomic Unit)**로 처리됩니다.
*   **RPC 활용:** Supabase 환경에서 복잡한 트랜잭션 로직은 **Postgres 함수** 내에 캡슐화되고, 클라이언트 또는 Vercel Serverless Function에서 **RPC(Remote Procedure Call)** 호출을 통해 실행됩니다.

#### 2. 낙관적 잠금 (Optimistic Locking)
*   CRDT가 관리하지 않는 **높은 수준의 구조적 변경** (예: WBS 계층 구조 이동)에 대해, Task 테이블에 **`version`** 필드(또는 `updated_at`)를 사용하여 **낙관적 잠금** 패턴을 구현합니다. 클라이언트가 읽었던 버전과 현재 서버 버전이 불일치하면 업데이트를 거부하여 데이터 무결성을 보장합니다.

#### 3. 행 레벨 보안 (RLS)
*   **중앙 집중식 접근 제어:** PostgreSQL의 **Row Level Security (RLS)** 기능을 활성화하여, 인증된 사용자(authenticated)가 자신이 속한 프로젝트의 데이터에만 접근하고 수정할 수 있도록 **데이터베이스 레벨**에서 접근을 통제합니다. 이는 애플리케이션 코드의 복잡성을 줄이고 보안을 강화합니다.

---

**아키텍처 비유:**

ZeroPM의 아키텍처는 **고성능 경주용 자동차**와 같습니다. **SVAR Gantt**는 부드럽고 반응성이 뛰어난 **외관(UI)**이며, **Web Worker**는 서버의 도움 없이 모든 복잡한 계산을 처리하는 **독립적인 엔진(Scheduling Engine)**입니다. **Yjs와 Supabase Realtime**은 운전자(사용자)들이 서로의 조작을 충돌 없이 반영하는 **실시간 무선 통신 시스템** 역할을 하며, **PostgreSQL 트랜잭션**은 차량의 핵심 안전장치처럼 데이터의 무결성을 보장하여 충돌(데이터 손상)이 발생하지 않도록 합니다. 이 모든 것은 **Vercel**이라는 **제로 비용 차고**에서 운영됩니다.
