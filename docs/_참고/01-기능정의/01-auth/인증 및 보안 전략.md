# 인증 및 보안 전략

ZeroPM 프로젝트는 **Zero-Cost** 인프라를 활용하여 고급 프로젝트 관리 기능을 구현하는 것을 목표로 하며, 이 목표를 달성하기 위해 **Supabase**의 기능을 핵심적인 **인증 및 보안** 솔루션으로 채택합니다. 보안과 접근 제어는 단순히 사용자를 확인하는 것을 넘어, 협업 환경에서 데이터의 무결성과 일관성을 유지하는 데 필수적인 요소입니다.

제공된 소스 자료와 대화 기록을 바탕으로 ZeroPM의 인증 및 보안 전략을 다음과 같이 상세히 논의합니다.

---

## 🔒 ZeroPM의 인증 및 보안 전략

ZeroPM은 **Supabase Auth**를 인증 기능에, **PostgreSQL의 행 레벨 보안(RLS)**을 데이터 접근 제어에, 그리고 **트랜잭션**과 **낙관적 잠금**을 데이터 무결성 및 동시성 관리에 활용합니다.

### I. 사용자 인증 (Authentication) 및 관리

ZeroPM은 Supabase의 내장 인증 기능을 활용하여 사용자 접근을 관리하며, 이는 **Zero-Cost** 운영 목표에 부합합니다.

1.  **인증 방법:**
    *   **이메일/비밀번호**를 사용한 기본적인 사용자 계정 생성 및 로그인 기능을 Supabase Auth를 통해 제공합니다.
    *   Google, GitHub 등 **소셜 OAuth 공급자**를 통한 인증을 지원합니다.
2.  **비용 효율성:**
    *   Supabase의 무료 티어(Free Tier)는 **월 50,000명의 활성 사용자(MAUs)**를 포함하므로, 소규모 팀 및 개인 사용에는 비용 걱정 없이 사용자 관리가 가능합니다.

### II. 데이터 접근 제어 (Authorization) 및 RLS (Row Level Security)

데이터 접근을 데이터베이스 레벨에서 통제하는 **행 레벨 보안(RLS)**은 ZeroPM의 보안 핵심 요소입니다.

1.  **RLS의 역할 및 구현:**
    *   RLS는 Postgres 정책(Policy)을 생성하여 **개별 행(Row) 수준**에서 데이터 접근 규칙을 구현할 수 있게 합니다.
    *   RLS는 반드시 **활성화**되어야 하며, 이는 데이터베이스에 대한 접근 방식에 관계없이 **일관된 접근 제어**를 보장합니다.
    *   RLS 정책은 `SELECT`, `INSERT`, `UPDATE`, `DELETE`와 같은 정책 유형을 지원합니다.
2.  **Supabase Auth와의 통합:**
    *   RLS는 **Supabase Auth**와 통합되어, 정책 내에서 `auth.uid()`와 같은 **내장 헬퍼 함수**를 사용하여 사용자 기반의 접근 제어를 구현할 수 있습니다.
    *   RLS 정책은 인증된 사용자(`authenticated`)가 자신이 속한 프로젝트의 데이터에만 접근하고 수정할 수 있도록 제한하는 데 사용됩니다. 예를 들어, `auth.uid()`가 해당 행의 `user_id`와 일치하는지 확인하는 방식으로 구현됩니다.
3.  **역할 기반 접근 제어 (RBAC):**
    *   소규모 팀 운영을 위해 사용자 프로필에 `admin` 또는 `member`와 같은 **역할(role)**을 정의하고, RLS 정책에서 이 역할을 활용하여 권한을 세분화할 수 있습니다.
    *   권한 데이터는 사용자가 수정할 수 없는 **`raw_app_meta_data`** 필드에 저장하여 RLS 정책에서 안전하게 참조하는 것이 권장됩니다.

### III. 데이터 무결성 및 동시성 관리

ZeroPM은 실시간 협업과 복잡한 스케줄링 계산 결과의 원자적 저장을 위해 데이터 무결성 보장 기능을 통합합니다.

1.  **트랜잭션 (Transactions) 및 원자성:**
    *   프로젝트 관리 시스템에서는 Task, Dependency, Resource 할당 등 **논리적으로 상호 연결된** 여러 데이터 변경이 **단일 원자적(Atomic) 단위**로 처리되어야 합니다.
    *   PostgreSQL의 **ACID 속성**(`Atomicity`, `Consistency`, `Isolation`, `Durability`)은 데이터의 신뢰성과 일관성을 보장합니다.
    *   Supabase 환경에서 복잡한 트랜잭션 로직(예: CPM 계산 결과를 일괄 저장하거나 Baseline을 설정하는 경우)은 **Postgres 함수** 내에 캡슐화하고 **RPC 호출**을 통해 실행되어야 합니다. 이는 클라이언트 측의 네트워크 실패나 연결 중단 시에도 데이터가 불완전하게 업데이트되는 것을 방지합니다.
    *   **격리 수준(Isolation Level):** Postgres는 `READ COMMITTED` (기본값)를 제공하지만, 금융 시스템, 예약 시스템 등 **최고 수준의 무결성**이 필요한 복잡한 로직에서는 **`SERIALIZABLE`** 격리 수준을 사용하는 것이 모든 동시성 이상 현상(Race Conditions)을 방지하는 데 선호됩니다.
2.  **동시 쓰기 충돌 처리:**
    *   **낙관적 잠금 (Optimistic Locking):** CRDT가 처리하지 못하는 구조적 변경(Task 삭제, WBS 구조 이동)에 대해 **낙관적 잠금** 패턴을 적용합니다. Task 레코드에 **`version`** 필드를 추가하고, 업데이트 시 클라이언트가 읽었던 버전과 서버의 현재 버전이 일치하는지 확인하여 충돌을 감지하고 무결성을 보장합니다.
    *   **실시간 잠금 (Soft Lock / Presence):** 충돌 없는 복제 데이터 타입(CRDT) 구현의 복잡성을 피하고 사용자 경험을 위해 **Supabase Presence** 기능을 활용합니다. 특정 사용자가 **어떤 작업을 편집 중인지** 실시간으로 표시하여 (프레즌스 기반의 소프트 락) 논리적 충돌을 예방합니다.

### IV. 협업을 위한 동기화 및 보안 아키텍처

ZeroPM은 Yjs(CRDT)를 사용해 데이터를 자동으로 병합하여 충돌을 해결하므로, 중앙 서버 의존적인 OT(Operational Transformation) 방식보다 분산 및 오프라인 환경에 유리합니다.

*   **CRDT 기반 동기화:** **Yjs**와 **Supabase Realtime**을 통합하여 동시 편집 시 데이터 충돌 없이 자동으로 병합되는 환경을 구축합니다.
*   **Realtime 쿼터:** Supabase 무료 티어는 **200개의 동시 연결**과 월 **200만 건의 메시지**를 지원하며, 이는 소규모 팀 협업에 충분한 규모입니다.

ZeroPM의 인증 및 보안 전략은 **Supabase RLS**를 통해 데이터 접근을 엄격히 제어하고, **PostgreSQL 트랜잭션**과 **낙관적 잠금**을 통해 데이터의 일관성을 유지하며, **CRDT**와 **Presence**를 통해 협업 환경에서의 충돌을 최소화하는 **다층적 방어 체계**로 요약될 수 있습니다.

**비유적 설명:**

ZeroPM의 **보안 시스템**은 마치 **국경 통제소**와 같습니다.

1.  **Supabase Auth (여권 발급):** 누가 합법적인 사용자(프로젝트 멤버)인지 확인합니다.
2.  **Row Level Security (비자 및 국경 검문):** 일단 통과된 사용자라도, "당신은 이 프로젝트(국가)의 이 데이터 행(지역)에 접근할 권한이 있다"는 RLS 정책(규칙)을 통과해야만 데이터를 볼 수 있습니다. 이는 중앙 서버의 실수를 방지하고 데이터 격리(Data Isolation)를 보장합니다.
3.  **트랜잭션 (공증 시스템):** 여러 개의 중요 변경 사항(Task, Resource, Dependency)을 동시에 처리할 때, 모든 변경이 성공하거나(COMMIT) 모두 실패하도록(ROLLBACK) 보장하여 데이터의 무결성을 지킵니다.
