# 고급 스케줄링 및 엔진 (Advanced Scheduling Engine)

**고급 스케줄링 및 엔진 (Advanced Scheduling Engine)** 기능 도출 및 분석은 MS Project 수준의 정밀한 기능을 구현하려는 ZeroPM 프로젝트의 핵심이자 가장 기술적인 도전 과제입니다.

ZeroPM은 고가의 상용 엔진을 사용하는 대신, **클라이언트 측 JavaScript/TypeScript 환경**에서 모든 복잡한 스케줄링 계산을 자체 알고리즘으로 처리하는 분리형 아키텍처(Decoupled Architecture)를 채택했습니다.

이하에서는 RCPSP (Resource-Constrained Project Scheduling Problem, 자원 제약 프로젝트 스케줄링 문제)의 해결을 목표로 하는 고급 스케줄링 엔진의 상세 기능 및 기술적 구현 방안을 심층적으로 분석합니다.

---

## I. 문제 정의: 자원 제약 프로젝트 스케줄링 (RCPSP)

프로젝트 스케줄링에서 가장 중요한 도전 과제는 **자원 제약 프로젝트 스케줄링 문제(RCPSP)**를 해결하는 것입니다.

1.  **RCPSP의 특성 및 목표:**
    RCPSP는 선행 관계(precedence relationship)를 따르는 일련의 활동(activities)들로 구성되며, 이 활동들은 **갱신 가능한(renewable) 제한된 자원**을 소비합니다.
    RCPSP의 목표는 이러한 자원 제약 조건과 선행 조건(precedence constraints)을 모두 충족하면서 **프로젝트 완료 기간(makespan, $C_{\max}$)을 최소화하는 일정(schedule)을 찾는 것**입니다.
2.  **문제의 복잡성:**
    RCPSP는 Job Shop Scheduling 문제의 일반화이며, **NP-hard** 문제에 속합니다. 따라서 프로젝트 규모가 커질 경우 정확한 최적 방법을 사용하는 것은 불가능하며, **휴리스틱(Heuristics)**을 사용하여 합리적인 응답 시간 내에 최상의 근사 해법을 도출해야 합니다.

## II. 임계 경로 분석법 (CPM) 기능 및 로직

고급 스케줄링 엔진은 RCPSP에 앞서, 자원 제약 없이 오직 선후행 관계만을 고려하여 프로젝트의 최소 기간을 산출하는 **임계 경로 분석법(Critical Path Method, CPM)**을 구현해야 합니다.

### 1. 그래프 모델링 및 유효성 검증

정확한 일정 계산을 위해 Task와 Dependency는 수학적 모델인 **방향성 비순환 그래프(DAG, Directed Acyclic Graph)**로 모델링됩니다.

| 상세 기능 | 설명 및 구현 전략 | 기술적 근거 |
| :--- | :--- | :--- |
| **DAG 구조 관리** | Task를 노드(Node)로, Dependency를 방향성 엣지(Edge)로 모델링합니다. JavaScript 환경에서 **`graphlib`**와 같은 라이브러리를 사용하여 이 그래프 구조를 효율적으로 관리합니다. | |
| **4가지 의존성 유형** | **FS (Finish-to-Start), SS (Start-to-Start), FF (Finish-to-Finish), SF (Start-to-Finish)**의 모든 유형을 지원하며, **Lag 및 Lead Time**을 적용할 수 있습니다. | |
| **순환 참조 감지 (Cycle Detection)** | 스케줄링을 시작하기 전에 의존성 그래프에 **순환(Cycle)**이 있는지 검증하는 것이 필수적입니다. 순환이 발견되면 프로젝트 완료가 불가능하므로 즉시 경고해야 합니다. | |
| **위상 정렬 (Topological Sort)** | 순환 감지 및 계산 순서 선형화를 위해 **Kahn의 알고리즘(BFS 기반)** 또는 **DFS 기반 위상 정렬**을 사용합니다. | |

### 2. CPM 계산 및 임계 경로 도출

CPM 계산은 두 가지 패스(Pass)로 나뉩니다.

| 상세 기능 | 계산 로직 및 목표 | 결과 필드 (Postgres) |
| :--- | :--- | :--- |
| **전진 계산 (Forward Pass)** | 프로젝트 시작 시점부터 진행하여 각 Task의 **조기 시작일(ES)**과 **조기 종료일(EF)**을 산출합니다. 모든 선행 작업의 $\text{EF}$ 중 최댓값이 후행 작업의 $\text{ES}$가 됩니다. | $\text{ES}$ (Early Start) |
| **후진 계산 (Backward Pass)** | 프로젝트 최종 완료 시점에서 역산하여 각 Task의 **늦은 완료일(LF)**과 **늦은 시작일(LS)**을 산출합니다. 모든 후행 작업의 $\text{LS}$ 중 최솟값이 선행 작업의 $\text{LF}$가 됩니다. | $\text{LF}$ (Late Finish) |
| **총 여유 시간 (Total Float)** | $\text{Total Float} = LS - ES$ (또는 $LF - EF$)를 계산합니다. 이 값은 해당 작업이 프로젝트 전체 완료일에 영향을 주지 않고 지연될 수 있는 최대 시간을 나타냅니다. | $\text{Total Float}$ |
| **임계 경로 시각화** | **Total Float이 $0$ 인 Task**들을 식별하여 **임계 경로(Critical Path)**로 정의하고, Gantt 차트에서 붉은색 등으로 시각적으로 강조합니다. | $\text{is\_critical}$ (BOOLEAN) |

### 3. 업무일 기반 일정 계산 (Business Calendar)

프로젝트 스케줄링의 정확도를 보장하기 위해, 계산 시 단순한 날짜 연산이 아닌 **유효한 업무일**을 반영해야 합니다.

| 상세 기능 | 설명 및 구현 전략 | 기술적 근거 |
| :--- | :--- | :--- |
| **비즈니스 캘린더 반영** | Task 기간 계산 시 **주말(Weekends) 및 사용자 정의 공휴일(Holidays)**을 제외한 유효 작업 시간만을 고려합니다. | |
| **캘린더 설정 저장** | 주말 설정, 공휴일 목록 등의 비즈니스 캘린더 정보는 `projects` 테이블의 `calendar_settings` 필드(JSONB 타입)에 저장하여 엔진 계산에 사용됩니다. | |
| **라이브러리 채택** | `dayjs-business-days`와 같은 라이브러리를 채택하여 사용자 정의 휴일 목록을 쉽게 반영하고 정확한 일정 계산을 수행합니다. | |

## III. 자원 스케줄링 (RCPSP) 및 충돌 관리 기능

ZeroPM은 자원 제약 최적화(Leveling)가 아닌, **충돌 감지 및 휴리스틱 기반 조정 지원**을 RCPSP의 핵심 목표로 설정합니다.

1.  **자원 모델링:**
    *   **갱신 가능한 자원 정의:** 인력(People), 장비(Equipment) 등 시간 단위마다 최대 가용량이 정해진 자원($R_k$)을 정의하고, 각 Task가 요구하는 자원의 양($r_{jk}$)을 설정해야 합니다.
    *   **자원 할당 및 단위:** Task에 자원을 50%, 100% 등의 단위로 할당하며, 이는 `assignments` 테이블에 `required_units`로 저장됩니다.
2.  **실시간 충돌 감지 로직:**
    *   엔진은 스케줄링된 모든 Task를 순회하며 특정 시점($t$)에 할당된 모든 Task의 자원 요구량 합계($\sum r_{jk}$)가 자원의 최대 가용 용량($R_k$)을 초과하는지 **실시간으로 감지**합니다.
    *   충돌 발생 시, Gantt 차트의 Task 바 색상을 변경하는 등 **시각적 피드백**을 제공하여 사용자에게 즉시 알립니다.
3.  **휴리스틱 기반 조정 지원:**
    *   자동 평준화 알고리즘(예: Burgess Method) 대신, 충돌 발생 시 사용자에게 **수동 조정을 위한 우선순위 정보를 제공**합니다.
    *   **LFT (Latest Finish Time):** 프로젝트 마감에 가장 임박하여 여유 시간이 거의 없는 Task($\text{LF}$ 값이 가장 빠른 Task)에 높은 우선순위를 부여하여, 이 작업부터 자원을 할당하도록 유도합니다.
    *   **GRD (Greatest Resource Demand):** 병목 현상을 유발할 가능성이 높은, 가장 많은 자원을 요구하는 Task에 우선순위를 부여합니다.

## IV. 엔진 성능 및 기술적 구현 최적화

엔진의 계산 복잡도가 높기 때문에, ZeroPM은 사용자 경험을 저해하지 않도록 **비동기 성능 최적화**에 집중해야 합니다.

1.  **Web Worker를 활용한 비동기 계산:**
    *   CPM 및 RCPSP 충돌 감지와 같은 모든 복잡한 알고리즘은 **Web Worker**를 활용하여 **백그라운드 스레드에서 비동기적으로 실행**되어야 합니다.
    *   이를 통해 계산 중에도 **UI 메인 스레드가 블로킹되는 현상** (멈춤)을 방지하고 사용자 응답성을 유지할 수 있습니다.
    *   계산이 완료되면, 결과 데이터(ES, LF, Float 등)만 메인 스레드로 전달되어 UI를 업데이트합니다.

2.  **PostgreSQL 트랜잭션 및 RPC:**
    *   스케줄링 엔진이 계산을 완료하고 Task의 ES, LF, Total Float 등 **여러 Task 속성을 데이터베이스에 일괄 저장**할 때, 데이터 무결성을 위해 **원자적 트랜잭션(Atomic Transaction)**이 필수적입니다.
    *   Supabase 환경에서 이 복잡한 트랜잭션 로직은 **Postgres 함수 내부에 캡슐화**되고, 클라이언트 또는 Vercel Serverless Function에서 **RPC(Remote Procedure Call) 호출**을 통해 실행됩니다. 이는 네트워크 실패에도 불구하고 모든 변경이 성공하거나 모두 롤백(ROLLBACK)되도록 보장합니다.

## V. [심화 분석] RCPSP 최적화를 위한 하이브리드 휴리스틱

RCPSP는 복잡도가 매우 높기 때문에, 최적해에 근접한 해법을 찾기 위해 **하이브리드 메타휴리스틱(Hybrid Metaheuristic)** 알고리즘이 널리 연구되고 있습니다. 소스 자료에서 제시된 **GANS (Genetic Algorithm + Neighborhood Search)** 접근 방식은 RCPSP 솔루션의 품질을 높이는 고급 기술의 예시입니다.

### 1. 하이브리드 알고리즘의 구성 (GANS)

GANS는 **유전 알고리즘(GA)**과 **이웃 탐색 전략(NS, Neighborhood Search)**을 통합하여 프로젝트 기간(makespan)을 최소화합니다.

| 구성 요소 | 역할 및 기능 | 기술적 세부 사항 |
| :--- | :--- | :--- |
| **자원 랭킹 (Resource Ranking)** | 자원의 **희소성(scarcity)** 정도에 따라 자원의 중요도를 예측하고 가중치($w_k$)를 할당합니다. 이는 GA의 크로스오버와 NS 연산에서 더 가치 있는 자원(scarce resources)의 사용을 최대화하는 데 활용됩니다. | 누적 자원을 가진 이완 문제(relaxed problem with cumulative resources)를 해결하여 자원 랭킹을 얻습니다. |
| **유전 알고리즘 (GA)** | 해법(스케줄)을 표현하는 염색체(chromosome, 활동 리스트 $L$) 집합(population $\Gamma$)을 유지하고, 교배(crossover)와 돌연변이(mutation)를 통해 새로운 세대의 염색체를 생성하여 최적화합니다. | **밀집 유전자 (Dense Genes)**: 현재 스케줄에서 총 자원 활용도가 가능한 한 높은 유전자 세그먼트(활동 집합)를 선택하여 자손 염색체에 전달하는 휴리스틱 규칙을 사용합니다. |
| **교배 연산자 (Crossovers)** | **Crossover A, Crossover B** 두 가지 교배 절차를 사용하여 유망한 세그먼트(Dense Genes)를 부모로부터 자손에게 전달합니다. | |
| **이웃 탐색 (Neighborhood Search, NS)** | GA의 전역 탐색(global search)과 달리, 특정 실행 가능한 스케줄($S$) 주변의 이웃($N(S)$)을 탐색하여 해법의 품질을 개선하는 지역 탐색(local search) 절차입니다. | **블록 선택(Block selection):** 핵심 활동($j$)에 가깝거나 겹치는 활동 집합($A^s_j$)을 선택하여 이 활동들만 재스케줄링합니다. |
| **Tabu List (금지 목록)** | NS가 지역 최적해(local optimum)에 갇히거나 순환(cyclicity)하는 것을 방지하기 위해, 이전에 방문했던 해법의 속성(예: 시작 시간의 합 $TS(L)$)을 기록하여 해당 해법을 일시적으로 금지합니다. | |

### 2. RCPSP 해결을 위한 전략적 기능 배분

GA와 NS 연산자는 특정 유형의 RCPSP 인스턴스에서 성능 차이를 보이므로, GANS 알고리즘은 문제의 복잡도(Critical Path 대비 편차 $\sigma$)에 따라 GA와 NS 연산자의 적용 비율을 조절하는 **자체 조정(Self-tuning)** 요소를 도입합니다.

*   **쉬운 인스턴스 ($\sigma < \sigma_1$):** 주로 **GA 연산자**를 사용합니다.
*   **어려운 인스턴스 ($\sigma \geq \sigma_2$):** 주로 **NS 연산자**를 활용합니다.

이처럼 고급 스케줄링 엔진은 단순한 날짜 계산을 넘어, **그래프 이론, 메타휴리스틱, 통계 분석** 등의 복잡한 알고리즘을 결합하여 MS Project급의 정밀한 최적화 기능을 제공합니다.
