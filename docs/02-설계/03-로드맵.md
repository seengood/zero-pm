# ZeroPM 로드맵 · 자원/비용/보고 · 기술 부채

> 문서 버전: v4.3 (2026-06-12 Week 2 보안 기반 완료 반영)
> 역할: 이 문서는 제품 로드맵(Phase 1~4), 자원/비용/보고 계획, 단기 안정화 계획, **기술 부채 마스터 레지스트리**(유일 SSOT)를 통합한다. 전 설계 문서의 이슈 ID(H/M/L)는 §4 표를 참조한다.
> 참고: 제품 Phase 정본은 `docs/01-유저스토리/02-기능우선순위-NFR.md`, 수용 기준은 `docs/01-유저스토리/04-수용기준-AC.md`, 마이그레이션 정본은 `docs/02-설계/02-데이터-보안.md`.

기술 의존 순서는 단일 제품 Phase 체계와 산문으로 표현한다. 핵심 선행 관계만 요약하면 다음과 같다.

- Auth → RLS: 인증 없이 보안 강제 불가
- WBS → 스케줄링: 스케줄링은 태스크 그래프 구조에 의존
- 스케줄링 → 간트 UI: start/end 날짜 없이 간트 바 렌더링 불가
- 스케줄링 → 자원: 자원 사용량은 일정상 시간 분포 기반
- 기준선(Baseline) → EVM: 획득 가치는 기준선 데이터에 의존

## 1. 제품 Phase 1~4 로드맵

제품 Phase는 사용자 가치 기준 출시 단계다. 정본은 `docs/01-유저스토리/02-기능우선순위-NFR.md`이며, 아래는 목표·핵심 기능·Exit Criteria 요약이다. Phase별 Entry/Exit Criteria 정식화는 **M-11** 대상이다.

### Phase 1 — "혼자 써도 가치 있는 도구" (현재 → Closed Beta)

- **목표**: 프리랜서 PM 1인이 실제 클라이언트 프로젝트를 ZeroPM으로 관리하고 "MS Project 없이도 되겠다"는 첫 경험을 얻는다.
- **핵심 기능**: 프로젝트/WBS CRUD · 간트(드래그/리사이즈/링크) · 의존성 자동 스케줄링(4종 링크+Lag) · 제약 조건 6종(5종 구현, ALAP 미완성) · 크리티컬 패스 시각화(임계경로 필드·RPC 구현 완료, 전진/후진 계산 알고리즘 미구현) · Auth+RLS · 줌 레벨.
- **Exit Criteria**: 가입 후 5분 내 간트 포함 프로젝트 생성 · FS 링크 생성 시 후행 태스크 날짜 즉시 자동 재계산 · 크리티컬 패스 시각 구분 · 외부 베타 5명 이상이 2주 이상 이탈 없이 사용 · P0 버그 0건, P1 버그 3건 이하.

### Phase 2 — "팀이 함께 쓸 수 있는 도구" (Closed Beta → Open Beta)

- **목표**: IT 팀 PM이 5~15명 팀과 스프린트/릴리즈 일정을 ZeroPM에서 관리하며 "Jira 타임라인보다 낫다"는 경험을 얻는다.
- **핵심 기능**: 실시간 멀티유저 동기화(안정화) · 기준선 비교(UI) · Undo/Redo · 태스크 코멘트/메모 · 멤버 초대·역할 관리 · 완료율(%) 입력·표시.
- **Exit Criteria**: 3인 이상 동시 편집 충돌 없는 동기화 · 역할(Viewer/Editor)별 편집 권한 정확 제한 · Undo/Redo 20단계 이상 정확 복원 · 기준선 저장·계획 대비 실적 비교 작동 · 5개 이상 팀(팀당 3인 이상)이 2주 이상 사용 · WAU 증가율 2주 연속 10% 이상.

### Phase 3 — "보고하고 공유할 수 있는 도구" (Open Beta → v1.0 정식 출시)

- **목표**: 프리랜서 PM이 클라이언트에게 링크/PDF를 전달하고, IT 팀 PM이 경영진에게 대시보드로 현황을 보여줄 수 있다.
- **핵심 기능**: 프로젝트 대시보드 · PDF 내보내기 · 마감 임박 알림 · 담당자 배정(자원 배정 기초) · 프로젝트 템플릿.
- **Exit Criteria**: 대시보드에 전체 태스크 수·완료율·크리티컬 패스 길이·기준선 편차 한 화면 표시 · 간트 PDF 내보내기 레이아웃 무결 · 공개 열람 링크(Viewer)로 비회원 읽기 가능 · CSAT 4.0/5.0 이상 · p95 응답 시간 NFR 기준 이하.

### Phase 4 — "성장하는 팀을 위한 도구" (v1.0 이후)

- **목표**: 팀 규모 15~30명까지 버티며, 자원 배정·EVM·활동 로그로 팀 운영의 투명성을 확보한다.
- **핵심 기능**: 자원 배정·현황 뷰(EVM 포함) · 커스텀 필드 · 활동 로그 · 다크 모드.
- **Exit Criteria**: 태스크 500개·동시 접속 20명에서 NFR 성능 충족 · 자원별 배정 현황 뷰(히트맵/바 차트) 작동 · 변경 이력 최근 100건 조회 2초 이내 · 커스텀 필드 3종(텍스트/숫자/날짜) 생성·간트 열 표시.

## 2. 자원·비용·보고 계획 (요약 — 전부 미구현)

자원 관리(RES)·비용 관리(EVM)는 제품 Phase 4, 보고서·분석(REP)은 제품 Phase 3에서 출시한다. 단, 자원·비용에 의존하지 않는 보고서(대시보드·간트 PDF·CSV·진도 보고서)는 자원/비용 완성 이전에도 독립 구현 가능하다. 현재 `resources`/`assignments`/`costs` 테이블은 스키마만 존재하며, 부가 테이블 RLS 커버리지는 **H-2** 대상이다.

### 2.1 자원 관리 (RES, Phase 4)

ZeroPM의 자원은 시간 단위마다 가용량이 재충전되는 **갱신 가능한 자원(Renewable Resource)** 이다. 자원 제약 프로젝트 스케줄링(RCPSP) 맥락에서 자원의 최대 가용 용량($R_k$)을 저장한다.

- **데이터 모델**: `resources`(name/type[human·equipment·material]/capacity/cost_per_hour/calendar_id/version), `assignments`(task_id/resource_id/required_units/actual_units, `UNIQUE(task_id, resource_id)`, 태스크 삭제 시 CASCADE), `resource_calendars`(자원별 근무일·시간, 기본 8시간/일, 근무 요일은 스케줄링과 동일하게 **7일 전부**).
- **과부하 감지(Overallocation)**: 특정 시점 $t$의 활성 태스크 자원 요구 합계가 최대 가용 용량을 초과하면 충돌. $\sum_{j \in \text{active}(t)} r_{jk} > R_k \implies$ 충돌. 간트 바 경고색·자원 히스토그램 강조·충돌 목록 패널로 피드백한다(`src/lib/resourceConflict.ts` 신규 예정).
- **자원 평준화(휴리스틱 개요)**: RCPSP는 NP-hard이므로 완전 자동 최적화 대신 **충돌 감지 + 휴리스틱 기반 조정 제안**을 채택한다. **LFT**(Latest Finish Time, `late_finish` 빠른 태스크 우선 — CPM 후진 계산 의존, 현재 미구현)와 **GRD**(Greatest Resource Demand, 자원 요구 큰 태스크 우선) 두 휴리스틱으로 수동 조정용 우선순위 정보를 제공한다. 자원 희소성 가중치 기반 GANS(GA+이웃 탐색) 메타휴리스틱은 계산 비용이 커 1차 범위 밖 연구 항목으로 둔다.

### 2.2 비용·EVM (Phase 4)

- **비용 유형**: `hourly`(투입 시간 × 요율) · `fixed`(태스크 일회성) · `direct`(직접비) · `indirect`(간접비). `costs` 테이블에 저장하며 RLS는 H-2 대상.
- **EVM 핵심 값/지표**: PV(계획 가치)·EV(획득 가치)·AC(실제 비용)·BAC(완료 시 예산) 위에서 — **CV** = EV − AC, **SV** = EV − PV, **CPI** = EV / AC, **SPI** = EV / PV, EAC = BAC / CPI, ETC = EAC − AC. 순수 함수(`src/lib/evm.ts` 신규 예정)로 구현해 단위 테스트 커버리지를 확보한다.
- **S-Curve**: PV vs EV 시계열로 비용·일정 편차를 시각화.
- **비용 기준선(PMB)**: 승인된 최초 비용 계획을 `baseline_tasks` 스냅샷(baseline_start/finish/duration/cost/work 등 약 20개 참조점)으로 고정. `baselines` JSONB 정규화는 **L-6** 대상.

### 2.3 보고서·분석 (REP, Phase 3 / 일부 Phase 2)

전부 미구현이며 요약만 둔다.

- **프로젝트 대시보드(AC-15)**: 프로젝트 오버뷰 카드(진행률·마감 임박·지연 태스크 수), 상태별 파이 차트, 담당자별 바 차트, 내 작업/Today's Tasks. 지연 강조 — 마감 임박(`due_date <= today + 3일` 미완료), 지연(`due_date < today` 미완료, 배지+경고색+지연 일수), 빈 상태(프로젝트 0개 시 CTA).
- **간트 PDF/이미지 내보내기(AC-16)**: PDF(A4/A3 가로)·PNG/JPEG, 현재 뷰/전체 프로젝트 선택, 태스크명·날짜·의존성 화살표·크리티컬 패스·기준선 포함, 텍스트 셀렉터블, 전체 캡처 30초 이내. `html2canvas`+`jsPDF` 또는 Puppeteer 서버사이드. SVAR는 마운트 후 props 무시 특성 때문에 줌·스크롤을 `ganttApiRef.current.exec(...)`로 프로그래매틱 제어.
- **번다운 차트 / S-Curve**: 남은 작업량(태스크 수 또는 스토리 포인트) 추이를 이상선과 비교. 태스크 완료 이력을 `task_progress_history` 시계열 테이블에 저장.
- **진도 보고서**: 전체/WBS 롤업/담당자별 완료율, 마일스톤(duration=0) 달성 현황, 리소스 활용도 히트맵.
- **CSV 내보내기(AC-19, Phase 2 출시)**: 태스크 ID·이름·시작일·기간·종료일·부모 ID·의존성·제약 조건·진행률·크리티컬 여부(UTF-8). 가져오기 흐름은 WBS/데이터 도메인 소관.
- **읽기 전용 공유 링크(AC-13)**: `/projects/{id}/report?token=...` 형식, 만료 가능 토큰(`shared_links` 테이블), RBAC `viewer` 수준 읽기 전용, RLS는 H-2 대상.
- **보고용 Materialized View(L-8)**: 프로젝트별 진행률·상태 분포·EVM 핵심 지표를 사전 집계해 트랜잭션 경로와 분리, 성능 격리.

## 5. CI/CD·테스트·모니터링·배포·운영 요약

### CI/CD (M-12)

- `.github/workflows/ci.yml`: PR 시 lint + `npm audit --audit-level=high` + `npm test --coverage` + build 모두 통과 필수.
- 빌드는 SVAR Gantt CSS 로컬 복사본(`src/styles/gantt-svar.css`)을 전제로 한다. `Parsing CSS source code failed` 발생 시 `cp node_modules/@svar-ui/react-gantt/dist/index.css src/styles/gantt-svar.css` 단계 추가.

### 테스트 매트릭스

| 레이어 | 도구 | 대상 | 목표 커버리지 |
|---|---|---|---|
| 단위 | Jest(jsdom) | `src/lib/`, `src/hooks/` | lib ~100%, hooks ≥95% |
| 컴포넌트 | Jest + RTL | `src/components/` | ≥90% |
| 통합 | Jest | Supabase Auth(`npm run test:auth`) | 인증 플로우 전체 |
| E2E | Playwright(L-3) | 프로젝트 생성·의존성 전파·Realtime sync | 핵심 3개 |
| DB 보안 | pgTAP | `supabase/tests/` | 모든 테이블 RLS |

### 모니터링

- 에러 트래킹: Sentry(L-4, Month 2). DBObserver 실패·낙관적 잠금 충돌·스케줄링 예외·동기화 끊김 수집. anon key·이메일 등 민감 정보는 전송 전 스크럽.
- 성능 기준: 프로젝트 로드 쿼리 < 200ms(인덱스 추가 후), cascade/CPM 재계산 < 100ms(500개), Realtime 지연 < 500ms.
- Supabase Free → Pro 전환: 동시 접속 50명 지속 초과 / Realtime 한도 80% 도달 / DB 500MB 초과.

### 배포 (Vercel)

- `main` → Production, `staging` → Preview, PR → Preview 자동.
- 환경변수(H-5): `NEXT_PUBLIC_` 접두사는 공개 가능 값만. `SUPABASE_SERVICE_ROLE_KEY`는 서버 전용, `NEXT_PUBLIC_` 절대 금지·클라이언트 번들 미포함.
- 롤백: Vercel Promote to Production / 소프트 삭제(M-6) 적용 후 `deleted_at IS NOT NULL` 레코드 복구로 데이터 복원.

### 프로덕션 배포 전 최소 체크리스트

- 보안: 전 테이블 RLS 전수 점검(H-2) ✅ · Service Role Key 격리(H-5) ✅ · `UserRole` 통일(H-5) ✅ · 보안 헤더(M-8) · JWT 만료/refresh rotation(M-7).
- 데이터 무결성: ~~핵심 인덱스 7개(H-4)~~ ✅ · ~~`updateTask` RPC 전환(H-1)~~ ✅ · ~~`constraint_date` TIMESTAMPTZ(M-5)~~ ✅ · ~~`save_cpm_results` 배치(M-4)~~ ✅ · ~~소프트 삭제(M-6)~~ ✅.
