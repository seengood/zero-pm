## 1. 보안 개선 대상 요약

### High 우선순위 (즉시 조치)

| # | 항목 | 현재 문제 | 목표 상태 |
|---|------|-----------|-----------|
| H-1 | Service Role Key 서버 전용 격리 | 환경 변수 분류 전략 미명시, `NEXT_PUBLIC_` 실수 가능성 | 서버 전용 변수 목록 명문화, ESLint 규칙 차단 |
| H-2 | RLS 전체 테이블 커버리지 확보 | `links`, `resources`, `assignments`, `costs`, `baselines` RLS 누락 가능 | 모든 테이블 SELECT/INSERT/UPDATE/DELETE 정책 + pgTAP 테스트 |
| H-3 | UserRole 체계 통일 | `database.ts`의 `admin\|member\|observer`와 `project_members.role`의 `admin\|planner\|viewer` 불일치 | `owner\|editor\|viewer` 단일 체계로 통일 |
| H-4 | 시크릿 스캔 자동화 | GitHub Push Protection 및 pre-commit 스캔 미설정 | `gitleaks` pre-commit 훅 + GitHub Secret Scanning 활성화 |

### Medium 우선순위 (1개월 내 조치)

| # | 항목 | 현재 문제 | 목표 상태 |
|---|------|-----------|-----------|
| M-1 | JWT 만료 및 토큰 무효화 전략 | 토큰 만료 설정 및 강제 로그아웃 전략 미명시 | Access token 1시간, refresh rotation 활성화, 권한 변경 즉시 무효화 |
| M-2 | 보안 HTTP 헤더 설정 | `next.config.js`에 보안 헤더 없음 | HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy 추가 |
| M-3 | 클라이언트 계산값 DB 레벨 검증 | CPM 필드 클라이언트 직접 쓰기 가능, 서버 검증 없음 | DB `CHECK` 제약 조건, RLS로 클라이언트 직접 쓰기 차단 |
| M-4 | 공유 링크 보안 설계 | 공유 링크 토큰 보안 설계 없음 | 128비트 토큰, 만료일, 조회 제한, anon RLS 정책 |
| M-5 | 의존성 취약점 자동 스캔 | `npm audit` CI 통합 없음, Dependabot 미설정 | GitHub Actions `npm audit` + Dependabot 자동화 |
| M-6 | Supabase 리전 결정 | 리전 선택 기준 미명시 | 국내 서비스 기준 `ap-northeast-2`(서울) 명시 |

## 2. RBAC 역할 체계 통일 (H-3)

### 현재 불일치 상황

```
database.ts UserRole:       'admin' | 'member' | 'observer'
project_members.role:       'admin' | 'planner' | 'viewer'
```

두 체계가 동기화되지 않아 RLS 정책에서 어느 값을 기준으로 삼는지 모호하다. RLS가 `member`를 체크하는데 DB에 `planner`가 저장되어 있으면 의도치 않은 접근 차단 또는 허용이 발생한다.

### 통일 방향: owner | editor | viewer 3단계 체계

| 역할 | 설명 | 해당 기존 역할 |
|------|------|---------------|
| `owner` | 프로젝트 소유자, 모든 권한 + 멤버 관리 | 기존 `admin` |
| `editor` | 프로젝트 편집자, 태스크/링크/기준선 CRUD | 기존 `member`, `planner` |
| `viewer` | 읽기 전용, 공유 링크 포함 | 기존 `observer`, `viewer` |

### 역할별 허용 액션 매트릭스

| 리소스 | 액션 | owner | editor | viewer |
|--------|------|:-----:|:------:|:------:|
| tasks | SELECT | O | O | O |
| tasks | INSERT | O | O | X |
| tasks | UPDATE (일반 필드) | O | O | X |
| tasks | UPDATE (CPM 필드: is_critical, total_float) | O | X (RPC만) | X |
| tasks | DELETE | O | O | X |
| links | SELECT | O | O | O |
| links | INSERT | O | O | X |
| links | UPDATE | O | O | X |
| links | DELETE | O | O | X |
| baselines | SELECT | O | O | O |
| baselines | INSERT | O | O | X |
| baselines | UPDATE | O | O | X |
| baselines | DELETE | O | X | X |
| project_settings | SELECT | O | O | O |
| project_settings | UPDATE | O | X | X |
| project_settings | DELETE | O | X | X |
| project_members | SELECT | O | O | X |
| project_members | INSERT/DELETE (초대/강퇴) | O | X | X |

### database.ts 수정 내용

```typescript
// src/types/database.ts

// 기존: export type UserRole = 'admin' | 'member' | 'observer';
// 변경: 3단계 통일 체계
export type UserRole = 'owner' | 'editor' | 'viewer';

export interface UserProfile {
    id: string;
    display_name: string | null;
    avatar_url: string | null;
    role: UserRole; // 시스템 수준 역할 (현재는 'owner'만 특수 의미)
    created_at: string;
    updated_at: string;
}

export interface ProjectMember {
    id: string;
    project_id: string;
    user_id: string;
    role: UserRole; // 프로젝트 수준 역할: 'owner' | 'editor' | 'viewer'
    created_at: string;
}
```

### RLS 정책에서 역할 체크 방식

공통 헬퍼 함수를 통해 역할 체크를 중앙화한다. 구체적인 함수 설계는 3절을 참고한다.

```sql
-- 역할 계층을 숫자로 매핑하여 "최소 역할" 비교를 간단하게 처리
-- owner=3, editor=2, viewer=1
CREATE OR REPLACE FUNCTION role_level(r text) RETURNS int AS $$
  SELECT CASE r
    WHEN 'owner'  THEN 3
    WHEN 'editor' THEN 2
    WHEN 'viewer' THEN 1
    ELSE 0
  END;
$$ LANGUAGE sql IMMUTABLE;

-- 예시: tasks 테이블 INSERT 정책 (editor 이상)
CREATE POLICY "tasks_insert_editor"
  ON tasks FOR INSERT
  WITH CHECK (
    is_project_member(project_id, 'editor')
  );
```

## 3. 전체 테이블 RLS 커버리지 설계 (H-2)

### 현재 RLS 미적용 가능 테이블

- `links` — 의존성 구조 전체 노출 위험
- `resources` — 리소스 목록 노출
- `assignments` — 업무 배정 정보 노출
- `costs` — 예산/비용 정보 (민감)
- `baselines` — 기준선 데이터 노출

### 공통 헬퍼 함수 설계

```sql
-- is_project_member: 현재 사용자가 해당 프로젝트의 특정 역할 이상인지 확인
-- min_role: 'viewer' | 'editor' | 'owner'
CREATE OR REPLACE FUNCTION is_project_member(
  p_project_id uuid,
  p_min_role   text DEFAULT 'viewer'
)
RETURNS boolean
LANGUAGE sql
STABLE
SECURITY DEFINER
AS $$
  SELECT EXISTS (
    SELECT 1
    FROM project_members pm
    WHERE pm.project_id = p_project_id
      AND pm.user_id    = auth.uid()
      AND role_level(pm.role) >= role_level(p_min_role)
  )
  OR EXISTS (
    -- 프로젝트 소유자는 project_members 행 없이도 허용
    SELECT 1
    FROM projects p
    WHERE p.id       = p_project_id
      AND p.owner_id = auth.uid()
  );
$$;
```

### pgTAP 테스트 시나리오 5개

```sql
-- supabase/tests/rls_coverage.sql

BEGIN;
SELECT plan(5);

-- 시나리오 1: viewer는 links를 읽을 수 있다
-- Given: user_b가 project_a의 viewer로 등록된 상황
-- When:  user_b 세션으로 links 테이블 SELECT
-- Then:  project_a의 links가 반환됨
SELECT ok(
  (
    SELECT count(*) > 0
    FROM links
    WHERE project_id = :'project_a_id'
  ),
  'viewer can select links in their project'
) SET LOCAL role = :'user_b_role';

-- 시나리오 2: viewer는 links를 삽입할 수 없다
-- Given: user_b가 project_a의 viewer로 등록된 상황
-- When:  user_b 세션으로 links INSERT 시도
-- Then:  RLS 오류로 거부됨
SELECT throws_ok(
  $$ INSERT INTO links(source, target, type, lag, project_id)
     VALUES ('task-1', 'task-2', 'e2s', 0, :'project_a_id') $$,
  '42501',
  'new row violates row-level security policy',
  'viewer cannot insert links'
) SET LOCAL role = :'user_b_role';

-- 시나리오 3: 다른 프로젝트 멤버는 costs를 읽을 수 없다
-- Given: user_c가 project_b의 editor이고 project_a와는 무관한 상황
-- When:  user_c 세션으로 project_a의 costs SELECT 시도
-- Then:  0건 반환 (RLS로 필터링)
SELECT is(
  (SELECT count(*)::int FROM costs WHERE project_id = :'project_a_id'),
  0,
  'non-member cannot read costs from another project'
) SET LOCAL role = :'user_c_role';

-- 시나리오 4: editor는 baselines를 INSERT할 수 있지만 DELETE는 불가
-- Given: user_b가 project_a의 editor인 상황
-- When:  INSERT 성공 후 DELETE 시도
-- Then:  INSERT는 성공, DELETE는 RLS 오류
SELECT lives_ok(
  $$ INSERT INTO baselines(project_id, name, description, data)
     VALUES (:'project_a_id', 'B1', null, '{}') $$,
  'editor can insert baseline'
) SET LOCAL role = :'user_b_editor_role';

SELECT throws_ok(
  $$ DELETE FROM baselines WHERE project_id = :'project_a_id' $$,
  '42501',
  'new row violates row-level security policy',
  'editor cannot delete baselines'
) SET LOCAL role = :'user_b_editor_role';

SELECT * FROM finish();
ROLLBACK;
```

## 4. 공유 링크 보안 설계 (AC-13, AC-17, AC-20 지원)

### shared_links 테이블 스키마

```sql
CREATE TABLE shared_links (
  id          uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id  uuid        NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  token       text        NOT NULL UNIQUE,  -- 128비트 엔트로피 토큰
  created_by  uuid        NOT NULL REFERENCES auth.users(id),
  permission  text        NOT NULL DEFAULT 'viewer'
                          CHECK (permission IN ('viewer')),  -- 공유는 viewer만
  scope_type  text        NOT NULL DEFAULT 'project'
                          CHECK (scope_type IN ('project', 'section')),
  scope_task_ids  uuid[]  DEFAULT NULL,  -- AC-20: 섹션 공유 시 허용 task id 배열
  expires_at  timestamptz DEFAULT (now() + interval '30 days'),
  max_views   int         DEFAULT NULL,  -- NULL = 무제한
  view_count  int         NOT NULL DEFAULT 0,
  is_active   boolean     NOT NULL DEFAULT true,
  created_at  timestamptz NOT NULL DEFAULT now()
);

-- 토큰 조회 성능 인덱스
CREATE INDEX idx_shared_links_token ON shared_links(token) WHERE is_active = true;
```

### 128비트 엔트로피 토큰 생성

UUID v4는 122비트의 랜덤성을 제공하지만, 추가 엔트로피로 안전성을 높인다.

```typescript
// src/lib/sharedLinks.ts

/**
 * 128비트 이상의 엔트로피를 가진 공유 링크 토큰 생성.
 * UUID v4 (122비트) + 6자리 hex (24비트) = 실질적 146비트 엔트로피.
 * URL-safe 형태로 반환.
 */
export function generateShareToken(): string {
  const uuid = crypto.randomUUID().replace(/-/g, '');  // 32자 hex
  const extra = Array.from(crypto.getRandomValues(new Uint8Array(4)))
    .map(b => b.toString(16).padStart(2, '0'))
    .join('');  // 8자 hex
  return `${uuid}${extra}`;  // 총 40자, 160비트
}

export async function createSharedLink(
  projectId: string,
  options: {
    scopeType?: 'project' | 'section';
    scopeTaskIds?: string[];
    expiresInDays?: number;
    maxViews?: number;
  } = {}
): Promise<{ token: string; url: string }> {
  const token = generateShareToken();
  const expiresAt = options.expiresInDays
    ? new Date(Date.now() + options.expiresInDays * 86400000).toISOString()
    : null;

  const { error } = await supabase.from('shared_links').insert({
    project_id:     projectId,
    token,
    scope_type:     options.scopeType ?? 'project',
    scope_task_ids: options.scopeTaskIds ?? null,
    expires_at:     expiresAt,
    max_views:      options.maxViews ?? null,
  });

  if (error) throw error;

  return {
    token,
    url: `${process.env.NEXT_PUBLIC_APP_URL}/share/${token}`,
  };
}
```

### anon 역할 RLS 정책: active 토큰 존재 시만 SELECT 허용

```sql
-- shared_links 자체는 anon 접근 불가 (토큰 목록 열거 방지)
ALTER TABLE shared_links ENABLE ROW LEVEL SECURITY;

CREATE POLICY "shared_links_select_owner"
  ON shared_links FOR SELECT
  USING (created_by = auth.uid() OR is_project_member(project_id, 'owner'));

-- tasks: 유효한 공유 토큰이 있으면 anon도 SELECT 가능
CREATE POLICY "tasks_select_via_share_token"
  ON tasks FOR SELECT
  USING (
    -- 기존 인증 멤버 접근
    is_project_member(project_id, 'viewer')
    OR
    -- 유효한 공유 토큰을 통한 anon 접근
    EXISTS (
      SELECT 1 FROM shared_links sl
      WHERE sl.project_id = tasks.project_id
        AND sl.is_active   = true
        AND (sl.expires_at IS NULL OR sl.expires_at > now())
        AND (sl.max_views  IS NULL OR sl.view_count < sl.max_views)
        -- 실제 토큰 값은 요청 헤더/쿠키에서 검증 (Next.js API Route에서 처리)
        -- RLS 레벨에서는 토큰 헤더를 set_config로 전달
        AND sl.token = current_setting('request.share_token', true)
    )
  );
```

### 섹션 범위 제한 공유 (AC-20): scope_task_ids 배열 필터링

```sql
-- scope_type = 'section'인 경우 scope_task_ids에 포함된 태스크만 노출
CREATE POLICY "tasks_select_via_section_share"
  ON tasks FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM shared_links sl
      WHERE sl.is_active = true
        AND (sl.expires_at IS NULL OR sl.expires_at > now())
        AND sl.token = current_setting('request.share_token', true)
        AND (
          -- 전체 프로젝트 공유
          sl.scope_type = 'project' AND sl.project_id = tasks.project_id
          OR
          -- 섹션 공유: scope_task_ids에 task.id가 포함
          sl.scope_type = 'section' AND tasks.id = ANY(sl.scope_task_ids)
        )
    )
  );
```

### Next.js API Route: 토큰 검증 및 set_config 전달

```typescript
// src/app/api/share/[token]/route.ts

import { createClient } from '@supabase/supabase-js';
import { NextRequest, NextResponse } from 'next/server';

export async function GET(
  req: NextRequest,
  { params }: { params: { token: string } }
) {
  const { token } = params;

  // 토큰 길이 기본 검증 (40자 hex)
  if (!/^[0-9a-f]{40}$/.test(token)) {
    return NextResponse.json({ error: 'Invalid token' }, { status: 400 });
  }

  // set_config를 통해 토큰을 RLS 정책에 전달
  const supabase = createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  );

  await supabase.rpc('set_share_token', { token });

  const { data, error } = await supabase.from('tasks').select('*');

  if (error) return NextResponse.json({ error: error.message }, { status: 403 });

  // 조회 횟수 증가 (별도 RPC로 처리)
  await supabase.rpc('increment_share_view_count', { p_token: token });

  return NextResponse.json(data);
}
```

### 열거 공격(Enumeration Attack) 방지

- 토큰 길이: 40자 hex = 160비트 랜덤 공간. 초당 10억 회 시도해도 전체 공간 탐색에 $2^{160} / 10^9$초 소요 (천문학적).
- 토큰 조회 실패 시 응답 시간을 균일하게 유지 (타이밍 어택 방지): 존재하지 않는 토큰도 동일한 처리 경로.
- `shared_links` 테이블 자체는 anon에게 SELECT 불가 → 토큰 목록 열거 불가.
- Rate limiting: Supabase Edge Function 또는 Vercel Middleware에서 토큰 조회 IP당 분당 10회 제한.

// src/middleware.ts

import { createServerClient } from '@supabase/ssr';
import { NextResponse, type NextRequest } from 'next/server';

export async function middleware(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request });

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll();
        },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value }) =>
            request.cookies.set(name, value)
          );
          supabaseResponse = NextResponse.next({ request });
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          );
        },
      },
    }
  );

  // refresh token이 유효하면 access token 자동 갱신
  // IMPORTANT: getUser()는 서버에서 토큰을 검증하므로 getSession()보다 안전
  const { data: { user } } = await supabase.auth.getUser();

  const isAuthRoute = request.nextUrl.pathname.startsWith('/auth');
  const isPublicShare = request.nextUrl.pathname.startsWith('/share');
  const isStaticAsset = request.nextUrl.pathname.startsWith('/_next');

  // 보호된 라우트 접근 시 미인증이면 로그인 페이지로 리다이렉트
  if (!user && !isAuthRoute && !isPublicShare && !isStaticAsset) {
    const url = request.nextUrl.clone();
    url.pathname = '/auth/login';
    return NextResponse.redirect(url);
  }

  return supabaseResponse;
}

export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico).*)',
  ],
};

### SVAR Gantt 인라인 스타일로 인한 CSP 제약

SVAR React Gantt는 내부적으로 인라인 스타일(`style="..."` 속성)과 동적 스타일 삽입을 사용한다. 따라서 엄격한 CSP `style-src 'self'` 적용 시 Gantt 렌더링이 깨진다.

현재 CSP를 `next.config.js`에 바로 추가하면 다음 문제가 발생한다:
- `style-src 'self'` → SVAR Gantt 인라인 스타일 차단
- `style-src 'unsafe-inline'` → XSS 공격 벡터 확대

### 단계적 CSP 도입 계획 (Nonce 기반)

**1단계 (현재): 핵심 헤더 적용, CSP 제외**
- HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy 즉시 적용
- CSP는 Report-Only 모드로 위반 수집만 수행

**2단계 (M-2 달성 후 1개월): Report-Only CSP**

```typescript
// src/middleware.ts에 추가
import { NextResponse } from 'next/server';

function generateNonce(): string {
  const array = new Uint8Array(16);
  crypto.getRandomValues(array);
  return Buffer.from(array).toString('base64');
}

// 미들웨어에서 nonce 생성 및 헤더 삽입
export function middleware(request: NextRequest) {
  const nonce = generateNonce();
  const cspHeader = [
    `default-src 'self'`,
    `script-src 'self' 'nonce-${nonce}'`,
    `style-src 'self' 'nonce-${nonce}' 'unsafe-inline'`,  // SVAR 대응
    `img-src 'self' data: blob:`,
    `connect-src 'self' ${process.env.NEXT_PUBLIC_SUPABASE_URL} wss://*.supabase.co`,
    `font-src 'self'`,
    `report-uri /api/csp-report`,
  ].join('; ');

  const response = NextResponse.next();
  // 위반 수집만, 차단 없음
  response.headers.set('Content-Security-Policy-Report-Only', cspHeader);
  response.headers.set('x-nonce', nonce);
  return response;
}
```

**3단계: SVAR Gantt CSP 호환 확인 후 강제 모드 전환**
- SVAR Gantt 인라인 스타일 목록 파악
- nonce 또는 hash 기반으로 허용 목록 작성
- `Content-Security-Policy-Report-Only` → `Content-Security-Policy`로 전환

## 9. Optimistic UI 롤백 보안 (DBObserver 실패 시)

### 현재 문제

`DBObserver`와 `UIObserver`가 taskEventEmitter를 통해 동시에 이벤트를 수신한다. DB 저장이 실패해도 `UIObserver`는 이미 React state를 갱신한 상태이므로 UI가 성공 상태를 잘못 보여주는 불일치가 발생한다.

### task:rollback 이벤트 설계

```typescript
// src/lib/taskEventEmitter.ts 이벤트 타입 추가

export type TaskEventType =
  | 'task:created'
  | 'task:updated'
  | 'task:deleted'
  | 'task:rollback'  // DB 저장 실패 시 UI 롤백 요청
  | 'link:created'
  | 'link:updated'
  | 'link:deleted';

export interface TaskRollbackPayload {
  taskId: string;
  previousTask: Task;  // DB 저장 전 원본 task 상태
  reason: string;      // 롤백 원인 (에러 메시지)
}
```

### DBObserver 롤백 emit 패턴

```typescript
// src/lib/taskObservers.ts

export class DBObserver {
  handle(event: TaskEvent): void {
    if (event.type === 'task:updated') {
      const { task, previousTask } = event.payload;

      // DB 저장 시도
      updateTask(task)
        .catch((error) => {
          console.error('[DBObserver] task 저장 실패, 롤백 요청:', error);

          // 저장 실패 시 롤백 이벤트 emit
          taskEventEmitter.emit({
            type: 'task:rollback',
            payload: {
              taskId: task.id,
              previousTask,  // 저장 전 원본 상태
              reason: error.message,
            } satisfies TaskRollbackPayload,
            timestamp: Date.now(),
          });
        });
    }
  }
}
```

### UIObserver 롤백 핸들러 패턴

```typescript
// src/lib/taskObservers.ts

export class UIObserver {
  constructor(
    private setTasks: React.Dispatch<React.SetStateAction<Task[]>>,
    private ganttApiRef: React.RefObject<GanttAPI>
  ) {}

  handle(event: TaskEvent): void {
    if (event.type === 'task:updated') {
      const { task } = event.payload;
      this.applyTaskUpdate(task);
    }

    if (event.type === 'task:rollback') {
      const { taskId, previousTask, reason } = event.payload as TaskRollbackPayload;

      console.error(`[UIObserver] 롤백 실행: task ${taskId} - ${reason}`);

      // React state 롤백
      this.setTasks(prev =>
        prev.map(t => t.id === taskId ? previousTask : t)
      );

      // SVAR Gantt 내부 스토어 롤백
      this.ganttApiRef.current?.exec('update-task', {
        id: taskId,
        ...previousTask,
      });

      // 사용자에게 저장 실패 알림 (toast 등)
      // notifyUser('저장 실패: 변경 사항이 롤백되었습니다.');
    }
  }

  private applyTaskUpdate(task: Partial<Task> & { id: string }): void {
    this.setTasks(prev =>
      prev.map(t => t.id === task.id ? { ...t, ...task } : t)
    );
  }
}
```
