# ZeroPM 설계 개요 · 기능

> 문서 버전: v4.0 (2026-06-12 단순화 통합)

이 문서는 `docs/02-설계/`의 진입점이다. 제품 개요, 설계 문서 구성, 현재 구현 상태, 전체 기능 목록(영역별 + 제품 Phase + 구현 상태), 우선순위 요약, UX 설계 원칙을 한곳에 모았다.

---

## 1. 제품 개요

ZeroPM은 MS Project 수준의 스케줄링 기능을 웹으로 제공하는 PM 도구다. 자체 스케줄링 엔진(의존성 기반 날짜 전파, 6종 제약 조건), 고성능 간트 차트, 실시간 협업을 핵심으로 한다. 프리랜서 1인 PM부터 IT 소·중규모 팀(5~30명)까지를 타겟으로 하며, Zero-Cost 인프라(Vercel Hobby + Supabase Free)에서 운영 가능하도록 설계한다.

### 기술 스택

| 영역 | 기술 | 비고 |
|------|------|------|
| 프론트엔드 | Next.js 16 (App Router) / React 19.2 / TypeScript 5 | 개발 포트 3005 |
| 간트 렌더링 | SVAR React Gantt | GPLv3 — 마운트 후 React props 변경 무시 (재배포 시 라이선스 유의) |
| 백엔드/DB | Supabase (Postgres + Auth + RLS + Realtime) | Free 티어 운영 가능 |
| CRDT | Yjs (y-websocket) | 오프라인 우선 동기화 (부분 구현) |
| 스케줄링 엔진 | 자체 구현 (`ganttScheduler.js` + `scheduling.ts`) | 클라이언트 측 실행 |

## 4. 기능 목록

각 기능은 **영역 코드 + 2자리 번호**로 식별한다(예: `SCH-04`). "제품 Phase" 열은 §2의 제품 Phase(1~4)다.

### 4.1 인증 및 보안 (AUTH)

Supabase Auth(인증) + PostgreSQL RLS(접근 제어) + 트랜잭션/낙관적 잠금(무결성)의 다층 방어 체계. RLS를 DB 레벨에서 강제해 모든 접근 경로에서 단일하게 작동시키고, 구조적 변경은 `version` 기반 낙관적 잠금으로 보호한다.

| ID | 기능명 | 요약 | Phase | 상태 |
|----|--------|------|-------|------|
| AUTH-01 | 이메일/비밀번호 인증 | Supabase Auth 계정 생성·로그인 | 1 | 완료 |
| AUTH-02 | 소셜 OAuth 로그인 | Google·GitHub 공급자 | 1 | 완료 |
| AUTH-03 | 행 레벨 보안 (RLS) | 자신이 속한 프로젝트 데이터만 접근. 부가 테이블 커버리지 보강 필요 | 1 | 완료 |
| AUTH-04 | 역할 기반 접근 제어 (RBAC) | `owner`/`editor`/`viewer` 단일 enum, RLS에서 활용 | 1 | 완료 (enum 통일 진행) |
| AUTH-05 | 프로젝트 멤버 초대·관리 | owner가 멤버 초대·역할 할당. `project_invitations`는 설계만 완료 | 2 | 미구현 |
| AUTH-06 | 낙관적 잠금 | 삭제·WBS 이동 등 구조 변경을 `version` 비교로 충돌 감지 | 1 | 완료 |
| AUTH-07 | PostgreSQL 트랜잭션 | 묶인 변경을 RPC 트랜잭션으로 원자 처리. CRUD는 개별 호출 방식 잔존 | 1 | 부분 |

> RBAC 역할: `owner`(삭제·멤버 관리·공유·전 CRUD) / `editor`(태스크·링크·기준선 CRUD) / `viewer`(읽기 전용). RLS 표준 패턴: SELECT=`is_project_member(project_id,'viewer')`, INSERT/UPDATE/DELETE=`is_project_member(project_id,'editor')`(일부 owner 전용). RPC 우회·enum 통일 등 잔여 이슈는 03-로드맵 참조.

### 4.2 실시간 협업 (COL)

`postgres_changes` 기반 크로스탭/멀티유저 동기화 + Yjs(CRDT) 기반 충돌 없는 병합의 하이브리드 전략. OT는 트리(WBS)+그래프(DAG) 구조에서 구현 난이도가 극도로 높아, 연산 순서와 무관하게 수렴하는 CRDT를 채택했다.

| ID | 기능명 | 요약 | Phase | 상태 |
|----|--------|------|-------|------|
| COL-01 | 크로스탭/멀티유저 동기화 | `useRealtimeSync`가 `postgres_changes`로 tasks/links 구독, 에코 중복 제거 후 React·SVAR 스토어 갱신 | 1 | 완료 |
| COL-02 | CRDT 동기화 (Yjs) | Yjs + Realtime으로 동시 편집 자동 병합. 훅은 있으나 GanttView 미통합 | 2 | 부분 |
| COL-03 | 프레즌스 소프트 락 | 편집 중인 Task 실시간 시각화로 충돌 사전 회피. 훅·컴포넌트 존재, UI 미연결 | 2 | 부분 |
| COL-04 | Optimistic UI | 액션 즉시 UI 갱신, 서버 응답 비대기. `ganttIntercepts` + UIObserver | 1 | 완료 |
| COL-05 | 오프라인 지원 | CRDT 상태를 IndexedDB에 영구 저장, 복구 시 자동 동기화 | 3 | 미구현 |
| COL-06 | Realtime Broadcast (P2P 경로) | 임시 상태 교환, 장기 WebRTC P2P 전환으로 메시지 절감 | 4 | 미구현 |

### 4.3 핵심 프로젝트 관리 및 WBS (WBS)

| ID | 기능명 | 요약 | Phase | 상태 |
|----|--------|------|-------|------|
| WBS-01 | 프로젝트 CRUD | `projects` 테이블(`owner_id`, `title`, `calendar_settings`) | 1 | 완료 |
| WBS-02 | 무한 계층형 WBS (Task CRUD) | 인접 리스트(`parent_id`), 트리 조회는 재귀 CTE | 1 | 완료 |
| WBS-03 | WBS 코드 자동 재계산 | 구조 변경 시 1·1.1·1.1.1 계층 코드 재계산 | 2 | 미구현 |
| WBS-04 | 요약 작업 자동 롤업 | 하위 데이터로 상위 Summary 기간·진척·비용 집계. `type` 필드는 있으나 집계 로직 미구현 | 2 | 부분 |
| WBS-05 | 커스텀 태스크 유형 | 기본 외 사용자 정의 유형을 SVAR `taskTypes`로 추가 | 2 | 미구현 |
| WBS-06 | 기준선 (Baseline) 관리 | 최대 11개 기준선 스냅샷 저장. 저장 기능은 있으나 비교 UI 미구현 | 2 | 부분 |
| WBS-07 | 중간 계획 (Interim Plan) | 진행 중 시작/종료일 스냅샷 최대 10개 저장 | 3 | 미구현 |
| WBS-08 | 태스크 의존성 (4종 링크) | FS/SS/FF/SF + Lag/Lead. `links` 테이블에 정규화 저장 | 1 | 완료 |
| WBS-09 | 순환 참조 감지 | 위상 정렬로 사이클 감지·경고. 현재는 전파 시 visited Set 차단만, 명시적 UI 없음 | 1 | 미구현 |
| WBS-10 | 태스크 완료율(%) 입력 | `progress`(0~100) 입력·간트 표시 | 2 | 완료 |
| WBS-11 | 태스크 메모/설명 | `description` 편집·저장 | 2 | 완료 |

### 4.4 스케줄링 엔진 (SCH)

렌더링(SVAR Gantt)과 스케줄링 로직(자체 엔진)을 분리한 **분리형 아키텍처**. 상용 엔진 비용을 피해 CPM을 JS/TS로 자체 구현한다. 현재 엔진은 downstream chain 전파에 집중되어 있고, CPM 전진/후진 계산은 미구현이다(스키마·RPC만 존재).

| ID | 기능명 | 요약 | Phase | 상태 |
|----|--------|------|-------|------|
| SCH-01 | 의존성 기반 자동 전파 | `createRecalculateFunction`이 후행 링크를 재귀 순회, visited Set으로 사이클 방지 | 1 | 완료 |
| SCH-02 | 4종 의존성 계산 (+Lag) | `calculateSuccessorDate(predecessor, successor, linkType, lag=0)` | 1 | 완료 |
| SCH-03 | 제약 조건 처리 | `applyConstraint(task, calculatedStart, calculatedEnd)`. `asap`(기본)/`mso`/`mfo`/`snet`/`fnlt` 완료, `alap` 미완성 | 1 | 부분 |
| SCH-04 | CPM 전진/후진 계산 | ES/EF·LS/LF 산출, `save_cpm_results` RPC 저장. **계산 로직 없음** | 1 | 미구현 |
| SCH-05 | Total Float 계산 | `Total Float = LS − ES`. 계산 로직 없음 | 1 | 미구현 |
| SCH-06 | 임계 경로 시각화 | Float 0 Task를 `is_critical`로 강조. 스키마·타입만 완성 | 1 | 미구현 |
| SCH-07 | 업무일 기반 계산 | 주말·공휴일 제외. **현재 `workingDays=7`(주말 미제외)**, `calendar_settings`로 개선 예정 | 2 | 부분 |
| SCH-08 | 날짜 정규화·UTC | `dateUtils.ts` UTC 헬퍼. end 배타적, 표시 시 `start+duration−1`을 23:59로. start는 00:00 정규화 | 1 | 완료 |
| SCH-09 | SVAR 스토어 직접 갱신 | 프로그래매틱 업데이트는 `ganttApiRef.current.exec(...)`로 직접 적용 | 1 | 완료 |
| SCH-10 | Web Worker 비동기 계산 | CPM/RCPSP를 백그라운드 스레드 실행, 결과만 전달 | 2 | 미구현 |
| SCH-11 | DAG 그래프 모델링 | `@dagrejs/graphlib` 도입. 현재는 링크 배열 직접 순회 | 3 | 미구현 |

> 제약 조건: `asap`(기본)/`alap`(미완성)/`mso`/`mfo`/`snet`/`fnlt`. 엔진 내부 종료일은 **배타적(exclusive)**.

### 4.5 자원 및 비용 관리 (RES)

RCPSP는 NP-hard이므로 자동 평준화 대신 **충돌 감지 + 휴리스틱 조정 지원**을 목표로 한다. **관련 테이블(`resources`/`assignments`/`costs`/`baseline_tasks`)은 스키마만 존재하며 아래 기능은 전부 미구현**이다.

| ID | 기능명 | 요약 | Phase | 상태 |
|----|--------|------|-------|------|
| RES-01 | 자원 정의 | 인력·장비 등 갱신 가능한 자원 정의 | 4 | 미구현 (스키마만) |
| RES-02 | 자원 할당·단위 설정 | Task에 자원 할당, 50%/100% 단위 요구량 | 4 | 미구현 (스키마만) |
| RES-03 | 담당자 배정 | 단순 담당자 배정 표시 (과부하 경고 없음) | 3 | 미구현 |
| RES-04 | 실시간 자원 충돌 감지 | 가용 용량 초과 시점 감지·피드백 | 4 | 미구현 |
| RES-05 | 휴리스틱 조정 지원 | LFT·GRD 우선순위 정보 제공 | 4 | 미구현 |
