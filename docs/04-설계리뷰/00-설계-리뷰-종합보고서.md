**분석 일자**: 2026-06-11
**분석 대상**: ZeroPM 웹 기반 PM 도구 전체 설계
**분석 관점**: PM/기획, UX/인터랙션, 소프트웨어 아키텍처, 데이터베이스, 보안 (5개 전문 관점)

## Executive Summary (경영진 요약)

### 전체 설계 성숙도 평가

| 관점 | 성숙도 등급 |
|------|------------|
| 소프트웨어 아키텍처 | B+ |
| 데이터베이스 설계 | B |
| 보안 | C+ |
| PM/기획 | C |
| UX/인터랙션 설계 | D+ |
| **종합** | **C+** |

ZeroPM은 기술 엔지니어링 기반이 탄탄한 프로젝트다. 이벤트 기반 아키텍처, View/Brain 분리, CPM 스케줄링 엔진의 순수 함수 설계, RLS 기반 접근 제어 등은 시장 출시 수준의 제품을 만들 수 있는 토대를 갖추고 있다. 그러나 사용자 중심 설계(UX, 페르소나, 수용 기준)와 운영 안전성(보안 커버리지, 에러 모니터링, 테스트 자동화) 측면에서는 즉각적인 보완이 필요하다.

### 가장 중요한 강점 3가지

1. **View/Brain 분리와 이벤트 기반 아키텍처**: SVAR Gantt 렌더링과 스케줄링 엔진이 명확히 분리되어 있고, `taskEventEmitter` 싱글턴을 통한 DB 저장/UI 갱신/스케줄링 전파의 관심사 분리가 잘 구현되어 있다. 이는 기능 확장과 유지보수성 모두에 기여하는 구조적 강점이다.
2. **CPM 스케줄링 엔진의 완성도**: `scheduling.ts`의 순수 함수 설계, `visited Set` 기반 사이클 방지, stale React state 우회를 위한 직접 값 전달 전략은 실용적 문제 해결의 깊이를 보여준다. 단위 테스트로 검증된 비즈니스 로직이 코어에 자리잡고 있다.
3. **Supabase Auth + RLS 기반 보안 골격**: 인증과 접근 제어의 기본 틀이 검증된 인프라 위에 구축되어 있으며, pgTAP 기반의 보안 테스트가 존재한다는 점은 보안 품질에 대한 의식이 있음을 보여준다.

### 가장 시급한 개선사항 3가지

1. **낙관적 잠금 실제 적용**: `update_task_with_version` RPC가 구현되어 있으나 `tasks.ts`의 `updateTask()`가 이를 우회하고 있다. 다중 사용자 환경에서 Lost Update 문제가 현재 진행 중인 버그다.
2. **RLS 전체 테이블 커버리지 확보**: `links`, `resources`, `assignments`, `costs` 등 부가 테이블의 RLS 정책 적용 여부가 불명확하며, `UserRole`과 `project_members.role`의 역할 enum 불일치가 의도치 않은 접근 허용/차단으로 이어질 수 있다.
3. **DB 저장 실패 시 UI 블로킹 문제**: `emit()`이 `Promise.all`로 모든 리스너를 병렬 await하는 구조에서 `DBObserver` 실패가 `UIObserver`도 차단한다. 에러 경계 분리가 없는 상태로 프로덕션 운영 시 사용자 경험이 크게 저하된다.

### 즉시 조치 필요 리스크

- `links`, `resources`, `costs` 테이블에 RLS가 미적용된 경우 타 프로젝트 데이터 전체 노출 가능성이 있다. 즉각 점검이 필요하다.
- Service Role Key의 서버 전용 격리 여부를 즉시 확인해야 한다. 단 한 곳이라도 클라이언트 번들에 포함되면 데이터베이스 전체 접근 권한이 노출된다.
- `tasks(project_id)`, `links(source)`, `links(target)` 인덱스가 없어 현재도 모든 프로젝트 로드 쿼리가 시퀀셜 스캔으로 실행되고 있다.

## 종합 강점 분석

다섯 개 관점의 분석에서 공통적으로 평가된 강점은 다음과 같다.

**아키텍처 설계의 방향성 정합성**: 이벤트 기반 설계, 순수 함수 스케줄링 엔진, 실시간 동기화의 echo 필터링 등 핵심 기술 결정이 문제 도메인에 잘 맞는다. 아키텍처 문서가 코드와 실제로 일치하는 수준으로 관리되고 있으며, 기술 부채의 위치가 명확히 식별되어 있다.

**데이터 모델의 도메인 적합성**: tasks, links, resources, assignments, costs, baselines 등 PM 도메인의 핵심 엔티티가 3NF를 준수하며 설계되어 있다. FK + CASCADE 전략과 Supabase Auth 연동도 일관성 있게 적용되어 있다.

**비즈니스 로직의 테스트 가능성**: `scheduling.ts`의 순수 함수 분리 덕분에 `scheduling.test.ts`, `scheduling_scenario.test.ts`, `ganttScheduler.test.js`가 핵심 계산 로직을 실질적으로 커버한다. 이는 View/Brain 분리의 직접적인 효과다.

**실용적인 기술 선택**: Supabase Free Tier + Vercel Hobby Plan 조합이 초기 단계에서 합리적이며, SVAR Gantt를 마운트 후 직접 API로 제어하는 접근은 라이브러리 제약을 현실적으로 해결한 방법이다.

|---|------|------|--------|-----------|

| H-2 | 보안 | `links`, `resources`, `assignments`, `costs` 등 부가 테이블 RLS 커버리지 불명확 | 타 프로젝트 데이터 전체 노출 가능 | 모든 테이블에 SELECT/INSERT/UPDATE/DELETE RLS 정책 명시, pgTAP 테스트로 커버 |

## 권고 로드맵

### 1개월 이내 (즉시 안정화)

이 단계의 목표는 현재 동작 중인 버그와 보안 취약점을 제거하고, 프로덕션 운영의 최소 기준을 확보하는 것이다.

- 핵심 인덱스 6개 추가(`CONCURRENTLY` 옵션 사용)
- `updateTask()` 낙관적 잠금 RPC 전환
- 모든 테이블 RLS 커버리지 전수 점검 및 보완
- Service Role Key 환경 변수 분류 점검
- `UserRole` enum 단일 체계 통일
- `console.log` 잔류 코드 제거
- `constraint_date` 컬럼 `TIMESTAMPTZ` 통일 마이그레이션
- `emit()` 에러 경계 분리 (DB 실패와 UI 갱신 독립화)
- GitHub Actions 기본 CI 구성 (test + lint + npm audit)
- 보안 HTTP 헤더 설정

### 3개월 이내 (아키텍처 견고화)

이 단계의 목표는 팀 확장과 기능 개발 가속을 위한 구조적 기반을 갖추는 것이다.

- `GanttView.jsx` 계층 분리 (상태/핸들러/렌더링)
- `ganttBridge.js` SVAR API 어댑터 레이어 신설
- `useGanttObservers`, `useRealtimeSync` 단위 테스트 작성
- `save_cpm_results` 배치 UPDATE 구조 전환
- 소프트 삭제 컬럼 추가 (`deleted_at`)
- JWT 만료 및 토큰 무효화 전략 구현
- 충돌 해결 UI 및 오프라인 상태 배너 설계 및 구현

## 결론

ZeroPM은 기술적 토대가 탄탄하게 구축된 프로젝트다. 이벤트 기반 아키텍처와 CPM 스케줄링 엔진의 완성도, Supabase 기반의 실시간 협업 구조는 시장의 실제 문제를 해결할 수 있는 수준의 시스템을 만들 수 있는 기반이다.

그러나 5개 관점의 교차 분석을 통해 명확해진 것은 현재 프로젝트가 **엔지니어링 문서로서는 완성도가 높으나, 제품 문서로서는 사용자와의 연결 고리가 부재하다**는 점이다. 페르소나, 사용자 스토리, 수용 기준, 화면 흐름도가 없다는 것은 기능이 "무엇인가"는 정의되어 있지만 "누구를 위해 왜 만드는가"가 정의되지 않았다는 의미다. 이 공백은 팀이 확장될수록 커뮤니케이션 비용과 방향성 혼선으로 직결된다.

즉각적으로 조치해야 할 두 가지 우선순위는 다음과 같다. 첫째, 낙관적 잠금 우회 버그와 RLS 커버리지 불확실성을 포함한 데이터 무결성 및 보안 취약점을 제거하는 것이다. 이는 현재 동작 중인 문제이며 기능 개발보다 선행되어야 한다. 둘째, DB 저장 실패 시 UI 블로킹 문제를 해소하여 프로덕션 안정성의 기본 기준을 확보하는 것이다.

기술은 준비되어 있다. 다음 단계는 그 기술을 실제 사용자 문제에 연결하는 작업이며, 위 로드맵의 단계적 실행을 통해 ZeroPM이 기술적 완성도와 제품적 완성도를 동시에 갖춘 도구로 발전할 수 있을 것이다.
