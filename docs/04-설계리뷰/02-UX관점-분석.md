# ZeroPM UX/인터랙션 설계 심층 분석 보고서

**작성일:** 2026-06-11
**분석 대상:** ZeroPM 기획/설계 문서 전체
**분석 관점:** 시니어 UX 디자이너 / 인터랙션 설계 전문가

---

## 요약 (Executive Summary)

ZeroPM의 기술 아키텍처는 매우 정교하게 설계되어 있으나, 사용자 경험 설계 관점에서는 **기술 중심의 문서화 편향**이 두드러진다. 핵심 PM 기능(간트 차트, 스케줄링, 실시간 협업)의 구현 방식은 상세히 정의된 반면, 사용자가 실제로 화면을 어떻게 탐색하고, 어떤 상황에서 어떤 피드백을 받는지에 대한 설계가 체계적으로 부재하다. 이는 향후 사용성 문제로 직결될 가능성이 높으며, 특히 MS Project 급의 복잡한 기능을 처음 접하는 사용자에게 높은 진입 장벽이 될 수 있다.

## 1. 사용자 경험 설계 완성도

### 현황 평가

| 항목 | 존재 여부 | 비고 |
|---|---|---|
| 화면 흐름도 (User Flow / Screen Flow) | **부재** | - |
| 와이어프레임 / 목업 | **부재** | - |
| 정보 구조 (IA) | **부분적** | 기능 목록 수준에 그침 |
| 온보딩 플로우 | **부재** | - |

### 문제점

**화면 흐름도 부재**는 이 프로젝트의 가장 큰 UX 설계 공백이다. 현재 문서는 "어떤 기능이 있는가"는 정의하고 있으나, "사용자가 목표를 달성하기 위해 어떤 경로로 이동하는가"는 정의되어 있지 않다. 예를 들어, 신규 사용자가 프로젝트를 생성하고 첫 번째 태스크를 등록한 뒤 팀원을 초대하는 경로, 또는 기존 사용자가 지연된 태스크를 발견하고 일정을 재조정하는 시나리오가 화면 수준에서 명세되어 있지 않다.

**정보 구조 측면**에서 기능 분류(대시보드, 보고서, 간트, 칸반, 캘린더)는 나열되어 있으나, 이들 간의 탐색 체계(글로벌 내비게이션, 사이드바 구조, 컨텍스트 전환 방식)가 정의되어 있지 않다. 특히 간트 뷰 / 칸반 뷰 / 캘린더 뷰 간의 전환이 탭 방식인지, 페이지 이동인지, 같은 데이터의 다른 표현인지조차 명시되지 않았다.

**온보딩 부재**는 치명적이다. ZeroPM은 CPM, WBS, 제약 조건, 기준선 등 일반 사용자에게 낯선 전문 용어와 개념을 다수 포함한다. 이에 대한 점진적 노출(Progressive Disclosure) 전략이나 빈 상태(Empty State) 설계가 없으면, 첫 사용 경험에서 이탈률이 높아질 것이다.

### 개선 권고사항

- 핵심 사용자 시나리오 최소 3개(프로젝트 생성 → 첫 간트 구성, 일정 지연 대응, 팀원 협업)에 대한 User Flow 작성
- 메인 화면(간트 뷰)에 대한 로우파이 와이어프레임 제작. 좌측 WBS 패널, 우측 타임라인, 툴바 영역의 레이아웃 확정이 선행되어야 개발 방향이 명확해짐
- 빈 프로젝트 상태(Empty State)에 대한 UI 설계: "첫 번째 태스크를 추가해보세요" 수준의 가이드와 샘플 데이터 제공 옵션 고려
- 온보딩 체크리스트 또는 인터랙티브 튜토리얼 설계. 특히 "링크 생성으로 의존성 연결하기", "제약 조건 설정하기" 같은 고급 기능에 대한 컨텍스트 기반 힌트(Tooltip / Coach Mark) 설계 필요

## 2. 인터랙션 설계 평가

### 2-1. 간트 차트 인터랙션

#### 현황 평가

SVAR React Gantt를 활용한 드래그, 리사이즈, 링크 생성 인터랙션의 기술적 구현(인터셉터, 이벤트 이미터, 스케줄링 전파)은 상세히 설계되어 있다. 그러나 이 인터랙션들의 **사용자 피드백 레이어(Feedback Layer)**가 정의되어 있지 않다.

#### 문제점

- **드래그 중 시각적 피드백 부재:** 사용자가 태스크 바를 드래그할 때, 이동 가능한 날짜 범위 표시, 스냅(Snap) 단위(일/주/월), 충돌 발생 예상 구역 하이라이팅 등의 피드백 설계가 없다.
- **스케줄링 전파 결과 가시화 미정:** 한 태스크의 날짜를 변경했을 때 후행 태스크들이 연쇄적으로 이동하는 "캐스케이드 효과"가 사용자에게 어떻게 표시되는지 설계되어 있지 않다. 갑자기 여러 태스크가 이동하면 사용자는 혼란을 느낄 수 있다.
- **링크 생성 UX 미정:** 태스크 간 의존성 연결 시, 어떤 핸들(Handle)을 드래그하는지, 연결 가능한 대상이 하이라이팅되는지, 연결 완료 시 확인 피드백이 어떻게 제공되는지 정의되어 있지 않다.
- **실행 취소(Undo/Redo) 설계 부재:** 잘못된 드래그나 링크 생성을 되돌릴 수 있는 Undo/Redo 기능의 UX 설계가 전혀 언급되지 않았다. 이는 PM 도구에서 사용자가 가장 빈번하게 필요로 하는 안전장치다.

#### 개선 권고사항

- 드래그 조작 중 날짜 툴팁(Tooltip) 실시간 표시 설계 (예: "2026-07-01 → 2026-07-15, 15일")
- 캐스케이드 이동 시 영향받는 태스크를 일시적으로 강조 표시 후 정착하는 애니메이션 설계
- Ctrl+Z / Ctrl+Y 키보드 단축키와 함께 툴바에 Undo/Redo 버튼 배치

### 2-2. 오프라인 모드 및 충돌 UX

#### 현황 평가

낙관적 잠금(Optimistic Locking)의 기술적 구현(`version` 필드)은 정의되어 있으나, 충돌이 실제로 발생했을 때 **사용자에게 어떤 UI를 보여줄지**에 대한 설계가 전무하다.

#### 문제점

- **충돌 해결 UI 미정:** 두 사용자가 동시에 같은 태스크를 수정했을 때 버전 충돌이 발생하면, 사용자에게 어떤 다이얼로그나 인디케이터를 보여줄 것인가? "내 변경 사항 유지", "서버 버전으로 덮어쓰기", "두 버전 비교" 중 어떤 옵션을 제공할 것인가?
- **오프라인 상태 인식 UI 부재:** 네트워크 단절 시 사용자가 오프라인 상태임을 인지하고, 로컬에서 작업한 내용이 나중에 동기화될 것임을 안내하는 배너 또는 상태 인디케이터 설계가 없다.
- **동기화 실패 시 처리:** Supabase 저장 실패 시 사용자에게 어떤 에러 메시지를 보여주고, 재시도 옵션을 어떻게 제공하는지 정의되어 있지 않다.

#### 개선 권고사항

- 충돌 발생 시 비파괴적(Non-destructive) 해결 다이얼로그 설계: 변경 내용을 나란히 비교하고 선택할 수 있는 UI
- 상단 고정 배너로 오프라인/동기화 중/동기화 완료 상태를 명확히 표시
- 저장 실패 시 Toast 알림 + 재시도 버튼 제공, 실패한 변경 사항을 로컬에 임시 보존하는 UX 흐름 설계

### 2-3. 실시간 협업 UX

#### 현황 평가

Supabase Presence로 "누가 편집 중인지" 표시하는 소프트 락(Soft Lock)의 기술적 기반은 정의되어 있다. 그러나 이 정보를 **화면에서 어떻게 시각화할 것인지**에 대한 UX 설계가 없다.

#### 문제점

- **프레즌스 표시 방식 미정:** 다른 사용자가 편집 중인 태스크 바에 아바타 아이콘을 오버레이할 것인지, 컬러 테두리로 구분할 것인지, 호버 시 팝오버로 표시할 것인지 결정되어 있지 않다.
- **커서/포커스 위치 공유 없음:** Google Docs 수준의 실시간 커서 위치 공유는 계획에 있으나(Yjs CRDT 델타 교환), 간트 차트에서의 구체적 표현 방식(어떤 셀/바에 누가 포커스를 두고 있는지)이 정의되지 않았다.
- **편집 잠금 상태의 UX 처리:** 다른 사용자가 편집 중인 태스크를 내가 클릭했을 때, 단순히 "잠금" 아이콘만 보여줄 것인지, 읽기 전용으로 열 것인지, 아니면 강제 편집을 허용하되 충돌 처리를 후처리할 것인지에 대한 정책이 없다.

#### 개선 권고사항

- 사용자별 고유 컬러를 할당하고, 해당 사용자가 편집 중인 태스크 바에 컬러 테두리 + 아바타 뱃지 표시
- 협업 패널(우측 사이드바 또는 플로팅 패널)로 현재 프로젝트에 접속 중인 팀원 목록 및 각자의 포커스 위치 표시
- 소프트 락 상태에서 편집 시도 시 "OOO님이 현재 편집 중입니다. 계속 편집하시겠습니까?" 확인 다이얼로그 제공

### 2-4. 에러 상태 및 로딩 상태 UX

#### 현황 평가

로딩/에러 상태 설계가 문서 어디에도 언급되지 않았다. `console.error`만 남아 있다는 코드 주석이 유일한 에러 처리 언급이다.

#### 문제점

- 대용량 프로젝트(수천 개 태스크) 초기 로딩 시 **스켈레톤 UI(Skeleton Screen)** 또는 진행률 표시 설계 부재
- 스케줄링 재계산(캐스케이드) 중 UI 블로킹 방지는 Web Worker로 처리되나, 계산 중임을 사용자에게 알리는 **비주얼 인디케이터** 설계 없음
- API 오류, 권한 오류, 네트워크 타임아웃 등 다양한 에러 케이스에 대한 에러 메시지 분류 체계 없음

#### 개선 권고사항

- 간트 차트 초기 로딩 시 태스크 바 영역에 Skeleton Screen 적용
- 스케줄링 재계산 중 상태 바(Status Bar) 또는 미세한 프로그레스 인디케이터 표시 (사용자 조작은 계속 허용)
- 에러 메시지 분류 체계 수립: 정보성(Info) / 경고(Warning) / 오류(Error) / 치명적(Fatal) 4단계로 구분하여 Toast, 인라인 메시지, 모달을 각각 적용

## 3. 접근성(Accessibility) 설계

### 현황 평가

접근성 관련 내용이 기획/설계 문서 전체에 단 한 줄도 언급되지 않았다. 이는 엔터프라이즈 PM 도구로서 심각한 설계 공백이다.

### 문제점

**WCAG 2.1 준수 계획 부재:** 공공기관이나 대기업 도입을 염두에 둔다면 AA 수준 준수는 법적 요구사항이 될 수 있다. 특히 색상 대비, 포커스 가시성, 키보드 조작 가능성은 기본 요건이다.

**간트 차트 키보드 내비게이션 미설계:** 간트 차트는 본질적으로 마우스 중심 인터랙션(드래그, 리사이즈)으로 설계되어 있다. SVAR Gantt의 키보드 접근성 지원 수준을 확인하고, 부족한 부분을 보완하는 설계가 필요하나 언급이 없다. 예를 들어, 화살표 키로 태스크 이동, Enter로 편집 진입, Tab으로 다음 태스크 이동 등의 키보드 패턴이 정의되어야 한다.

**Critical Path 색상 전용 표시 문제:** 임계 경로 태스크를 "빨간색"으로 강조하는 방식은 적녹 색각 이상자(전체 남성의 약 8%)에게 식별이 불가능하다. 색상 외에 패턴, 아이콘, 텍스트 레이블을 병용하는 대안 설계가 없다.

**스크린 리더 지원:** 간트 차트의 시각적 타임라인 정보를 스크린 리더 사용자에게 어떻게 전달할 것인지(ARIA 레이블, 라이브 리전, 대체 텍스트 테이블 뷰 등) 설계되어 있지 않다.

### 개선 권고사항

- WCAG 2.1 AA를 목표 기준으로 명시하고, 각 컴포넌트 개발 시 접근성 체크리스트 적용
- Critical Path 표시에 색상 + 아이콘(경고 심볼) + 텍스트 뱃지 3중 병용 설계
- 간트 차트 좌측 WBS 그리드 영역에 대한 키보드 내비게이션 패턴 정의 (마우스와 동등한 기능 접근 보장)
- 접근성 전용 "목록 뷰" 전환 옵션 제공: 간트의 시각적 표현을 HTML 테이블로 대체 표시하여 스크린 리더 호환성 확보

## 4. 반응형 및 다디바이스 지원

### 현황 평가

문서 전체에 반응형 설계, 모바일 지원에 대한 언급이 전무하다. 기술 스택(Next.js, SVAR Gantt)은 데스크톱 브라우저를 기본 타겟으로 설계된 것으로 보인다.

### 문제점

**간트 차트의 모바일 적합성:** 간트 차트는 넓은 수평 공간과 정밀한 드래그 조작이 필요한 UI다. 스마트폰 화면(320~430px)에서의 경험은 본질적으로 데스크톱과 동일할 수 없다. 이에 대한 전략이 없다면, 모바일 접속 시 사용성이 매우 낮은 화면을 제공하게 된다.

**태블릿 인터랙션 미정의:** 태블릿(768~1024px)은 터치 기반이면서 상대적으로 넓은 화면을 제공한다. 드래그 인터랙션의 터치 이벤트 처리(Touch Event vs. Pointer Event), 핀치 줌으로 타임라인 스케일 조정 등의 설계가 없다.

**반응형 레이아웃 분기점 미정:** 데스크톱 / 태블릿 / 모바일 각 브레이크포인트에서 레이아웃이 어떻게 변화하는지(사이드바 접힘, 패널 축소, 뷰 전환 등) 정의되어 있지 않다.

### 개선 권고사항

- **모바일 전략 명시:** "모바일에서는 간트 대신 칸반/목록 뷰를 기본으로 제공하고, 간트는 가로 스크롤 읽기 전용으로 제공"하는 식의 명확한 정책 결정 필요
- 태블릿에서는 Pointer Events API를 활용한 터치 드래그 지원, 핀치 줌으로 타임라인 줌 레벨 조정 설계
- 모바일 전용 태스크 상세 편집 화면(풀스크린 모달 또는 바텀 시트) 설계: 간트 바 탭 시 상세 편집으로 진입

---

## 5. UX 품질 지표

### 현황 평가

사용성 테스트 계획이나 UX 측정 지표가 기획 문서에 정의되어 있지 않다. 구현 계획(Implementation Plan)도 기술적 완성도 체크리스트만 존재하며, 사용자 검증 단계가 포함되어 있지 않다.

### 문제점

- **사용성 테스트 계획 없음:** 각 Phase 완료 후 실제 사용자 대상 검증 단계가 없으면, 기술적으로 완성되어도 사용하기 어려운 제품이 될 수 있다.
- **UX 메트릭 미정의:** 어떤 기준으로 UX가 "충분히 좋다"고 판단할 것인지 기준이 없다. Task Success Rate, Time-on-Task, Error Rate, System Usability Scale(SUS) 점수 등의 기준 없이는 UX 개선 우선순위를 결정하기 어렵다.
- **분석 도구 계획 없음:** 실제 사용 패턴을 파악하기 위한 행동 분석 도구(Hotjar, Mixpanel, PostHog 등) 도입 계획이 없다.

### 개선 권고사항

- Phase 3(Core UI) 완료 후 내부 사용성 테스트 5명 이상 진행, "간트에서 두 태스크 간 의존성 연결하기" 등 핵심 태스크 성공률 측정
- 목표 SUS 점수 70점 이상(Good 수준) 설정
- 오픈소스 분석 도구(PostHog 셀프호스팅)를 활용한 주요 기능 클릭 이벤트 수집 계획 수립

## 결론

ZeroPM은 기술적 완성도 면에서 MS Project에 견줄 수 있는 수준을 지향하고 있으나, 현재 기획/설계 문서는 **엔지니어링 스펙**에 치우쳐 있으며 **사용자 경험 스펙**은 사실상 미착수 상태다. 특히 Undo/Redo 부재, 충돌 해결 UI 미정, 온보딩 부재는 기술 구현이 완성되어도 제품 사용성을 크게 저해할 요소들이다.

권고 우선순위 H1~H4 항목(User Flow, 와이어프레임, Undo/Redo, 충돌 UI)은 Phase 3 UI 구현 착수 전에 반드시 완성되어야 하며, 이를 선행하지 않으면 기술적으로 올바른 구현이 사용자에게는 혼란스러운 경험으로 전달되는 역설적 상황이 발생할 수 있다.
