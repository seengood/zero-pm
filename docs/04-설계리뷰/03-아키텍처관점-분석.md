## 1. 아키텍처 패턴 평가

### 1.1 이벤트 기반 아키텍처(EDA)의 적절성과 완성도

**현황 평가**

설계 의도는 훌륭합니다. `taskEventEmitter` 싱글턴이 DB 저장(`DBObserver`), UI 갱신(`UIObserver`), 스케줄링 전파(`ScheduleObserver`)를 분리하는 Observer 패턴은 이 도메인에 적합한 선택입니다. `useGanttObservers.js`의 cleanup 처리(`unsubscribers.forEach(unsub => unsub())`)도 올바르게 구현되어 있습니다.

**문제점**

- **emit이 fire-and-forget이 아니다**: `emit()`이 `Promise.all`로 모든 리스너를 병렬 await합니다. `DBObserver`가 Supabase 네트워크 요청 중 실패하면 `UIObserver`도 reject되어 UI 업데이트가 막힙니다. DB 저장 실패가 UI를 멈추는 결합이 남아 있습니다.
- **이벤트 순서 보장 불가**: `link:created` 핸들러에서 `DBObserver.handleLinkCreated` → `UIObserver` → `ScheduleObserver` 순으로 await하지만, `DBObserver`가 Supabase에서 새 `link.id`를 받아 `link.id`를 변이(mutation)시키는 구조입니다(`link.id = data.id`). 이 변이는 공유 객체를 직접 수정하는 것으로 추후 레이스 컨디션의 원인이 될 수 있습니다.
- **console.log가 프로덕션 코드에 잔류**: `taskEventEmitter.ts` 52, 56번 줄과 `ganttIntercepts.js` 37, 58, 67, 80번 줄에 `console.log`가 그대로 남아 있습니다. CLAUDE.md에는 디버그 로그 제거 완료라고 명시되어 있으나 실제 코드와 불일치합니다.

**개선 권고**

```
// DB 실패가 UI를 막지 않도록 관심사 분리
const promises = [
    dbObserver.handleTaskUpdated(event).catch(e => console.error('[DB]', e)),
    uiObserver.handleTaskUpdated(event),  // 항상 성공
];
await Promise.all(promises);
```

DB 저장 실패 시 UI를 낙관적으로 유지하고 별도 에러 토스트를 표시하는 분기가 필요합니다.

### 1.2 View/Brain 분리 설계의 강점과 약점

**강점**

`scheduling.ts`의 `calculateSuccessorDate`, `applyConstraint`는 순수 함수로 작성되어 테스트가 매우 용이합니다. SVAR Gantt가 렌더링 전용으로 격리되어 있어 Gantt 라이브러리 교체 시 비즈니스 로직 보호가 가능합니다.

**약점**

분리가 완전하지 않습니다. `GanttView.jsx`가 529줄로, 상태 관리(tasksRef, linksRef), 이벤트 emit, 스케줄링 함수 생성(`useMemo`), 옵저버 초기화, 실시간 동기화, Gantt 렌더링까지 모두 담당합니다. Brain과 View의 경계가 `GanttView.jsx` 안에서 혼재합니다. `processedTasks`에서 `applyConstraint`를 직접 호출하는 것도 View 계층이 스케줄링 로직에 의존하는 결합입니다.

**개선 권고**

`GanttView.jsx`를 다음으로 분리하는 것을 권고합니다.

- `useGanttState.js`: tasks/links 상태 + refs 관리
- `useGanttEventHandlers.js`: handleTaskCreate/Update/Delete/Link 핸들러
- `GanttView.jsx`: 렌더링 전용 (200줄 이하 목표)

---

### 1.3 싱글턴 EventEmitter의 리스크

**메모리 누수 리스크**

`useGanttObservers.js`의 `useEffect` 의존성 배열이 `[tasksRef, linksRef, setTasks, setEditingTask, setLinks, recalculateAffectedTasks]`로 설정되어 있습니다. `recalculateAffectedTasks`는 `useMemo([], [])`로 안정화되어 있으나, `setTasks`, `setEditingTask`, `setLinks`는 React가 참조 안정성을 보장하지 않는 경우 effect가 재실행되어 이전 리스너가 정리되지 않고 누적될 수 있습니다. (단, `useState`의 setter는 실제로는 안정적이지만 의존성 배열에 포함되면 eslint-plugin-react-hooks 경고를 유발합니다.)

**이벤트 순서 리스크**

Hot Reload(개발 환경) 시 싱글턴이 모듈 캐시에 살아있어 리스너가 중복 등록될 수 있습니다. 현재는 cleanup이 올바르게 구현되어 있어 실제 누수로 이어지지 않지만, 프로덕션 빌드에서 `<React.StrictMode>`가 활성화된 경우 effect가 두 번 실행되어 리스너가 두 개 등록되었다가 하나만 정리되는 시나리오가 가능합니다.

**개선 권고**

싱글턴 유지는 합리적이지만, 개발 환경 디버깅을 위해 `getListenerCount()` 메서드를 추가하여 중복 등록 여부를 런타임에 확인할 수 있도록 하는 것이 유용합니다.

---

### 1.4 모노레포 vs 마이크로서비스 전략

현재 모노레포(단일 Next.js 앱) 구조는 **현 단계에서 올바른 선택**입니다. 팀 규모와 기능 범위를 고려할 때 마이크로서비스 전환은 오버엔지니어링입니다. 다만 `src/lib/` 내 파일들이 JS/TS 혼재(`ganttScheduler.js`, `scheduling.ts`)하여 경계가 불명확합니다. 향후 스케줄링 엔진을 Web Worker로 이전할 때 이 경계를 패키지 단위로 분리하는 것이 자연스러운 진화 경로입니다.

## 2. 확장성 및 성능 설계

### 2.1 클라이언트 측 스케줄링 엔진의 확장 한계

**현황**

`createRecalculateFunction`은 후행 작업을 재귀 DFS로 순회합니다. `visited Set`으로 사이클을 방지하고, stale React state 문제를 `updatedTaskData` 직접 전달로 우회한 구현은 실용적으로 잘 작동합니다.

**한계**

- **태스크 1,000개 이상**: 재귀 깊이가 깊은 선형 의존성 체인(A→B→C→...→Z 형태)에서 호출 스택이 깊어집니다. JavaScript 엔진의 기본 스택 한계는 약 10,000~15,000 프레임이므로 실제 PM 데이터 규모에서는 문제가 되지 않으나, 각 단계에서 `handleTaskUpdate` (DB 저장 포함) await가 발생하므로 100개 체인이면 100회 순차 DB 쓰기가 발생합니다.
- **전체 CPM 재계산 부재**: 현재 구현은 변경된 태스크의 후행만 재계산하는 **증분 전파(incremental propagation)** 방식입니다. Forward/Backward Pass를 통한 전체 CPM(임계 경로, 여유 시간 계산)은 구현 계획상 완료(`[x]`)로 표시되어 있으나 실제 코드에서는 `early_start`, `late_finish`, `total_float`, `is_critical` 컬럼 업데이트 로직이 없습니다.

**개선 권고**

스케줄링 cascade의 DB 쓰기를 배치 처리로 전환하는 것이 급선무입니다.

```typescript
// 현재: 각 후행마다 updateTask() 개별 호출
// 개선: 전체 cascade 완료 후 한 번에 bulk update
await supabase.from('tasks').upsert(updatedSuccessors, { onConflict: 'id' });
```

### 2.2 Web Worker 도입 계획의 현실성

구현 계획에는 완료(`[x]`)로 표시되어 있으나, 실제 코드 어디에도 Web Worker 파일이 없습니다(`find . -name '*.worker.*'` 결과 없음). 현재 스케줄링은 메인 스레드에서 동기적으로 실행됩니다. 태스크 수백 개 수준에서는 체감되지 않겠지만, 전체 CPM 재계산(Forward/Backward Pass + Float 계산) 구현 시 반드시 Worker로 이전해야 합니다. Next.js에서 Web Worker는 `new Worker(new URL('./scheduler.worker.ts', import.meta.url))`로 간단히 도입 가능합니다.

---

### 2.3 SVAR Gantt 가상화의 성능 한계

SVAR React Gantt의 행 가상화는 수천 개 태스크에서 60FPS를 보장하지만, **현재 구현에서 병목은 다른 곳**에 있습니다.

- `processedTasks`는 `useMemo([tasks])`로 감싸여 있어 임의의 task 하나 변경 시 전체 tasks 배열을 순회하여 Date 객체를 재생성합니다. 1,000개 태스크에서 매 드래그 이벤트마다 전체 배열 재처리가 발생합니다.
- SVAR Gantt에 `tasks`를 props로 넘기는 구조인데, CLAUDE.md에 명시된 대로 SVAR는 마운트 후 props 변경을 무시합니다. 그러나 `processedTasks`가 변경되면 SVAR 컴포넌트가 리렌더링되는 오버헤드는 여전히 발생합니다.

---

### 2.4 Supabase Free Tier 제약의 실질적 영향

| 제약 | 한계값 | 실질적 영향 |
|------|--------|------------|
| 동시 연결 | 200개 | 프로젝트당 1 채널이므로 사용자 200명 동시 접속 시 신규 연결 거부 |
| Realtime 메시지 | 월 200만 건 | task 드래그 1회 = UPDATE 이벤트 1건. 사용자 10명이 하루 100회씩 = 월 약 30만 건 (여유 있음) |
| DB 크기 | 500MB | CPM 필드 + 버전 컬럼으로 task당 약 2KB. 25만 task까지 허용 |

현 단계에서 Free Tier 제약은 실질적 장애물이 아닙니다. **단, `useRealtimeSync`에서 Supabase 채널 구독이 `projectId`가 바뀔 때마다 새 채널을 생성하는 구조로, 재연결 로직에 `channel.subscribe()` 상태 확인이 없어 네트워크 재접속 시 채널이 누적될 가능성**이 있습니다.

## 3. 운영 가능성(Operability)

### 3.1 에러 트래킹 및 모니터링

**현황**: 에러 처리가 `console.error`에 국한되어 있습니다. 프로덕션에서 `DBObserver`의 Supabase 쓰기 실패, 스케줄링 cascade 중 예외 등이 사용자에게 통보되지 않고 조용히 무시됩니다.

**개선 권고 (High)**

- Sentry 도입: `npm install @sentry/nextjs`. `taskObservers.ts`의 catch 블록에 `Sentry.captureException(error)` 추가.
- 낙관적 업데이트 롤백 패턴 구현: DB 저장 실패 시 UIObserver가 이전 상태로 되돌리는 로직 필요.

---

### 3.2 성능 모니터링

Next.js App Router는 `next/headers`와 `useReportWebVitals`를 통해 Core Web Vitals를 수집할 수 있습니다. 현재 이 설정이 없습니다. 간트 차트처럼 JavaScript 연산이 많은 앱에서 `INP(Interaction to Next Paint)`가 주요 지표가 됩니다.

---

### 3.3 배포 파이프라인 및 CI/CD

`.github/workflows/` 디렉토리가 없어 CI/CD 파이프라인이 구성되어 있지 않습니다. 현재는 Vercel의 자동 배포(git push → 배포)만 사용 중인 것으로 보입니다. `npm test` + `npm run lint`를 PR 필수 통과 조건으로 설정하는 GitHub Actions 워크플로우가 필요합니다.

---

### 3.4 롤백 전략

Supabase 마이그레이션은 타임스탬프 기반(`20260610000000_...`) 순방향 전용입니다. 롤백 SQL이 없는 구조로, 잘못된 마이그레이션 배포 시 수동으로 되돌려야 합니다. 각 마이그레이션에 대응하는 `rollback_*.sql` 파일을 `supabase/rollbacks/`에 관리하는 컨벤션을 권고합니다.

## 4. 의존성 리스크

### 4.1 SVAR React Gantt (GPLv3) 리스크

**현황**: SVAR React Gantt v2.3.4가 GPLv3이고, 프로젝트 본체는 MIT입니다. SaaS로 ZeroPM을 운영하는 경우 GPLv3의 "소스 공개" 의무가 SaaS에 적용되는지에 대해 법적 해석이 엇갈립니다(이른바 "SaaS 루프홀"). 그러나 라이브러리 공급사(SVAR)가 상용 라이선스를 요구할 경우 협상력이 없습니다.

**추가 리스크**: 현재 `ganttApiRef.current.exec('update-task', ...)` 형태로 SVAR 내부 API를 직접 호출하는 코드가 `useGanttObservers.js`, `useRealtimeSync.ts`, 향후 기능 곳곳에 분산되어 있습니다. SVAR 버전 업그레이드 시 내부 API 변경에 취약합니다.

**개선 권고 (Medium)**: SVAR API 호출을 `ganttBridge.js` 단일 파일로 추상화하여 어댑터 레이어를 구성합니다. 라이브러리 교체나 버전 업그레이드 시 수정 범위를 이 파일로 한정합니다.

---

### 4.2 Yjs + y-websocket 운영 부담

`y-websocket` 서버(`NEXT_PUBLIC_YJS_WS_URL`, 기본 `ws://localhost:1234`)를 별도로 운영해야 합니다. Vercel Hobby Plan에서는 WebSocket 서버를 직접 호스팅할 수 없으므로, 별도 서버(Railway, Fly.io 등) 비용이 발생합니다. 현재 Yjs는 `useYjs.ts`에서 사용되지만, 실제 동기화는 `useRealtimeSync.ts`의 `postgres_changes`로 처리됩니다. 두 동기화 메커니즘이 병행 운영되어 중복입니다. Yjs의 실질적 역할(CRDT 문서 동기화 vs 프레즌스 vs 커서 공유)을 명확히 정의하고 하나로 통합하는 것을 권고합니다.

---

### 4.3 Next.js App Router와 SVAR Gantt 충돌

`GanttView.jsx`의 `'use client'` 선언으로 충돌은 잘 관리되고 있습니다. 다만 `processedTasks` 계산, 옵저버 초기화, 실시간 동기화가 모두 클라이언트 번들에 포함됩니다. 초기 데이터 로딩(`initialTasks`, `initialLinks`)은 Server Component에서 Supabase SSR 클라이언트로 수행하는 구조가 올바릅니다. 이 경계를 유지하는 것이 중요합니다.

## 5. 테스트 가능성(Testability)

### 5.1 현황 평가

`scheduling.test.ts`(85줄), `scheduling_scenario.test.ts`, `ganttScheduler.test.js`(126줄)가 순수 함수와 스케줄링 로직을 잘 커버합니다. 이는 View/Brain 분리의 직접적인 이점입니다.

**문제점**

- `useGanttObservers.js`의 테스트가 없습니다. 옵저버가 올바르게 등록/해제되는지, `skipRecalculation=true` 시 SVAR API가 호출되는지 검증하는 테스트가 필요합니다.
- `useRealtimeSync.ts`의 테스트가 없습니다. Supabase Realtime 콜백의 dedup 로직이 핵심 비즈니스 로직임에도 테스트되지 않습니다.
- `GanttView.test.jsx`는 SVAR Gantt를 mock으로 대체하여 존재하지만, 실제 이벤트 흐름(드래그 → emit → DB 저장 → UI 갱신)의 통합 테스트가 없습니다.

### 5.3 E2E 테스트 전략

`@playwright/test`가 devDependency에 있으나 테스트 파일이 없습니다. Gantt 차트 드래그 동작은 `page.dragAndDrop()`으로 테스트하기 어렵습니다. 우선순위가 높은 E2E 시나리오:

1. 로그인 → 프로젝트 생성 → 태스크 생성 (critical path, 회귀 방지)
2. 의존성 링크 생성 후 선행 태스크 이동 시 후행 자동 재계산 확인
3. 두 탭에서 같은 프로젝트 접속 후 한쪽 변경이 다른 탭에 반영되는지 확인 (Realtime sync)

## 6. API 계층 설계

### 6.1 Next.js Route Handler vs Server Action

현재 Supabase 브라우저 클라이언트를 직접 사용하여 클라이언트에서 DB에 접근합니다. `src/app/auth/callback/route.ts`만 Route Handler로 구현되어 있고, 나머지 데이터 접근은 전부 클라이언트 직접 접근입니다. 이는 초기 개발 속도 측면에서 합리적이지만 다음 한계가 있습니다.

- RLS 정책이 유일한 보안 계층이 됩니다. 클라이언트가 `supabase_anon_key`를 직접 사용하므로 RLS 설정 오류 시 전체 데이터가 노출됩니다.
- Vercel Serverless Function에서 복잡한 트랜잭션(Baseline 생성, CPM 결과 일괄 저장)을 처리해야 할 때 Server Action 또는 Route Handler가 필요합니다.

**개선 권고 (Medium)**: 현 구조를 유지하되, Baseline 생성과 같은 복잡한 원자적 쓰기는 Supabase RPC(PostgreSQL 함수)로 캡슐화합니다. 클라이언트에서 `supabase.rpc('create_baseline', {...})`를 호출하는 방식이 Serverless Function 없이 원자성을 보장하는 가장 현실적인 접근입니다.

---

### 6.2 API 버전 관리

현재 API 버전 관리 전략이 없습니다. Supabase REST API는 Supabase 버전에 종속되므로, 클라이언트-서버 계약 변경 시 직접 영향을 받습니다. `src/types/database.ts`의 타입 정의가 버전 관리의 역할을 대신하고 있습니다. 현 단계에서는 이것으로 충분하며, 외부 API를 공개하는 시점에 `/api/v1/` 접두사 전략을 도입하면 됩니다.

## 결론

ZeroPM의 아키텍처는 **View/Brain 분리와 이벤트 기반 설계라는 방향성이 올바릅니다**. 특히 `scheduling.ts`의 순수 함수 설계, `visited Set` 기반 사이클 방지, 실시간 동기화의 echo 필터링은 실용적인 문제 해결이 돋보입니다.

가장 시급한 과제는 두 가지입니다. 첫째, DB 쓰기 실패가 UI를 멈추지 않도록 에러 경계를 강화하는 것. 둘째, 스케줄링 cascade의 N회 DB 왕복을 배치로 줄이는 것입니다. 이 두 가지가 해결되면 현재 아키텍처는 수백 태스크 규모의 실사용 환경에서 충분한 견고성을 발휘할 것으로 판단됩니다.
