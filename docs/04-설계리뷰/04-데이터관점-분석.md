## 1. 스키마 설계 평가

### 현황 평가

전반적으로 3NF를 준수하는 관계형 설계를 갖추고 있다. `tasks`, `links`, `resources`, `assignments`, `costs` 테이블은 각각의 엔티티를 명확히 분리하고, 반복 그룹이나 부분 함수적 종속 없이 설계되어 있다. FK 참조와 ON DELETE CASCADE 전략도 일관성 있게 적용되어 있다.

### 문제점

**인덱스 누락**: 실제 마이그레이션 파일 전체를 검토한 결과, `20251205000000_add_task_constraints.sql`에 `idx_tasks_constraint` 하나만 존재하고, 가장 빈번하게 실행되는 쿼리 경로인 `tasks.project_id`, `links.project_id`, `links.source`, `links.target`에 인덱스가 없다. `tasks.ts`의 `getTasks()`와 `getLinks()` 함수가 매번 시퀀셜 스캔을 할 가능성이 높다.

```sql
-- 현재 없는 필수 인덱스들
-- tasks(project_id) → 프로젝트 로드 시 전체 태스크 조회
-- tasks(parent_id)  → 계층 탐색
-- links(project_id) → 프로젝트 링크 전체 로드
-- links(source)     → 후행 태스크 조회 (스케줄링 전파 핵심)
-- links(target)     → 선행 태스크 조회
```

**JSONB 설계의 이중성 문제**: `baselines` 테이블이 초기 스키마(`20251203070000`)에서는 `baseline_tasks` 별도 테이블로 정규화되어 설계되었다가, `20251204000000_add_baselines.sql`에서 `data JSONB NOT NULL` 단일 컬럼으로 완전히 교체되었다. 기준선 전체 태스크 스냅샷이 JSONB 하나에 들어가는 구조로, 이는 크기 제한(PostgreSQL의 행 크기 제한 8KB, TOAST 저장의 성능 불이익)과 인덱스 불가 문제를 동시에 안고 있다. 반면 `calendar_settings`의 JSONB 사용(`{"weekends": [0,6], "holidays": []}`)은 변경 주기가 낮고 JSON 스키마가 안정적이므로 적절한 선택이다.

**CPM 필드 설계의 불일치**: `src/types/database.ts`의 `Task` 타입에는 `early_start`, `early_finish`, `late_start`, `late_finish`가 모두 선언되어 있다. 그러나 실제 마이그레이션의 `tasks` 테이블 DDL(20251203070000)에는 이 네 필드가 정의되어 있지만 `update_task_with_version` RPC 함수는 CPM 필드를 업데이트하지 않는다. `save_cpm_results` RPC는 CPM 전용이지만 `version` 컬럼을 증가시키지 않아, CPM 재계산 직후 낙관적 잠금 버전이 클라이언트의 캐시와 불일치하는 잠재적 버그가 존재한다.

### 개선 권고

```sql
-- 즉시 적용 필요한 인덱스
CREATE INDEX CONCURRENTLY idx_tasks_project_id ON tasks(project_id);
CREATE INDEX CONCURRENTLY idx_tasks_parent_id ON tasks(parent_id) WHERE parent_id IS NOT NULL;
CREATE INDEX CONCURRENTLY idx_tasks_project_sort ON tasks(project_id, sort_order NULLS LAST);
CREATE INDEX CONCURRENTLY idx_links_project_id ON links(project_id);
CREATE INDEX CONCURRENTLY idx_links_source ON links(source);
CREATE INDEX CONCURRENTLY idx_links_target ON links(target);
-- 후행 탐색용 복합 인덱스 (스케줄링 엔진 핵심 경로)
CREATE INDEX CONCURRENTLY idx_links_source_project ON links(source, project_id);
```

`baselines.data` JSONB는 중기적으로 `baseline_tasks` 테이블로 재정규화하거나, 최소한 JSONB 내 핵심 필드에 GIN 인덱스를 추가해야 한다. `save_cpm_results` 함수는 `version = version + 1`을 포함하도록 수정이 필요하다.

## 2. 계층 구조 설계 평가

### 현황 평가

`parent_id`를 이용한 인접 리스트(Adjacency List) 모델을 채택하고 있다. 이는 삽입과 이동이 빠른 반면, 다단계 계층 조회에 재귀 CTE(`WITH RECURSIVE`)가 필요하다는 트레이드오프를 갖는다.

### 문제점

**재귀 CTE의 성능 스케일 문제**: WBS 5~6단계(예: 1.1.1.1.1)의 구조에서 Summary Task의 진행률 롤업(`progress`)을 계산하려면 전체 하위 트리를 순회해야 한다. 현재 `getTasks()`는 프로젝트 전체 태스크를 플랫 배열로 클라이언트에 로드하고, 트리 구성은 클라이언트 측에서 처리한다. 태스크가 수백 건 이상이면 초기 로드 페이로드가 급격히 증가한다.

**서버 측 롤업 연산 부재**: 현재 Summary Task의 `progress`와 `duration`은 클라이언트 측 스케줄링 엔진에서 계산되어 다시 DB에 저장되는 방식으로 동작하는 것으로 보이나, 실제 DB 함수나 트리거로 롤업을 강제하는 로직이 없다. 이는 다중 사용자 편집 시 롤업 값의 일관성이 클라이언트의 정상 동작에 전적으로 의존하게 되는 취약점이다.

**대안 비교**:

| 모델 | 삽입/이동 | 계층 조회 | 롤업 계산 | 적합성 |
|---|---|---|---|---|
| **인접 리스트 (현재)** | O(1) | 재귀 CTE 필요 | 클라이언트 처리 | 중소 규모 프로젝트 적합 |
| **Closure Table** | O(depth) | O(1) JOIN | 서버에서 집계 가능 | 읽기 집약적 환경 |
| **Nested Set** | O(n) - 전체 갱신 | O(1) 범위 쿼리 | 집계 용이 | 정적 계층에만 적합 |
| **ltree (PostgreSQL 전용)** | O(1) | 경로 연산자 사용 | 집계 용이 | PostgreSQL 환경 최적 |

### 개선 권고

단기적으로는 인접 리스트를 유지하되, PostgreSQL의 `ltree` 확장을 `path` 컬럼으로 병행 관리하는 **하이브리드 전략**이 현실적이다. `ltree`는 `parent_id` 변경 시 트리거로 자동 갱신할 수 있어 WBS 경로 표시(`1.2.3`)와 하위 트리 조회를 O(1)로 처리할 수 있다.

```sql
ALTER TABLE tasks ADD COLUMN wbs_path ltree;
CREATE INDEX idx_tasks_wbs_path ON tasks USING GIST(wbs_path);
```

중장기적으로, Summary Task 롤업은 PostgreSQL 트리거로 서버 측에서 강제하여 다중 사용자 환경에서의 일관성을 보장해야 한다.

## 3. 동시성 제어 설계 평가

### 현황 평가

`version INTEGER DEFAULT 1` 컬럼을 통한 낙관적 잠금과 `update_task_with_version` RPC 함수가 구현되어 있다. 그러나 현재 `src/lib/tasks.ts`의 `updateTask()` 함수는 이 RPC를 전혀 사용하지 않고 직접 `.update()` Supabase 클라이언트 호출을 한다. 즉, **낙관적 잠금 RPC가 있지만 실제로는 우회되고 있다**.

### 문제점

**낙관적 잠금의 비일관적 적용**: `src/lib/optimisticLocking.ts`가 별도로 존재하지만, `tasks.ts`의 `updateTask()` 함수는 버전 확인 없이 무조건 업데이트한다. 이는 두 사용자가 동시에 같은 태스크를 편집할 때 마지막 쓰기가 이전 것을 덮어쓰는 **Lost Update 문제**를 방지하지 못한다.

**CRDT와 낙관적 잠금의 역할 경계 모호성**: 설계 문서는 "CRDT가 담당하지 않는 구조적 변경 시 낙관적 잠금 사용"이라고 명시하지만, CLAUDE.md의 아키텍처 설명을 보면 실제 Yjs는 현재 크로스탭 동기화에 활용되지 않고 있으며(`useRealtimeSync.ts`가 `postgres_changes`로 대체), Yjs는 y-websocket 서버가 필요한 선택적 기능으로 분리되어 있다. 현재 상태에서 동시 편집 충돌 방지의 실질적 책임은 낙관적 잠금에 있으나, 이것이 우회되고 있는 상황이다.

**트랜잭션 경계 불완전성**: `save_cpm_results` RPC가 루프로 개별 UPDATE를 수행한다. 이 함수는 `EXCEPTION` 블록으로 오류를 잡지만, PostgreSQL의 PL/pgSQL에서 `EXCEPTION` 블록이 있어도 그 전에 이미 커밋된 개별 UPDATE는 롤백되지 않는다. 따라서 CPM 결과를 50개 태스크 중 30개만 저장하다 실패하면 불완전한 상태가 DB에 남는다. 이를 해결하려면 `SAVEPOINT`나 단일 배치 UPDATE를 사용해야 한다.

### 개선 권고

즉시 조치로 `updateTask()` 함수를 `update_task_with_version` RPC를 호출하도록 교체하고, 버전 충돌 시 클라이언트에 재시도 로직을 추가해야 한다. `save_cpm_results`는 루프 대신 `unnest()`를 이용한 단일 배치 UPDATE로 교체해야 원자성이 보장된다.

```sql
-- save_cpm_results 개선안 (배치 처리)
UPDATE tasks AS t
SET
  early_start = v.early_start::TIMESTAMPTZ,
  late_finish = v.late_finish::TIMESTAMPTZ,
  total_float = v.total_float::INTEGER,
  is_critical  = v.is_critical::BOOLEAN,
  version      = t.version + 1,
  updated_at   = NOW()
FROM jsonb_to_recordset(p_results) AS v(id UUID, early_start TEXT, late_finish TEXT, total_float TEXT, is_critical TEXT)
WHERE t.id = v.id;
```

## 4. 마이그레이션 전략 평가

### 현황 평가

Supabase 표준 타임스탬프 마이그레이션 파일 방식을 사용하고 있다. 파일 수는 8개로 관리가 가능한 수준이다. `IF NOT EXISTS`, `IF EXISTS` 구문의 사용으로 멱등성을 부분적으로 확보하고 있다.

### 문제점

**파괴적 마이그레이션 존재**: `20251204000000_add_baselines.sql`은 `DROP TABLE IF EXISTS baseline_tasks; DROP TABLE IF EXISTS baselines;` 후 재생성하는 방식이다. 이는 기존 데이터를 가진 프로덕션 환경에서 실행 시 **데이터 손실**을 초래한다. 설계 변경(JSONB 단일 컬럼으로 재설계)이 있었더라도 `ALTER TABLE` + `ADD COLUMN` + 데이터 마이그레이션 방식으로 처리했어야 한다.

**하위 호환성 전략 부재**: 컬럼 이름 변경, 타입 변경, NOT NULL 제약 추가 등 파괴적 변경에 대한 명시적 정책이 없다. 예를 들어 `constraint_date`의 타입이 `20251205000000` 마이그레이션에서 `TIMESTAMP`(타임존 없음)로 추가되었는데, 기존 `tasks` 테이블의 다른 날짜 필드는 모두 `TIMESTAMPTZ`이다. 이런 불일치가 마이그레이션 단계에서 검증되지 않았다.

**롤백 절차 부재**: 각 마이그레이션 파일에 대응하는 롤백 스크립트(`down` 마이그레이션)가 없다. Supabase CLI는 롤백을 직접 지원하지 않으므로, 별도의 `rollback/` 디렉토리에 역방향 SQL을 관리하는 관행이 필요하다.

### 개선 권고

- **즉시**: `constraint_date` 컬럼을 `TIMESTAMPTZ`로 통일하는 수정 마이그레이션 추가
- **정책 수립**: 컬럼/타입 변경은 반드시 3단계(신규 컬럼 추가 → 데이터 마이그레이션 → 구 컬럼 제거)로 수행
- **롤백 스크립트**: 각 마이그레이션 파일에 대응하는 `supabase/rollbacks/YYYYMMDD_description.sql` 파일을 병행 관리
- **대규모 ALTER**: 프로덕션에서 `ALTER TABLE tasks ADD COLUMN` 실행 시 `CONCURRENTLY`가 지원되지 않는 작업은 반드시 유지 보수 윈도우를 계획

## 5. 데이터 보존 및 감사(Audit) 평가

### 현황 평가

기준선(`baselines`) 테이블이 특정 시점의 스냅샷을 저장하는 유일한 보존 메커니즘이다. 삭제 전략은 FK + ON DELETE CASCADE로, 프로젝트/태스크 삭제 시 모든 하위 데이터가 즉시 물리적으로 삭제된다.

### 문제점

**변경 이력 추적 부재**: 태스크의 이름 변경, 기간 변경, 담당자 변경 등 모든 편집 이력이 DB에 기록되지 않는다. `updated_at`과 `version` 컬럼은 현재 상태만 반영할 뿐, 이전 값이 무엇이었는지 알 수 없다. 프로젝트 감사나 분쟁 상황에서 변경 추적이 불가능하다.

**소프트 삭제 미구현**: `tasks` 및 `links` 테이블에 `deleted_at` 컬럼이 없어 삭제된 태스크를 복구할 수 없다. CASCADE 삭제로 인해 링크, 할당, 비용 정보도 함께 영구 삭제된다.

**Yjs 영속성 미구현**: 설계 문서에 `Yjs Persistence Table` (선택 사항)으로 언급되어 있으나 실제 마이그레이션에는 존재하지 않는다. 브라우저 새로고침 후 Yjs 문서 상태 복구가 불완전할 수 있다.

### 개선 권고

**단기**: 소프트 삭제 구현

```sql
ALTER TABLE tasks ADD COLUMN deleted_at TIMESTAMPTZ DEFAULT NULL;
ALTER TABLE links ADD COLUMN deleted_at TIMESTAMPTZ DEFAULT NULL;
-- RLS에 deleted_at IS NULL 조건 추가
-- 복구 RPC 함수 추가
```

**중기**: PostgreSQL `audit` 확장 또는 트리거 기반 감사 로그 테이블 도입

```sql
CREATE TABLE task_audit_log (
  id          BIGSERIAL PRIMARY KEY,
  task_id     UUID NOT NULL,
  operation   TEXT NOT NULL CHECK (operation IN ('INSERT','UPDATE','DELETE')),
  changed_by  UUID REFERENCES auth.users(id),
  old_data    JSONB,
  new_data    JSONB,
  changed_at  TIMESTAMPTZ DEFAULT NOW()
);
```

이 테이블은 RLS가 아닌 `SECURITY DEFINER` 트리거로만 쓰기를 허용하고, 읽기는 프로젝트 관리자(admin)에게만 허용하는 정책이 적합하다.

## 6. 성능 최적화 평가

### 현황 평가

`getTasks()`와 `getLinks()`는 각각 별도의 순차 쿼리로 실행된다(`tasks.ts` 참조). 프로젝트 로드 시 두 번의 네트워크 왕복이 발생한다. 현재 규모에서는 문제없지만, 구조적 개선 여지가 있다.

### 문제점

**N+1 리스크**: 현재 `getTasks()`는 단일 쿼리이므로 직접적인 N+1 문제는 없다. 그러나 Kanban View, Calendar View 등 보조 뷰 구현 시 태스크별 `assignments`, `resources`, `costs`를 별도 조회한다면 N+1이 발생할 수 있다. 현재는 관련 뷰가 미구현이라 잠재적 위험이다.

**페이지네이션 부재**: `getTasks()`에 페이지네이션 로직이 없다. 1,000건 이상의 태스크를 가진 대형 프로젝트에서는 초기 로드 시간과 클라이언트 메모리 사용량이 문제가 된다. 간트 차트는 가상화로 렌더링을 최적화하지만, 데이터 자체는 전부 메모리에 올라와 있다.

**RLS 정책의 쿼리 비용**: tasks/links RLS 정책이 `EXISTS (SELECT 1 FROM projects WHERE ... OR EXISTS (SELECT 1 FROM project_members ...))`의 이중 서브쿼리 구조다. 태스크 행마다 이 서브쿼리가 실행된다. `projects(id, owner_id)`와 `project_members(project_id, user_id)`에 적절한 인덱스(1번 섹션 권고사항)가 없다면 성능 저하가 심각하다.

**Materialized View 미활용**: CPM 필드(`early_start`, `total_float`, `is_critical`)와 롤업 데이터를 집계하는 분석용 뷰가 없다. 보고서/대시보드(Phase 6) 구현 시 실시간 집계는 비용이 높다.

### 개선 권고

**즉시**: 위 1번 섹션의 인덱스 추가 (RLS 성능 개선에도 직결됨)

**단기**: `getTasks`와 `getLinks`를 단일 RPC로 병합하여 왕복 횟수 절감

```sql
CREATE OR REPLACE FUNCTION get_project_data(p_project_id UUID)
RETURNS JSONB AS $$
  SELECT jsonb_build_object(
    'tasks', (SELECT jsonb_agg(t) FROM tasks t WHERE t.project_id = p_project_id AND t.deleted_at IS NULL ORDER BY t.sort_order),
    'links', (SELECT jsonb_agg(l) FROM links l WHERE l.project_id = p_project_id AND l.deleted_at IS NULL)
  );
$$ LANGUAGE SQL SECURITY DEFINER STABLE;
```

**중기**: 커서 기반 페이지네이션 구현. 초기 로드는 뷰포트 기준 상위 200건만, 스크롤 시 추가 로드하는 방식으로 대형 프로젝트 지원

**장기**: 보고서 전용 Materialized View 구성 (Phase 6 시작 전 완료 필요)

```sql
CREATE MATERIALIZED VIEW project_summary AS
SELECT
  p.id AS project_id,
  COUNT(t.id) FILTER (WHERE NOT t.is_critical) AS normal_tasks,
  COUNT(t.id) FILTER (WHERE t.is_critical) AS critical_tasks,
  AVG(t.progress) AS avg_progress,
  MIN(t.start_date) AS project_start,
  MAX(t.start_date + (t.duration || ' minutes')::INTERVAL) AS project_end
FROM projects p
LEFT JOIN tasks t ON t.project_id = p.id AND t.deleted_at IS NULL
GROUP BY p.id;

CREATE UNIQUE INDEX ON project_summary(project_id);
-- REFRESH MATERIALIZED VIEW CONCURRENTLY project_summary; -- 주기적 실행
```

## 총평

ZeroPM의 데이터베이스 설계는 PM 도구에 필요한 핵심 엔티티(태스크, 링크, 자원, 비용, 기준선)를 잘 식별하고 관계형 구조로 설계했다는 점에서 출발점이 견실하다. Supabase Auth 연동, RLS 기반 RBAC, 낙관적 잠금 RPC 등 보안과 동시성의 골격도 갖추고 있다.

그러나 **현재 가장 심각한 리스크는 두 가지**다. 첫째, 인덱스 부재로 인해 태스크 수가 증가할수록 핵심 쿼리 성능이 급격히 저하될 것이다. 둘째, 낙관적 잠금 RPC가 실제 데이터 접근 레이어(`tasks.ts`)에서 우회되고 있어 다중 사용자 환경의 데이터 무결성이 사실상 보장되지 않고 있다. 이 두 가지(H-1, H-2)는 다른 어떤 기능 개발보다 우선적으로 처리되어야 한다.
