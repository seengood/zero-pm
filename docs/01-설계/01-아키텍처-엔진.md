# ZeroPM 아키텍처 & 엔진

> 문서 버전: v4.0 (2026-06-12 단순화 통합)

이 문서는 ZeroPM의 시스템 아키텍처, 스케줄링 엔진, 실시간 협업을 하나로 통합한다. 기존 `03-시스템-아키텍처.md`, `04-스케줄링-엔진.md`, `07-실시간-협업.md`를 대체한다. 개선·미구현 과제의 일정과 우선순위는 산문으로만 기술하며, 구체적 계획은 `03-로드맵`을 참조한다.

---

## 1. 아키텍처 원칙

ZeroPM의 핵심 설계는 **렌더링과 스케줄링 로직의 분리**이며, 두 영역은 직접 함수 호출이 아니라 싱글턴 이벤트 이미터를 통해 통신한다.

- **View / Brain 분리** — 렌더링(View)은 SVAR React Gantt가, 비즈니스 로직(Brain)은 자체 스케줄링 엔진(`ganttScheduler.js`, `scheduling.ts`)이 담당한다. `GanttView.jsx`가 둘을 연결하는 오케스트레이터다.
- **이벤트 기반 Observer** — 컴포넌트 간 통신은 `taskEventEmitter` 싱글턴을 통한다. `DBObserver`/`UIObserver`/`ScheduleObserver`는 각각 단일 관심사만 담당하며 독립적으로 실패할 수 있다.
- **낙관적 UI** — DB 저장 성공 여부와 무관하게 UI를 즉시 갱신한다. DB 실패 시 에러 토스트를 표시하고 필요 시 롤백한다.
- **안정적 Ref 패턴** — `recalculateAffectedTasks`는 `useMemo([])`로 감싸 재초기화를 방지한다. 최신 task/link 데이터는 클로저에 캡처된 상태가 아니라 항상 `tasksRef.current` / `linksRef.current`로 읽는다. 이 빈 의존성 배열과 ref 참조 패턴이 깨지면 옵저버 재등록·중복 실행 버그가 발생하므로 절대 변경하지 않는다.
- **Observer 병합 업데이트** — `DBObserver`/`UIObserver`는 `{ ...currentTask, ...task }`로 병합해 부분(partial) 이벤트가 다른 필드를 지우지 않게 한다.

### 기술 스택

| 레이어 | 기술 | 비고 |
|--------|------|------|
| 프레임워크 | Next.js 16 (App Router) | 포트 3005 |
| UI | React 19.2 / TypeScript 5 | |

## 2. 이벤트 아키텍처

### 2.1 전체 데이터 흐름

```
사용자 액션 (Gantt에서 드래그/리사이즈/링크 생성)
  → ganttIntercepts.js  (api.intercept: resize-task/update-task/add-link …)
       유효성 검사, 날짜 정규화 (start 00:00:00, 표시 end 23:59:59)
  → GanttView.jsx — handleTaskUpdate / handleLinkCreate …
       │
       ├─ taskEventEmitter.emit('task:updated' | 'link:created' | …)
       │     ├─▶ DBObserver   → Supabase tasks/links 영속화 (병합)
       │     ├─▶ UIObserver   → setTasks/setLinks React 상태 갱신 (병합)
       │     └─▶ ScheduleObserver (link 이벤트 시) → recalculateAffectedTasks(link.source)
       │
       └─ handleTaskUpdate가 직접 recalculateAffectedTasks(task.id, task) 호출
            → 스케줄링 전파 (§3)
```

`emit()`은 등록된 리스너를 await한다. 리스너는 `{ type, payload, timestamp }` 형태의 이벤트 객체를 받는다. 현재는 DBObserver 실패가 다른 옵저버를 블로킹할 수 있는 구조이며, DB 실패를 낙관적 UI와 분리(에러 토스트만 표시)하는 개선이 예정되어 있다 (상세는 03-로드맵 참조).

### 2.2 이벤트 정의

| 이벤트 | 발생 시점 | Payload | Observer 처리 |
|--------|-----------|---------|---------------|
| `task:created` | 새 task 생성 | `task` (전체) | DB Insert, UI 추가 |
| `task:updated` | task 속성 변경 (드래그 포함) | `task` (병합), `skipRecalculation` 플래그 | DB Update, UI 갱신, (필요시) 재계산 |
| `task:deleted` | task 삭제 | `taskId` | DB Delete, UI 제거 |
| `link:created` | 의존성 연결 | `link` | DB Insert, UI 추가, 재계산 |
| `link:updated` | 링크 속성 변경 | `link` | DB Update, UI 갱신, 재계산 |
| `link:deleted` | 링크 삭제 | `linkId` | DB Delete, UI 제거, 재계산 |

### 2.3 옵저버

- **DBObserver** — Supabase에 영속화한다(`src/lib/tasks.ts`). 업데이트는 `{ ...current, ...task }` 병합.
- **UIObserver** — `setTasks`/`setLinks`로 React 상태를 병합 갱신한다.
- **ScheduleObserver** — `link:created|updated|deleted` 수신 시 `recalculateAffectedTasks(link.source, sourceTask)`를 호출해 스케줄링 전파를 트리거한다.

핵심 파일: `taskEventEmitter.ts`(싱글턴 pub/sub), `taskObservers.ts`(세 옵저버), `useGanttObservers.js`(옵저버 인스턴스화·구독·정리).

### 2.4 SVAR 직접 업데이트 패턴

**SVAR Gantt는 마운트 후 `tasks`/`links` React props 변경을 무시한다.** 초기 마운트 시 props로 내부 스토어를 초기화한 뒤로는 React props를 반영하지 않는다. 따라서 모든 프로그래매틱 업데이트(스케줄링 전파, 크로스탭 동기화)는 반드시 `ganttApiRef.current.exec(...)`로 SVAR 내부 스토어에 직접 적용해야 한다.

```javascript
ganttApiRef.current.exec('update-task', { id, start, end, duration }); // start/end는 Date
ganttApiRef.current.exec('add-task', task);
ganttApiRef.current.exec('delete-task', { id });
ganttApiRef.current.exec('add-link', link);
ganttApiRef.current.exec('delete-link', { id });
```

프로그래매틱 `exec()`는 SVAR 인터셉트를 재트리거할 수 있으므로, `isSchedulerUpdateRef.current = true`로 토글해 인터셉트 재진입을 방지한다.

```javascript
// ganttIntercepts.js
api.intercept('update-task', (event) => {
  if (isSchedulerUpdateRef.current) return; // 스케줄러 업데이트면 인터셉트 건너뜀
  // … 사용자 액션 처리
});
```

**CSS 셋업 주의:** Turbopack이 `node_modules`의 압축 CSS를 파싱하지 못하므로 `src/styles/gantt-svar.css`로 로컬 복사해야 한다.

```bash
cp node_modules/@svar-ui/react-gantt/dist/index.css src/styles/gantt-svar.css
```

빌드 중 `Parsing CSS source code failed - Invalid empty selector` 오류 시 이 명령을 다시 실행한다. SVAR React Gantt는 GPLv3이므로 재배포 시 라이선스에 유의한다.

## 3. 스케줄링 엔진

스케줄링 엔진은 MS Project 수준의 일정 계산을 클라이언트 측에서 자체 알고리즘으로 처리한다. 의존성 전파·순수 계산은 구현 완료, CPM 전진/후진 계산은 미구현(DB 컬럼·타입만 정의)이다.

| 책임 | 파일 | 구현 상태 |
|------|------|----------|
| 의존성 전파 (Cascade) | `src/lib/ganttScheduler.js` | 구현 완료 |
| 순수 스케줄링 계산 | `src/lib/scheduling.ts` | 구현 완료 (`alap`만 미완성) |
| 임계 경로 분석 (CPM) | `src/types/database.ts` (컬럼·타입만) | 계산 로직 **미구현** |

### 3.1 의존성 전파 (재귀 DFS)

사용자 액션이 발생하면 변경 태스크에서 시작해 모든 후행(successor)의 날짜를 재귀적으로 재계산한다.

```
recalculateAffectedTasks(task.id, task)   ← ganttScheduler.js
  → 후행 링크 재귀 DFS 순회 (visited Set으로 사이클 방지)
  → 각 후행에 calculateSuccessorDate → calculateEndDate → applyConstraint 적용
  → handleTaskUpdate(updatedSuccessor, skipRecalculation=true)
       → useGanttObservers.js가 skipRecalculation=true 감지
       → ganttApiRef.current.exec('update-task', …)로 SVAR 스토어 직접 갱신
         (isSchedulerUpdateRef=true 로 인터셉트 재진입 방지)
```

**정본 시그니처:**

```javascript
createRecalculateFunction({ tasksRef, linksRef, handleTaskUpdate, toISOString })
  → recalculateAffectedTasks(changedTaskId, updatedTaskData = null, visited = new Set())
```

**stale ref 해결:** React state(`tasks`)는 비동기로 갱신되므로 재귀 시점에 `tasksRef.current`가 최신 값을 반영하지 못할 수 있다. 따라서 직전 단계에서 계산된 `updatedSuccessor`를 다음 재귀 호출의 `updatedTaskData` 인자로 직접 전달한다. `visited Set`은 전체 cascade 호출 트리에 걸쳐 공유되어 사이클·중복 방문을 막고, 전파 루프 방지를 위해 `skipRecalculation=true`로 emit한다.

현재 cascade는 후행 task마다 개별 DB 왕복이 발생한다. 변경분을 모아 1회 bulk upsert하는 최적화가 예정되어 있다 (상세는 03-로드맵 참조).

### 3.2 링크 타입 및 계산

SVAR Gantt는 MS Project 표준 라벨과 다른 코드를 쓴다(정의: `src/lib/constants.ts`). 혼동하지 않는다.

```
calculateSuccessorDate(predecessor, successor, linkType, lag = 0): Date | null
```

후행 시작일을 UTC 헬퍼(`addDaysUTC`)로 계산한다. `predEnd`는 배타적 종료일(없으면 `start + duration`). 후행 현재 시작일과 동일하면 `null`(변경 없음)을 반환한다.

| SVAR 코드 | MS Project | 후행 시작일(newStart) 계산 |
|----------|-----------|---------------------------|
| `e2s` | FS (Finish-to-Start) | `addDaysUTC(predEnd, 1 + lag)` |
| `s2s` | SS (Start-to-Start) | `addDaysUTC(predStart, lag)` |
| `e2e` | FF (Finish-to-Finish) | `addDaysUTC(addDaysUTC(predEnd, lag), -succDuration + 1)` |
| `s2e` | SF (Start-to-Finish) | `addDaysUTC(addDaysUTC(predStart, lag), -succDuration + 1)` |

`lag`: 양수=지연, 음수=Lead Time. FF/SF는 후행 종료를 기준점으로 보고 `종료 - duration + 1`로 시작일을 역산한다.

### 3.3 제약 조건 (6종)

```
applyConstraint(task, calculatedStart, calculatedEnd): { start, end }
```

링크 타입 계산 결과에 태스크의 `constraint_type`/`constraint_date`를 추가 적용해 최종 날짜를 결정한다. `constraint_date`가 없으면 계산값을 그대로 반환한다.

| 코드 | 명칭 | 동작 | 상태 |
|------|------|------|------|
| `asap` | As Soon As Possible | 계산값 그대로 (기본값) | 구현 완료 |
| `mso` | Must Start On | `start = date`, `end = +duration` | 구현 완료 |
| `mfo` | Must Finish On | `end = date`, `start = -duration` | 구현 완료 |
| `snet` | Start No Earlier Than | `start < date`일 때만 date로 밀어냄 | 구현 완료 |
| `fnlt` | Finish No Later Than | `end > date`일 때만 date로 당겨옴 | 구현 완료 |
| `alap` | As Late As Possible | 프로젝트 종료일 기준 역방향 스케줄 | **미완성** (현재 ASAP과 동일) |

`alap`은 소스에 `TODO: Implement ALAP with project end date`로 명시되어 있다.

### 3.4 임계 경로 분석 (CPM) — 미구현

CPM은 선후행 관계만으로 프로젝트 최소 기간과 일정 여유를 산출한다. **ZeroPM은 DB 컬럼(`early_start`, `late_finish`, `total_float`, `is_critical`)과 타입만 정의했고, 전진/후진 계산 로직은 작성하지 않았다.** 아래는 구현 시 목표 정의다.

- **전진 계산(Forward Pass)** — 시작 시점부터 각 태스크의 조기 시작(ES)·조기 종료(EF) 산출. 모든 선행 EF의 최댓값이 후행 ES.
- **후진 계산(Backward Pass)** — 완료 시점에서 역산해 늦은 완료(LF)·늦은 시작(LS) 산출. 모든 후행 LS의 최솟값이 선행 LF.
- **총 여유(Total Float)** — `LS - ES (= LF - EF)`. 완료일에 영향 없이 지연 가능한 최대 시간.
- **임계 경로** — `Total Float = 0`인 태스크 경로. 간트에서 강조(`is_critical`).

정확한 계산은 태스크/의존성을 DAG로 모델링하고 위상 정렬로 계산 순서를 선형화해야 한다. 현재 `ganttScheduler.js`의 `visited Set`은 cascade 중 사이클을 막을 뿐, 명시적 전체 그래프 순환 검증과 위상 정렬은 미구현이다. 결과 일괄 저장은 `save_cpm_results` RPC(배치 UPDATE + `version+1`)로 처리할 계획이다 (상세는 03-로드맵 참조).

## 4. Undo/Redo 설계 (미구현)

현재 이벤트 파이프라인은 단방향이다. 사용자 액션 하나가 cascade로 N개 태스크를 연쇄 변경하고 각각 DB에 영속화한다. Undo/Redo는 이를 깨지 않고 **별도 레이어로 추가**한다. 전체 설계는 완료되었으나 구현은 예정 상태다 (상세는 03-로드맵 참조).

**Command 패턴 (트랜잭션 경계):** 사용자의 단일 의도(드래그 1회, 리사이즈, 모달 저장, 링크 생성/삭제)를 하나의 트랜잭션으로 본다. cascade로 변경된 N개 태스크는 모두 같은 트랜잭션에 `ChangeRecord`로 누적되어 단일 Command로 묶인다(개별 cascade Undo는 무의미). cascade 중 `handleTaskUpdate(skipRecalculation=true)` 호출은 새 트랜잭션을 시작하지 않고 열린 트랜잭션에 change를 추가한다.

```
intercept 진입 → beginTransaction(label)
cascade 루프 → recordTaskChange / recordLinkCreate / recordLinkDelete
완료 → commitTransaction()   (DB 저장 실패 시 abortTransaction)
```

**executeUndo 3단계 복원:** Undo는 이벤트 파이프라인을 **우회**한다(`emit()`을 쓰면 옵저버가 재실행되고 cascade가 재트리거되어 무한루프). 각 ChangeRecord를 역순으로 (1) DB 복원, (2) `setTasks`/`setLinks` React 상태 복원, (3) `ganttApiRef.exec()` SVAR 스토어 복원의 3단계로 직접 처리한다. 링크 생성의 Undo는 삭제, 링크 삭제의 Undo는 재생성이다.

**그룹 Undo·스택:** undoStack/redoStack 2스택 구조. 새 액션 발생 시 redoStack을 비운다. `maxHistory = 50` 초과 시 가장 오래된 Command를 삭제하고, 프로젝트 전환·페이지 이동 시 스택을 초기화한다. DB가 진실의 원천이므로 Undo는 DB에도 이전 상태를 저장해 다른 사용자에게 revert가 전파된다. v1 범위는 날짜/기간 변경 + 링크 생성/삭제이며, 태스크 생성/삭제 Undo와 오프라인 큐잉은 v2다.

통합 지점: `undoManager.ts`(신규), `ganttIntercepts.js`(진입 시 `beginTransaction`), `ganttScheduler.js`(cascade 루프에서 record), `GanttView.jsx`(`executeUndo/Redo`+키보드+commit). `taskObservers.ts`/`taskEventEmitter.ts`는 변경 없음. 키보드: `Ctrl/Cmd+Z`=Undo, `Ctrl/Cmd+Shift+Z`/`Ctrl+Y`=Redo.

| 데이터 종류 | 동기화 경로 | 근거 |
|-------------|-------------|------|
| 태스크/링크 CRUD, 날짜·의존성 (구조적 데이터) | **postgres_changes** | DB가 스케줄링 결과의 SSOT, 별도 서버 불필요 |
| 프레즌스(누가 편집 중), 커서/선택 위치 | **Yjs (awareness)** | 휘발성·고빈도, DB 영속화 불필요 |
| 동시 텍스트 편집 (자유 텍스트) | **Yjs (Y.Text)** — 장기 | CRDT가 텍스트 병합에 강함 |

간트 데이터는 트리(`parent_id` 인접 리스트) + 그래프(`links`) 구조라 OT의 변환 함수 정의가 복잡한 반면 CRDT(Yjs)는 구조적 병합을 자동 보장한다. 다만 구조적 데이터의 SSOT는 DB이므로 현 단계에서 주 경로는 postgres_changes다. Yjs와 `useRealtimeSync`가 같은 데이터를 이중 동기화하면 에코·충돌이 생기므로, Yjs는 프레즌스/커서 전용으로 한정한다.

### 5.1 postgres_changes 동기화 (구현 완료)

구현 파일: `src/hooks/useRealtimeSync.ts`. 마운트 시 해당 프로젝트의 `tasks`/`links` 테이블 변경을 구독한다.

```
supabase.channel(`data-sync:${projectId}`)
  .on('postgres_changes', { event: '*', table: 'tasks', filter: `project_id=eq.${projectId}` }, handler)
  .on('postgres_changes', { event: '*', table: 'links', filter: `project_id=eq.${projectId}` }, handler)
  .subscribe()
```

UPDATE/INSERT/DELETE 모두 React 상태 갱신(`setTasks`/`setLinks`)과 SVAR 내부 스토어 갱신(`exec`)을 **항상 함께** 수행한다. publication에 두 테이블이 등록되어 있어야 이벤트가 발화한다(`supabase/migrations/20260610000000_enable_realtime.sql`):

```sql
alter publication supabase_realtime add table public.tasks;
alter publication supabase_realtime add table public.links;
```

### 5.2 에코 제거

원격 UPDATE가 자신이 방금 보낸 변경의 에코인지 판별해 불필요한 재렌더링을 막는다. **`start` / `duration` / `text` 3개 키 필드가 모두 일치하면 건너뛴다.**

```typescript
const localTask = tasksRef.current?.find((t) => String(t.id) === taskId);
if (localTask) {
  const sameStart    = toMs(localTask.start ?? localTask.start_date) === toMs(newRow.start_date);
  const sameDuration = localTask.duration === newRow.duration;
  const sameText     = localTask.text === newRow.text;
  if (sameStart && sameDuration && sameText) return; // 자신의 에코 — 무시
}
```

날짜 비교는 `toMs()`로 밀리초 변환 후 수행해 포맷 차이를 흡수한다. 키 필드 외(제약 조건, CPM 필드)만 변경된 경우 에코로 오인할 수 있어, 향후 `version`/`updated_at` 기반 비교로 정밀화하는 것이 바람직하다 (상세는 03-로드맵 참조). 관련 수용 기준은 `docs/00-개요/04` AC-07.

### 5.3 프레즌스 (UI 미연결)

구현 파일: `src/hooks/usePresence.ts`. Supabase Realtime Presence로 "누가 어떤 태스크를 편집 중인지"를 공유한다. `onlineUsers`(userId → PresenceState), `updateEditingTask(taskId?)`, `getUsersEditingTask(taskId)` API를 제공한다. 훅 로직은 완료됐으나 `GanttView.jsx`가 호출하지 않아 간트 UI에 미연결 상태다. 태스크 행 아바타/배지와 소프트 락 UX(편집 중 경고, 하드 블록 아님) 연결이 예정되어 있다 (상세는 03-로드맵 참조).

### 5.4 충돌 해결 (낙관적 잠금)

구현 파일: `src/lib/optimisticLocking.ts`. `tasks.version` 컬럼과 `update_task_with_version` RPC로 무결성을 보장한다.

```
클라이언트 읽기  → version = N
클라이언트 수정  → updateTaskWithOptimisticLock(taskId, N, updates)
서버 RPC        → version == N → UPDATE + version=N+1, 성공
                → version != N → Version conflict (최신 버전 재읽기 후 최대 3회 재시도)
```

`src/lib/tasks.ts`의 `updateTask()`가 RPC를 거치지 않고 `version` 검증 없이 직접 UPDATE하는 경로가 잔존해, 동시 편집 시 Lost Update가 발생할 수 있다. RPC 전환과 충돌 시 사용자 선택 다이얼로그("내 변경 유지 / 서버 사용")가 예정되어 있다 (상세는 03-로드맵 참조). AC-08 요건상 충돌 시 사용자 변경 내용은 임시 보존되어야 하며 유실 금지다(`docs/00-개요/04` AC-08).

### 5.5 오프라인 복귀

`useYjs.ts`의 `sync` 이벤트로 동기화 상태를 추적하고, Supabase 채널은 네트워크 복구 시 자동 재구독한다. 단 오프라인 기간 중 발생한 원격 변경은 수신되지 않으므로 별도 재동기화가 필요하다. 완전한 오프라인 지원(y-indexeddb 퍼시스턴스, 오프라인 델타 CRDT 병합)과 오프라인 상태 배너는 예정되어 있다 (상세는 03-로드맵 참조). 관련 수용 기준은 `docs/00-개요/04` AC-10.

**RBAC:** 프로젝트 멤버 역할은 `project_members.role` 단일 enum **`owner` / `editor` / `viewer`** 다. RLS 표준 패턴은 SELECT=`is_project_member(project_id,'viewer')`, 쓰기=`is_project_member(project_id,'editor')`(일부 owner 전용)으로 DB 레벨에서 권한을 강제한다.

## 6. 도메인 규칙 요약

### 6.1 날짜 처리

- 스케줄링 계산은 `src/lib/dateUtils.ts`의 UTC 헬퍼(`parseToUTC`, `addDaysUTC`)를 사용한다(DST 영향 없음).
- 엔진 내부 종료일(end)은 **배타적(exclusive)** — `start + duration`.
- 표시 레이어(`GanttView.jsx`)에서 `start + duration - 1`을 23:59:59로 변환(포함적).
- 시작일은 영속화 전 `00:00:00`으로 정규화한다.
- `workingDays = 7` (달력 일수, 주말 미제외). 주말·공휴일 제외 계산은 개선 예정 (상세는 03-로드맵 참조).

### 6.2 제약 조건

`asap`(기본값, 완료) / `mso` / `mfo` / `snet` / `fnlt`(완료) / `alap`(미완성). 상세 동작은 §3.3.

### 6.3 데이터 모델

- `tasks`: `parent_id`를 통한 인접 리스트, CPM 필드(`early_start`, `late_finish`, `total_float`, `is_critical` — 계산 미구현), 낙관적 잠금용 `version` 컬럼.
- `links`: 정규화된 `source` / `target` / `type` / `lag`.
- `projects`: 프로젝트 메타데이터.
- 타입 정의: `src/types/database.ts`. 상세 스키마는 별도 DB 스키마 문서.

### 6.4 링크 타입 코드 (혼동 주의)

`e2s`=FS, `s2s`=SS, `e2e`=FF, `s2e`=SF (SVAR 규약, `src/lib/constants.ts`).
