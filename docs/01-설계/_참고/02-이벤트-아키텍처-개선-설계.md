## 2. DB 저장 실패와 UI 업데이트 분리 설계 (H-1)

### 현재 구조의 문제

`taskEventEmitter.ts`의 `emit()`은 `Promise.all(listeners.map(l => l(event)))`로 모든 리스너를 병렬 await합니다. `DBObserver`가 Supabase 네트워크 오류로 실패하면 `Promise.all` 전체가 reject되어 `UIObserver`도 실행되지 않습니다.

```
emit('task:updated', event)
  → Promise.all([DBObserver, UIObserver, ScheduleObserver])
  → DBObserver 실패(네트워크 오류)
  → Promise.all reject → UIObserver 실행 안 됨
  → 사용자 화면: 드래그한 태스크가 원래 위치로 돌아가는 것처럼 보임
```

추가로, `handleLinkCreated`에서 `link.id = data.id`로 공유 객체를 직접 변이(mutation)시켜 이후 리스너가 오염된 객체를 받는 구조도 있습니다.

### 개선 설계: 관심사 분리

**원칙**: DB 실패는 낙관적 UI를 유지한 채 에러 알림만 표시한다.

```typescript
// src/lib/taskObservers.ts (수정 후)

export class DBObserver {
  async handleTaskUpdated(event: TaskEvent): Promise<void> {
    const { task } = event.payload;
    try {
      await updateTask(task);
    } catch (error) {
      // DB 실패를 상위로 전파하지 않고 여기서 처리
      console.error('[DBObserver] task 저장 실패:', error);
      Sentry.captureException(error, { extra: { taskId: task.id } });
      // 에러 토스트를 위한 별도 이벤트 emit
      uiEventEmitter.emit('toast:error', {
        message: '변경사항 저장에 실패했습니다. 잠시 후 재시도합니다.',
        retryAction: () => updateTask(task),
      });
      // 실패 시 UI 롤백을 원하면 여기서 이전 값으로 되돌리는 이벤트 emit
      // uiEventEmitter.emit('task:rollback', { taskId: task.id });
    }
  }

  async handleLinkCreated(event: LinkEvent): Promise<void> {
    const { link } = event.payload;
    try {
      const savedLink = await createLink(link);
      // 공유 객체 직접 변이 대신 새 객체 반환 패턴 사용
      return savedLink; // UIObserver/ScheduleObserver는 독립적으로 처리
    } catch (error) {
      console.error('[DBObserver] link 생성 실패:', error);
      Sentry.captureException(error);
      uiEventEmitter.emit('toast:error', { message: '링크 저장에 실패했습니다.' });
    }
  }
}
```

**emit 계층 수정** (`taskEventEmitter.ts`):

```typescript
// src/lib/taskEventEmitter.ts

async emit(type: EventType, payload: unknown): Promise<void> {
  const event = { type, payload, timestamp: Date.now() };
  const listeners = this.listeners.get(type) ?? [];

  // DB 실패가 UI를 막지 않도록 각 리스너를 독립적으로 실행
  // DBObserver는 내부에서 catch하므로 여기서 에러가 전파되지 않음
  await Promise.allSettled(listeners.map(listener => listener(event)));
}
```

`Promise.all` 대신 `Promise.allSettled`를 사용하면 개별 리스너 실패가 다른 리스너를 막지 않습니다. DBObserver 내부에서도 catch하는 이중 방어 구조입니다.

### 에러 토스트 emit 흐름

```
DBObserver.handleTaskUpdated()
  → Supabase 쓰기 실패
  → catch 블록에서 uiEventEmitter.emit('toast:error', {...})
  → ToastObserver (별도 등록) → React 토스트 컴포넌트 표시
  → UIObserver는 독립적으로 낙관적 상태 유지 (이미 실행 완료)
```

`uiEventEmitter`는 `taskEventEmitter`와 별도의 싱글턴으로, UI 전용 이벤트만 처리합니다. 또는 `taskEventEmitter`에 `'ui:toast:error'` 타입을 추가하는 방법도 가능합니다.

## 3. 스케줄링 Cascade DB 배치 처리 설계 (H-2)

### 현재 N회 개별 DB 왕복의 문제

```javascript
// ganttScheduler.js 현재 방식 (의사코드)
async function recalculate(taskId, taskData, visited = new Set()) {
  const successors = getSuccessors(taskId);
  for (const successor of successors) {
    const updated = applyConstraint(successor, taskData);
    await handleTaskUpdate(updated, skipRecalculation: true);  // ← 매번 DB 쓰기
    await recalculate(successor.id, updated, visited);          // ← 재귀
  }
}
```

task 10개가 선형 의존성 체인(A→B→C→...→J)으로 연결된 경우 A를 이동하면 9회 순차 DB 왕복이 발생합니다. 각 왕복이 50~200ms라면 1.8초 지연이 생깁니다.

### 개선안: cascade 완료 후 bulk upsert

```javascript
// src/lib/ganttScheduler.js (수정 후)

export function createRecalculateFunction({ tasksRef, linksRef, ganttApiRef, isSchedulerUpdateRef }) {
  return async function recalculateAffectedTasks(taskId, taskData) {
    const updatedTasks = new Map(); // id → updatedTask (cascade 중 수집)

    // 1단계: SVAR UI 업데이트 + 변경 수집 (DB 쓰기 없이)
    function propagate(currentId, currentData, visited = new Set()) {
      if (visited.has(currentId)) return;
      visited.add(currentId);

      const successorLinks = linksRef.current.filter(l => l.source === currentId);
      for (const link of successorLinks) {
        const successorTask = tasksRef.current.find(t => t.id === link.target)
          ?? updatedTasks.get(link.target);
        if (!successorTask) continue;

        const updatedSuccessor = applyConstraint(successorTask, currentData, link);
        updatedTasks.set(updatedSuccessor.id, updatedSuccessor);

        // SVAR 내부 스토어에 즉시 반영 (UI는 빠르게)
        if (ganttApiRef.current) {
          isSchedulerUpdateRef.current = true;
          ganttApiRef.current.exec('update-task', {
            id: updatedSuccessor.id,
            start: updatedSuccessor.start,
            end: updatedSuccessor.end,
            duration: updatedSuccessor.duration,
          });
          isSchedulerUpdateRef.current = false;
        }

        propagate(updatedSuccessor.id, updatedSuccessor, visited);
      }
    }

    propagate(taskId, taskData);

    // 2단계: cascade 완료 후 한 번에 bulk upsert
    if (updatedTasks.size > 0) {
      const tasksToUpdate = Array.from(updatedTasks.values()).map(t => ({
        id: t.id,
        start: normalizeToMidnight(t.start),
        end: t.end,
        duration: t.duration,
        version: (t.version ?? 0) + 1,
        updated_at: new Date().toISOString(),
      }));

      try {
        const { error } = await supabase
          .from('tasks')
          .upsert(tasksToUpdate, { onConflict: 'id' });
        if (error) throw error;
      } catch (err) {
        console.error('[Scheduler] bulk upsert 실패:', err);
        Sentry.captureException(err);
        uiEventEmitter.emit('toast:error', {
          message: '일정 재계산 저장에 실패했습니다.',
        });
      }
    }
  };
}
```

### 변경 요약

| 항목 | 현재 | 개선 후 |
|------|------|---------|
| DB 쓰기 횟수 | N회 (후행 task 수만큼) | 1회 (bulk upsert) |
| UI 갱신 시점 | DB 쓰기 완료 후 | cascade 중 즉시 (UI 먼저, DB 나중) |
| 실패 처리 | 개별 실패가 cascade 중단 | bulk 실패 시 UI는 유지, 에러 토스트 |

## 4. GanttView.jsx 컴포넌트 분리 설계 (M-1)

### 현재 529줄 단일 파일의 문제

`GanttView.jsx`가 다음을 모두 담당합니다:
1. `tasks`, `links` 상태 및 `tasksRef`, `linksRef` ref 관리
2. `handleTaskCreate`, `handleTaskUpdate`, `handleTaskDelete`, `handleLinkCreate`, `handleLinkDelete` 핸들러
3. `recalculateAffectedTasks` useMemo 생성
4. `useGanttObservers`, `useRealtimeSync`, `usePresence` 훅 초기화
5. `processedTasks` 변환 (Date 객체 생성, end date 계산)
6. SVAR Gantt 렌더링

Brain과 View의 경계가 파일 내부에 혼재하여 테스트 격리가 어렵습니다.

### 분리 계획

```
src/
  components/
    GanttView.jsx          ← 렌더링 전용 (목표: 150줄 이하)
  hooks/
    useGanttState.js       ← 상태 + refs 관리
    useGanttEventHandlers.js ← 이벤트 핸들러
    useGanttObservers.js   ← 기존 (옵저버 등록/해제)
    useRealtimeSync.ts     ← 기존 (실시간 동기화)
```

### 각 파일의 책임

**`useGanttState.js`**
- `tasks`, `links` useState
- `tasksRef`, `linksRef`, `ganttApiRef`, `isSchedulerUpdateRef` useRef
- `processedTasks` useMemo (Date 변환)
- `setTasks`, `setLinks`를 외부에 노출

```javascript
// src/hooks/useGanttState.js
export function useGanttState({ initialTasks, initialLinks }) {
  const [tasks, setTasks] = useState(initialTasks ?? []);
  const [links, setLinks] = useState(initialLinks ?? []);
  const tasksRef = useRef(tasks);
  const linksRef = useRef(links);
  const ganttApiRef = useRef(null);
  const isSchedulerUpdateRef = useRef(false);

  // ref를 state와 동기화
  useEffect(() => { tasksRef.current = tasks; }, [tasks]);
  useEffect(() => { linksRef.current = links; }, [links]);

  const processedTasks = useMemo(() => tasks.map(task => ({
    ...task,
    start_date: new Date(task.start),
    end_date: new Date(new Date(task.start).getTime() + (task.duration - 1) * 86400000 + 86399000),
  })), [tasks]);

  return {
    tasks, setTasks, links, setLinks,
    tasksRef, linksRef, ganttApiRef, isSchedulerUpdateRef,
    processedTasks,
  };
}
```

**`useGanttEventHandlers.js`**
- `handleTaskCreate`, `handleTaskUpdate`, `handleTaskDelete`
- `handleLinkCreate`, `handleLinkDelete`
- `recalculateAffectedTasks` useMemo (안정적 ref 의존, 재초기화 방지)

```javascript
// src/hooks/useGanttEventHandlers.js
export function useGanttEventHandlers({ tasksRef, linksRef, ganttApiRef, isSchedulerUpdateRef }) {
  // recalculateAffectedTasks는 stable ref에만 의존 → 재초기화 없음
  const recalculateAffectedTasks = useMemo(
    () => createRecalculateFunction({ tasksRef, linksRef, ganttApiRef, isSchedulerUpdateRef }),
    [] // eslint-disable-line react-hooks/exhaustive-deps
  );

  const handleTaskUpdate = useCallback((task, options = {}) => {
    taskEventEmitter.emit('task:updated', { task, ...options });
    if (!options.skipRecalculation) {
      recalculateAffectedTasks(task.id, task);
    }
  }, [recalculateAffectedTasks]);

  return { handleTaskCreate, handleTaskUpdate, handleTaskDelete,
           handleLinkCreate, handleLinkDelete, recalculateAffectedTasks };
}
```

**`GanttView.jsx` (렌더링 전용)**
- 두 훅을 조합하여 SVAR Gantt에 props 전달
- `ganttIntercepts.js` 연결

```jsx
// src/components/GanttView.jsx (목표 구조)
'use client';
export default function GanttView({ initialTasks, initialLinks, projectId }) {
  const state = useGanttState({ initialTasks, initialLinks });
  const handlers = useGanttEventHandlers(state);

  useGanttObservers({ ...state, ...handlers });
  useRealtimeSync({ ...state, projectId });

  return (
    <Gantt
      tasks={state.processedTasks}
      links={state.links}
      onDataUpdated={ref => { state.ganttApiRef.current = ref; }}
      // ... intercepts
    />
  );
}
```

### 이전 시 주의사항

- `recalculateAffectedTasks`의 `useMemo([], [])` 빈 의존성은 반드시 유지합니다. 안정적 ref 참조 패턴이 깨지면 옵저버가 재등록되어 중복 실행 버그가 발생합니다.
- `useGanttObservers`에 전달하는 `setTasks`, `setLinks`는 useState setter이므로 참조 안정성이 보장되지만, 의존성 배열에 추가하면 lint 경고가 발생합니다. 주석으로 의도를 명시합니다.
- 분리 순서: `useGanttState` 먼저 추출 → 테스트 통과 확인 → `useGanttEventHandlers` 추출 순으로 진행합니다.

## 5. SVAR API 어댑터 레이어 설계 (M-2)

### ganttBridge.js 신설 이유

현재 `ganttApiRef.current.exec('update-task', ...)` 호출이 다음 파일들에 분산되어 있습니다:
- `useGanttObservers.js`
- `useRealtimeSync.ts`
- (향후 추가될 기능들)

SVAR React Gantt 버전 업그레이드 시 `exec()` API 시그니처나 이벤트 이름이 변경되면 모든 파일을 수정해야 합니다. `ganttBridge.js` 하나만 수정하면 전파되도록 격리합니다.

### ganttBridge.js 인터페이스 정의

```javascript
// src/lib/ganttBridge.js

/**
 * SVAR Gantt 내부 API 어댑터.
 * ganttApiRef.current.exec() 직접 호출은 이 파일 외에서 하지 않습니다.
 *
 * @param {React.MutableRefObject} ganttApiRef - SVAR API 객체를 담은 ref
 * @param {React.MutableRefObject} isSchedulerUpdateRef - 인터셉트 재진입 방지 플래그
 */
export function createGanttBridge(ganttApiRef, isSchedulerUpdateRef) {
  function withGuard(fn) {
    return (...args) => {
      if (!ganttApiRef.current) return;
      isSchedulerUpdateRef.current = true;
      try {
        fn(...args);
      } finally {
        isSchedulerUpdateRef.current = false;
      }
    };
  }

  return {
    /**
     * @param {{ id: string, start: Date, end: Date, duration: number, text?: string }} task
     */
    updateTask: withGuard((task) => {
      ganttApiRef.current.exec('update-task', {
        id: task.id,
        start: task.start,
        end: task.end,
        duration: task.duration,
        ...(task.text !== undefined && { text: task.text }),
      });
    }),

    /**
     * @param {{ id: string, text: string, start: Date, end: Date, duration: number }} task
     */
    addTask: withGuard((task) => {
      ganttApiRef.current.exec('add-task', task);
    }),

    /**
     * @param {string} taskId
     */
    deleteTask: withGuard((taskId) => {
      ganttApiRef.current.exec('delete-task', { id: taskId });
    }),

    /**
     * @param {{ id: string, source: string, target: string, type: string, lag?: number }} link
     */
    addLink: withGuard((link) => {
      ganttApiRef.current.exec('add-link', link);
    }),

    /**
     * @param {string} linkId
     */
    deleteLink: withGuard((linkId) => {
      ganttApiRef.current.exec('delete-link', { id: linkId });
    }),
  };
}
```

### Worker와 메인 스레드 간 메시지 프로토콜

```typescript
// src/types/worker.ts

// 메인 → Worker
type ScheduleWorkerRequest =
  | {
      type: 'CASCADE';
      requestId: string; // 응답 매칭용
      payload: {
        rootTaskId: string;
        rootTaskData: TaskSnapshot;
        tasks: TaskSnapshot[];
        links: LinkSnapshot[];
      };
    }
  | {
      type: 'FULL_CPM';
      requestId: string;
      payload: {
        tasks: TaskSnapshot[];
        links: LinkSnapshot[];
        projectDeadline?: string; // ISO 날짜
      };
    };

// Worker → 메인
type ScheduleWorkerResponse =
  | {
      type: 'CASCADE_RESULT';
      requestId: string;
      payload: {
        updatedTasks: Record<string, TaskSnapshot>; // id → 갱신된 task
      };
    }
  | {
      type: 'CPM_RESULT';
      requestId: string;
      payload: {
        tasks: (TaskSnapshot & {
          early_start: string;
          late_finish: string;
          total_float: number;
          is_critical: boolean;
        })[];
      };
    }
  | {
      type: 'ERROR';
      requestId: string;
      payload: { message: string };
    };

// Date는 Worker로 전송 불가 (structuredClone에서 변환됨)
// 날짜는 ISO 문자열로 직렬화
type TaskSnapshot = {
  id: string;
  start: string;    // ISO string
  end: string;      // ISO string
  duration: number;
  constraint_type?: string;
  constraint_date?: string;
};
```

### 점진적 전환 전략 (기능 플래그 없이)

Worker 전환은 `createRecalculateFunction`의 반환 함수 내부에서 전환하므로 호출부는 변경되지 않습니다.

**1단계**: Worker 파일 생성, `ganttScheduler.js`에서 Worker 인스턴스 생성, 기존 동기 로직은 유지

**2단계**: `computeCascade` 로직을 Worker로 이전, Promise 래퍼로 메인 스레드와 동일한 인터페이스 제공

```javascript
// ganttScheduler.js 2단계 전환 패턴
function scheduleViaWorker(payload) {
  return new Promise((resolve, reject) => {
    const requestId = crypto.randomUUID();
    function handler(e) {
      if (e.data.requestId !== requestId) return;
      worker.removeEventListener('message', handler);
      if (e.data.type === 'ERROR') reject(new Error(e.data.payload.message));
      else resolve(e.data.payload);
    }
    worker.addEventListener('message', handler);
    worker.postMessage({ type: 'CASCADE', requestId, payload });
  });
}
```

**3단계**: 동기 fallback 코드 제거, 전체 CPM 계산(Forward/Backward Pass) Worker에 추가

Worker 도입 전제 조건: `scheduling.ts`가 DOM/React 의존성이 없는 순수 함수여야 합니다 (현재 이미 충족).

### .github/workflows/ci.yml 전체 구성

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint:
    name: Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run ESLint
        run: npm run lint

  test:
    name: Unit Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests with coverage
        run: npm test -- --coverage --ci --forceExit
        env:
          CI: true

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage-report
          path: coverage/
          retention-days: 7

  build:
    name: Build Check
    runs-on: ubuntu-latest
    needs: [lint, test]
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Copy SVAR Gantt CSS
        run: cp node_modules/@svar-ui/react-gantt/dist/index.css src/styles/gantt-svar.css

      - name: Build
        run: npm run build
        env:
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}

  audit:
    name: Security Audit
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run security audit
        run: npm audit --audit-level=high
        continue-on-error: true  # 초기에는 경고만, 안정화 후 false로 변경
```

### PR 필수 통과 조건

GitHub Repository Settings → Branches → Branch protection rules에서 `main` 브랜치에 다음 설정:

| 조건 | 설정값 |
|------|--------|
| Require status checks to pass | `lint`, `test`, `build` |
| Require branches to be up to date | 활성화 |
| Restrict force pushes | 활성화 |
| Do not allow bypassing the above settings | 활성화 |

`audit` 잡은 초기에는 필수 조건에서 제외하고, `npm audit --audit-level=high`가 안정적으로 통과된 후 추가합니다.

|------|------|------|------|
