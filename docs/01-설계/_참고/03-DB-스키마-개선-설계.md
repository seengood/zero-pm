## 2. 핵심 인덱스 마이그레이션 설계 (H-1)

### 마이그레이션 파일명: `20260611000001_add_critical_indexes.sql`

현재 `supabase/migrations/` 전체를 검토한 결과, `20251205000000_add_task_constraints.sql`에 `idx_tasks_constraint` 하나만 존재한다. 아래 7개 인덱스가 없으면 태스크 수 증가 시 모든 핵심 경로가 Sequential Scan으로 저하된다.

```sql
-- ============================================================
-- Migration: 20260611000001_add_critical_indexes.sql
-- Purpose: 핵심 쿼리 경로 인덱스 추가 (CONCURRENTLY - 테이블 잠금 없음)
-- ============================================================

-- [1] tasks.project_id
-- 해결 쿼리: getTasks() → SELECT * FROM tasks WHERE project_id = $1
-- RLS: tasks 정책의 EXISTS(SELECT 1 FROM projects WHERE ...) 서브쿼리 비용 절감
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_project_id
  ON tasks(project_id);

-- [2] tasks.parent_id (NULL 행 제외 - partial index로 크기 절감)
-- 해결 쿼리: WBS 트리 탐색, Summary Task 하위 태스크 조회
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_parent_id
  ON tasks(parent_id) WHERE parent_id IS NOT NULL;

-- [3] tasks(project_id, sort_order) - 복합 인덱스
-- 해결 쿼리: getTasks() ORDER BY sort_order — 정렬 포함 커버링 스캔
-- NULLS LAST는 인덱스 정의에 반영 (쿼리와 일치해야 Index Only Scan 가능)
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_project_sort
  ON tasks(project_id, sort_order NULLS LAST);

-- [4] links.project_id
-- 해결 쿼리: getLinks() → SELECT * FROM links WHERE project_id = $1
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_project_id
  ON links(project_id);

-- [5] links.source
-- 해결 쿼리: 스케줄링 엔진 ganttScheduler.js — 선행 태스크 기준 후행 링크 조회
-- 이 인덱스가 없으면 cascade scheduling 전파 시마다 Full Scan 발생
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_source
  ON links(source);

-- [6] links.target
-- 해결 쿼리: 역방향 종속성 탐색 (CPM 역방향 패스, 선행 태스크 조회)
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_target
  ON links(target);

-- [7] links(source, project_id) - 복합 인덱스
-- 해결 쿼리: 스케줄링 엔진이 특정 프로젝트 내 source 기준 후행 조회 시
-- source 단독 인덱스[5]에 project_id 필터가 추가될 때 Index Only Scan 가능
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_source_project
  ON links(source, project_id);

-- 검증 쿼리 (적용 후 실행)
-- SELECT indexname, tablename FROM pg_indexes
-- WHERE tablename IN ('tasks', 'links')
-- ORDER BY tablename, indexname;
```

### 각 인덱스 해결 경로 요약

| 인덱스 | 직접 수혜 쿼리 | RLS 수혜 |
|---|---|---|
| `idx_tasks_project_id` | `getTasks()` | tasks 전체 정책 (EXISTS 서브쿼리) |
| `idx_tasks_parent_id` | WBS 트리 탐색 | - |
| `idx_tasks_project_sort` | `getTasks() ORDER BY sort_order` | tasks SELECT 정책 |
| `idx_links_project_id` | `getLinks()` | links 전체 정책 |
| `idx_links_source` | `ganttScheduler.js` 후행 링크 조회 | - |
| `idx_links_target` | CPM 역방향 패스 | - |
| `idx_links_source_project` | 스케줄링 엔진 복합 필터 | - |

> **주의**: `CONCURRENTLY` 옵션은 트랜잭션 블록 안에서 실행할 수 없다. `supabase db push` 시 자동으로 트랜잭션을 래핑하므로, 이 마이그레이션은 `supabase migration up --include-all` 또는 직접 psql로 실행해야 한다. Supabase Dashboard의 SQL Editor에서 직접 실행하는 것이 가장 안전하다.

## 3. 낙관적 잠금 일관성 수정 (H-2)

### 현재 문제 코드 (`src/lib/tasks.ts`)

```typescript
// 문제: 버전 확인 없는 무조건 UPDATE → Lost Update 취약
export async function updateTask(
  taskId: string,
  updates: Partial<Task>,
  supabase: SupabaseClient = defaultSupabase
): Promise<{ data: Task | null; error: any }> {
  const { data, error } = await supabase
    .from('tasks')
    .update(updates)       // version 확인 없음
    .eq('id', taskId)
    .select()
    .single();
  // ...
}
```

**문제점**: `update_task_with_version` RPC가 DB에 구현되어 있고 `src/lib/optimisticLocking.ts`도 존재하지만, 실제 데이터 접근 레이어인 `tasks.ts`의 `updateTask()`는 이를 전혀 사용하지 않는다. 두 사용자가 동시에 같은 태스크를 수정하면 마지막 쓰기가 이전 것을 덮어쓴다(Lost Update).

### 개선된 코드 (`src/lib/tasks.ts`)

```typescript
// 버전 충돌 에러 타입 정의
export interface VersionConflictError {
  type: 'VERSION_CONFLICT';
  currentVersion: number;
  expectedVersion: number;
}

export type UpdateTaskResult =
  | { data: Task; error: null }
  | { data: null; error: string | VersionConflictError };

// 개선: update_task_with_version RPC 호출 방식
export async function updateTask(
  taskId: string,
  updates: Partial<Task>,
  supabase: SupabaseClient = defaultSupabase
): Promise<UpdateTaskResult> {
  // version이 없으면 RPC 없이 직접 업데이트 (스케줄러 내부 호출 등 하위 호환)
  if (updates.version === undefined) {
    const { data, error } = await supabase
      .from('tasks')
      .update({ ...updates, updated_at: new Date().toISOString() })
      .eq('id', taskId)
      .select()
      .single();
    if (error) {
      console.error('Error updating task:', taskId, error);
      return { data: null, error: error.message };
    }
    return { data: data as Task, error: null };
  }

  const expectedVersion = updates.version;
  // version 필드는 RPC가 관리하므로 updates에서 제거
  const { version: _v, ...updateFields } = updates;

  const { data: rpcResult, error: rpcError } = await supabase
    .rpc('update_task_with_version', {
      p_task_id: taskId,
      p_expected_version: expectedVersion,
      p_updates: updateFields,
    });

  if (rpcError) {
    console.error('Error calling update_task_with_version RPC:', taskId, rpcError);
    return { data: null, error: rpcError.message };
  }

  if (!rpcResult.success) {
    if (rpcResult.error === 'Version conflict') {
      // 409 Conflict에 해당하는 구조화된 에러 반환
      return {
        data: null,
        error: {
          type: 'VERSION_CONFLICT',
          currentVersion: rpcResult.current_version,
          expectedVersion: rpcResult.expected_version,
        },
      };
    }
    return { data: null, error: rpcResult.error };
  }

  // RPC 성공 후 최신 태스크 상태 반환
  const { data: freshTask, error: fetchError } = await supabase
    .from('tasks')
    .select('*')
    .eq('id', taskId)
    .single();

  if (fetchError) {
    return { data: null, error: fetchError.message };
  }
  return { data: freshTask as Task, error: null };
}
```

### 버전 충돌(409) 처리 — 클라이언트 재시도 패턴

충돌 발생 시 최신 서버 상태를 re-fetch하여 사용자에게 알리고, 재시도를 유도하는 패턴:

```typescript
// GanttView.jsx 또는 useGanttObservers.js의 DBObserver에서 사용 예시
async function handleTaskUpdateWithRetry(
  taskId: string,
  updates: Partial<Task>,
  maxRetries = 3
): Promise<Task | null> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const result = await updateTask(taskId, updates);

    if (result.error === null) {
      return result.data;
    }

    if (
      typeof result.error === 'object' &&
      result.error.type === 'VERSION_CONFLICT'
    ) {
      if (attempt < maxRetries - 1) {
        // 최신 상태 re-fetch 후 version 갱신하여 재시도
        const { data: latestTask } = await supabase
          .from('tasks')
          .select('*')
          .eq('id', taskId)
          .single();

        if (latestTask) {
          updates = { ...updates, version: latestTask.version };
          continue;
        }
      }
      // 최대 재시도 초과 — 사용자에게 충돌 알림 (UI 처리는 호출자 책임)
      console.error('Version conflict after max retries:', taskId);
      return null;
    }

    // 다른 에러 — 재시도 없음
    console.error('updateTask failed:', result.error);
    return null;
  }
  return null;
}
```

> **참고**: `update_task_with_version` RPC는 현재 `text`, `start_date`, `duration`, `progress`, `type` 필드만 업데이트한다(`supabase/migrations/20251203070000_init_schema.sql` 702~704라인). `constraint_type`, `constraint_date`, `parent_id`, `sort_order` 등 추가 필드를 지원하려면 RPC 함수 본문에 해당 필드를 추가해야 한다.

## 4. constraint_date TIMESTAMPTZ 통일 (H-3)

### 마이그레이션 파일명: `20260611000002_fix_constraint_date_timezone.sql`

`20251205000000_add_task_constraints.sql`에서 `constraint_date`가 `TIMESTAMP`(타임존 없음)로 추가되었으나, `tasks` 테이블의 모든 날짜 필드(`start_date`, `early_start`, `late_finish` 등)는 `TIMESTAMPTZ`다. 타임존 혼합 사용은 UTC 기반 스케줄링 엔진(`src/lib/dateUtils.ts`)에서 날짜 연산 오류를 유발할 수 있다.

```sql
-- ============================================================
-- Migration: 20260611000002_fix_constraint_date_timezone.sql
-- Purpose: constraint_date 타입을 TIMESTAMP → TIMESTAMPTZ로 통일
-- ============================================================

-- 기존 인덱스 삭제 (타입 변경 전 필요)
DROP INDEX IF EXISTS idx_tasks_constraint;

-- USING 절: TIMESTAMP 값을 UTC로 해석하여 TIMESTAMPTZ로 변환
-- 기존 데이터가 UTC 기준으로 입력되었다는 가정 (스케줄링 엔진 dateUtils.ts가 UTC 기반)
ALTER TABLE tasks
  ALTER COLUMN constraint_date
  TYPE TIMESTAMPTZ
  USING constraint_date AT TIME ZONE 'UTC';

-- 인덱스 재생성 (기존 idx_tasks_constraint와 동일 경로)
CREATE INDEX IF NOT EXISTS idx_tasks_constraint
  ON tasks(constraint_type, constraint_date);

-- 컬럼 코멘트 갱신
COMMENT ON COLUMN tasks.constraint_date IS
  'Date for constraint (required for mso, mfo, snet, fnlt). TIMESTAMPTZ - UTC 기반 스케줄링 엔진과 일치';

-- 롤백 참고 (supabase/rollbacks/20260611000002_rollback.sql에 저장)
-- ALTER TABLE tasks ALTER COLUMN constraint_date TYPE TIMESTAMP USING constraint_date AT TIME ZONE 'UTC';
```

> **주의**: `ALTER COLUMN ... TYPE`은 테이블 전체 재작성(full table rewrite)을 유발할 수 있다. tasks 테이블이 대용량(수십만 행)인 경우 유지보수 윈도우를 계획해야 한다. 현재 ZeroPM 규모에서는 즉시 적용 가능하다.

## 5. save_cpm_results RPC 개선 (H-4)

### 마이그레이션 파일명: `20260611000003_fix_save_cpm_results.sql`

### 현재 루프 방식의 원자성 문제

```sql
-- 현재 코드 (init_schema.sql, 726~753라인) 의 문제점:
FOR v_task IN SELECT * FROM jsonb_array_elements(p_results)
LOOP
  UPDATE tasks SET early_start = ..., late_finish = ..., ... WHERE id = v_task_id;
  v_updated_count := v_updated_count + 1;
END LOOP;
-- EXCEPTION 블록이 있어도 루프 중 성공한 UPDATE는 이미 커밋됨 (PL/pgSQL 특성)
-- → 50개 중 30개만 업데이트된 불완전 상태가 DB에 남을 수 있음
-- → version 미증가 → CPM 저장 후 클라이언트 버전 캐시와 불일치
```

### 개선된 배치 UPDATE SQL

```sql
-- ============================================================
-- Migration: 20260611000003_fix_save_cpm_results.sql
-- Purpose: save_cpm_results RPC를 원자적 배치 UPDATE로 교체
--          + version 증가 추가
-- ============================================================

CREATE OR REPLACE FUNCTION save_cpm_results(
  p_results JSONB
)
RETURNS JSONB AS $$
DECLARE
  v_updated_count INTEGER;
BEGIN
  -- jsonb_to_recordset: JSONB 배열을 가상 레코드셋으로 변환
  -- 단일 UPDATE 문 = 하나의 트랜잭션 단위 → 원자성 보장
  UPDATE tasks AS t
  SET
    early_start  = v.early_start::TIMESTAMPTZ,
    early_finish = v.early_finish::TIMESTAMPTZ,
    late_start   = v.late_start::TIMESTAMPTZ,
    late_finish  = v.late_finish::TIMESTAMPTZ,
    total_float  = v.total_float::INTEGER,
    is_critical  = v.is_critical::BOOLEAN,
    version      = t.version + 1,   -- 클라이언트 버전 캐시 일치 보장
    updated_at   = NOW()
  FROM jsonb_to_recordset(p_results) AS v(
    id           TEXT,
    early_start  TEXT,
    early_finish TEXT,
    late_start   TEXT,
    late_finish  TEXT,
    total_float  TEXT,
    is_critical  TEXT
  )
  WHERE t.id = v.id::UUID;

  -- 실제 업데이트된 행 수 반환
  GET DIAGNOSTICS v_updated_count = ROW_COUNT;

  RETURN jsonb_build_object(
    'success',       true,
    'updated_count', v_updated_count
  );
EXCEPTION
  WHEN OTHERS THEN
    -- 단일 배치 UPDATE 실패 시 전체 롤백 (원자성)
    RETURN jsonb_build_object(
      'success', false,
      'error',   SQLERRM
    );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- 함수 코멘트
COMMENT ON FUNCTION save_cpm_results(JSONB) IS
  'CPM 계산 결과를 원자적 배치 UPDATE로 저장. version +1 포함. 2026-06-11 개선';
```

### 클라이언트 측 version 동기화

`save_cpm_results` 호출 후 클라이언트의 `tasksRef`에 새 `version` 값을 반영해야 한다. 호출 측 코드(간트 스케줄링 엔진이 CPM을 계산하는 부분)에서 RPC 응답 후 영향받은 태스크를 re-fetch하거나, RPC 응답에 업데이트된 버전 정보를 포함시키는 방식을 고려할 수 있다.

## 6. 소프트 삭제 설계 (M-1)

### 마이그레이션 파일명: `20260611000004_add_soft_delete.sql`

```sql
-- ============================================================
-- Migration: 20260611000004_add_soft_delete.sql
-- Purpose: tasks, links에 소프트 삭제 지원 추가
--          deleted_at IS NULL 조건을 RLS 및 쿼리에 적용
-- ============================================================

-- -----------------------------------------------
-- 1. 컬럼 추가
-- -----------------------------------------------
ALTER TABLE tasks
  ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ DEFAULT NULL;

ALTER TABLE links
  ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ DEFAULT NULL;

-- 삭제된 행 조회용 인덱스 (복구 기능, 감사 조회에 필요)
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_deleted_at
  ON tasks(deleted_at) WHERE deleted_at IS NOT NULL;

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_deleted_at
  ON links(deleted_at) WHERE deleted_at IS NOT NULL;

-- -----------------------------------------------
-- 2. RLS 정책 수정 — tasks
-- 기존 정책을 DROP하고 deleted_at IS NULL 조건 추가하여 재생성
-- -----------------------------------------------

-- SELECT: 삭제되지 않은 태스크만 조회
DROP POLICY IF EXISTS "Users can view tasks in their projects" ON tasks;
CREATE POLICY "Users can view tasks in their projects"
  ON tasks FOR SELECT
  USING (
    deleted_at IS NULL
    AND EXISTS (
      SELECT 1 FROM projects
      WHERE projects.id = tasks.project_id
        AND (
          projects.owner_id = auth.uid()
          OR EXISTS (
            SELECT 1 FROM project_members
            WHERE project_members.project_id = projects.id
              AND project_members.user_id = auth.uid()
          )
        )
    )
  );

-- INSERT: 변경 없음 (신규 행은 deleted_at = NULL)
-- UPDATE: 변경 없음 (소프트 삭제는 UPDATE로 처리됨)
-- DELETE: 물리적 삭제는 관리자/시스템만 — 일반 사용자는 UPDATE로 소프트 삭제

-- -----------------------------------------------
-- 3. RLS 정책 수정 — links
-- -----------------------------------------------

-- 태스크 소프트 삭제 (CASCADE: 연결된 링크도 소프트 삭제)
CREATE OR REPLACE FUNCTION soft_delete_task(p_task_id UUID)
RETURNS JSONB AS $$
DECLARE
  v_now TIMESTAMPTZ := NOW();
BEGIN
  -- 태스크 소프트 삭제
  UPDATE tasks
  SET deleted_at = v_now, updated_at = v_now
  WHERE id = p_task_id AND deleted_at IS NULL;

  IF NOT FOUND THEN
    RETURN jsonb_build_object('success', false, 'error', 'Task not found or already deleted');
  END IF;

  -- 연결된 링크 소프트 삭제 (source 또는 target이 삭제된 태스크인 링크)
  UPDATE links
  SET deleted_at = v_now
  WHERE (source = p_task_id OR target = p_task_id)
    AND deleted_at IS NULL;

  RETURN jsonb_build_object('success', true);
EXCEPTION
  WHEN OTHERS THEN
    RETURN jsonb_build_object('success', false, 'error', SQLERRM);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- 태스크 복구 (소프트 삭제 취소)
CREATE OR REPLACE FUNCTION restore_task(p_task_id UUID)
RETURNS JSONB AS $$
BEGIN
  UPDATE tasks
  SET deleted_at = NULL, updated_at = NOW()
  WHERE id = p_task_id AND deleted_at IS NOT NULL;

  IF NOT FOUND THEN
    RETURN jsonb_build_object('success', false, 'error', 'Task not found or not deleted');
  END IF;

  RETURN jsonb_build_object('success', true);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## 8. project_members 완전 설계 (AC-17 지원)

### 마이그레이션 파일명: `20260611000006_fix_rbac_and_invitations.sql`

### 현재 UserRole enum 불일치 문제

`src/types/database.ts`의 `UserRole`은 `'admin' | 'member' | 'observer'`이고, `project_members.role` DB 컬럼도 동일한 값을 사용한다. 그러나 설계 리뷰에서 지적한 바와 같이, 이 역할 체계는 PM 도구의 권한 모델과 의미적으로 불명확하다:

| 현재 역할 | 의미 불명확성 |
|---|---|
| `admin` | 프로젝트 소유자와 다른 관리자를 어떻게 구분? |
| `member` | 편집 가능 여부가 명시적이지 않음 |
| `observer` | INSERT/UPDATE 차단은 RLS에 하드코딩 — 역할 추가 시 RLS 전체 수정 필요 |

### 통일된 역할 체계

| 역할 | 권한 | 기존 매핑 |
|---|---|---|
| `owner` | 프로젝트 삭제, 멤버 관리, 모든 편집 | 프로젝트 `owner_id` |
| `editor` | 태스크/링크 생성·수정·삭제 | 기존 `admin`, `member` |
| `viewer` | 읽기 전용 | 기존 `observer` |

```sql
-- 초대 수락 함수 (토큰 기반, 인증된 사용자)
CREATE OR REPLACE FUNCTION accept_invitation(p_token TEXT)
RETURNS JSONB AS $$
DECLARE
  v_invitation project_invitations%ROWTYPE;
BEGIN
  -- 유효한 초대 조회
  SELECT * INTO v_invitation
  FROM project_invitations
  WHERE token = p_token
    AND status = 'pending'
    AND expires_at > NOW();

  IF NOT FOUND THEN
    RETURN jsonb_build_object(
      'success', false,
      'error', 'Invalid or expired invitation token'
    );
  END IF;

  -- 이미 멤버인지 확인
  IF EXISTS (
    SELECT 1 FROM project_members
    WHERE project_id = v_invitation.project_id
      AND user_id = auth.uid()
  ) THEN
    -- 이미 멤버이면 초대만 accepted로 처리
    UPDATE project_invitations
    SET status = 'accepted', accepted_at = NOW()
    WHERE id = v_invitation.id;

    RETURN jsonb_build_object(
      'success', true,
      'project_id', v_invitation.project_id,
      'already_member', true
    );
  END IF;

  -- project_members에 추가
  INSERT INTO project_members (project_id, user_id, role)
  VALUES (v_invitation.project_id, auth.uid(), v_invitation.role);

  -- 초대 상태 업데이트
  UPDATE project_invitations
  SET status = 'accepted', accepted_at = NOW()
  WHERE id = v_invitation.id;

  RETURN jsonb_build_object(
    'success', true,
    'project_id', v_invitation.project_id,
    'role', v_invitation.role
  );
EXCEPTION
  WHEN OTHERS THEN
    RETURN jsonb_build_object('success', false, 'error', SQLERRM);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- TypeScript 타입 업데이트 필요 (src/types/database.ts)
COMMENT ON TABLE project_invitations IS
  'AC-17 팀원 초대 기능. token은 서버에서 randomBytes(16).toString(base64url)로 생성.
   7일 만료 기본값. accept_invitation(token) RPC로 수락 처리.';
```

### TypeScript 타입 업데이트 필요사항

`src/types/database.ts`의 `UserRole`을 다음과 같이 수정해야 한다:


### 프로덕션 배포 시 주의사항

1. **001 (인덱스)**: `CONCURRENTLY` 옵션 때문에 트랜잭션 블록 외부에서 실행해야 한다. `supabase db push`는 자동 트랜잭션을 감싸므로, 이 마이그레이션은 Supabase Dashboard SQL Editor 또는 직접 psql로 실행한다.

3. **006 (역할 체계 변경)**: 기존 `observer` → `viewer` 마이그레이션은 데이터 변환이므로 배포 전 백업을 필수로 확보한다. 프론트엔드의 `UserRole` 타입 참조 코드와 동시에 배포해야 한다(TypeScript 코드와 마이그레이션의 동시 배포).

4. **소프트 삭제 004 이후**: `deleteTask()`를 호출하는 모든 클라이언트 코드가 소프트 삭제 RPC로 전환되어야 한다. 마이그레이션과 코드 배포 순서: 마이그레이션 먼저 → 코드 배포 (역방향은 소프트 삭제 RPC가 없는 상태에서 코드가 호출하는 상황 발생).
