# A. 데이터

## 1. DB 설계 원칙

**Supabase(PostgreSQL)** 를 단일 데이터 저장소로 사용한다. 선택 근거:

- 관계형 구조 + 재귀 CTE를 활용하는 **인접 리스트(Adjacency List)** 모델이 간트 차트의 쓰기 성능과 구현 난이도 균형에서 최적이다.
- Supabase Auth → RLS → Realtime 파이프라인이 통합되어 추가 인프라 없이 실시간 협업과 보안을 동시에 구현한다.
- Vercel Hobby + Supabase Free 조합으로 Zero-Cost 운영이 가능하다.

### 1.1 인접 리스트 채택 근거

WBS 계층은 `tasks.parent_id`(자기참조 FK)로 표현한다. 간트 차트는 드래그 이동·삽입이 잦은 **쓰기 중심 워크로드**이므로, 노드 1개 UPDATE로 이동이 끝나는 인접 리스트가 경로 열거(ltree)·중첩 집합 대비 쓰기 비용이 가장 낮다. 트리 조회 성능은 `idx_tasks_parent_id`(Partial Index)와 `idx_tasks_project_sort`로 보완한다.

### 1.2 Vercel 환경과 역할 분담

```
Supabase / Postgres   (중앙 저장소)
  - 영구 데이터: projects, tasks, links, baselines (resources/assignments/costs = 스키마만)
  - RLS: 행 수준 접근 제어 (RBAC owner/editor/viewer)
  - RPC: 원자적 트랜잭션
      update_task_with_version  (낙관적 잠금)
      save_cpm_results          (CPM 배치 저장)
      soft_delete_task / restore_task (소프트 삭제)
      accept_invitation         (초대 수락)

Vercel / Next.js      (클라이언트 + 서버 레이어)
  - CPM/RCPSP 계산: 클라이언트 Web Worker (서버 부하 없음)
  - Optimistic UI: 로컬 상태 즉시 반영 후 DB 동기화
  - API Route / Edge Function: 토큰 생성(공유 링크·초대), 이메일 발송 등 서버 전용 작업

Supabase Realtime
  - postgres_changes: 크로스탭/다중 사용자 동기화 (tasks, links)
  - Presence: 소프트 락 UI
  - Broadcast: Yjs CRDT 델타 교환
```

## 2. 테이블 목록 및 관계

### 2.1 테이블 목록

| 테이블 | 역할 | 구현 상태 |
|---|---|---|
| `projects` | 프로젝트 최상위 컨테이너 | 구현 완료 |
| `tasks` | WBS 태스크 (CPM 필드 포함) | 구현 완료 |
| `links` | 태스크 간 의존성 (그래프 엣지) | 구현 완료 |
| `user_profiles` | 사용자 프로필, 글로벌 역할 | 구현 완료 |
| `project_members` | 프로젝트별 멤버십 + 역할 | 구현 완료 |
| `project_invitations` | 팀원 초대 토큰 관리 | 설계 완료·미구현 |
| `shared_links` | 공유 URL / 섹션 공유 링크 | 설계 완료·미구현 |
| `baselines` / `baseline_tasks` | 기준선 메타 + 태스크 스냅샷 | 부분 구현 (현재 JSONB 단일 컬럼) |
| `resources` / `assignments` / `costs` | 자원·할당·비용 | **스키마만 존재 (기능 미구현)** |

`resources` / `assignments` / `costs`는 스키마 정의만 있고 UI/스케줄러 연동이 없다(제품 Phase 4 영역, 향후 RCPSP/EVM 구현용 데이터 모델). `baselines`는 현재 JSONB 단일 컬럼 방식이며 정규화 복원은 후속 작업이다(상세는 03-로드맵 참조).

### 2.2 관계도

```
auth.users
  ├─ user_profiles (1:1)
  └─ projects (owner_id)
        ├─ project_members (N:M — role: owner/editor/viewer)
        │     └─ project_invitations (초대 토큰)        [설계완료·미구현]
        ├─ tasks (project_id)
        │     ├─ tasks (parent_id — WBS 자기참조)
        │     ├─ links (source/target → tasks.id)
        │     ├─ assignments (task_id → resources.id)   [스키마만]
        │     └─ costs (task_id)                        [스키마만]
        ├─ baselines → baseline_tasks (스냅샷)
        ├─ resources (project_id)                       [스키마만]
        └─ shared_links (project_id)                    [설계완료·미구현]
```

### 3.1 projects

### 3.2 tasks

CPM 결과, WBS 계층, 낙관적 잠금, 소프트 삭제를 모두 담는 핵심 테이블이다.

|---|---|---|---|
| `id` | UUID | PK | 태스크 ID |
| `project_id` | UUID | FK → projects(id), NOT NULL | 소속 프로젝트 |
| `parent_id` | UUID | FK → tasks(id), DEFAULT NULL | WBS 계층 (인접 리스트) |
| `sort_order` | INTEGER | DEFAULT NULL | 같은 부모 내 정렬 순서 |
| `text` | TEXT | NOT NULL | 태스크 이름 |
| `type` | TEXT | DEFAULT 'task' | task / milestone / summary |
| `start_date` | TIMESTAMPTZ | NOT NULL | 계획 시작일 (00:00:00 UTC 정규화) |
| `duration` | INTEGER | NOT NULL DEFAULT 1 | 기간 (캘린더 일수) |
| `progress` | FLOAT | DEFAULT 0.0 | 완료율 (0.0~1.0) |
| `early_start` | TIMESTAMPTZ | DEFAULT NULL | CPM: 가장 빠른 시작일 |
| `early_finish` | TIMESTAMPTZ | DEFAULT NULL | CPM: 가장 빠른 완료일 |
| `late_start` | TIMESTAMPTZ | DEFAULT NULL | CPM: 가장 늦은 시작일 |
| `late_finish` | TIMESTAMPTZ | DEFAULT NULL | CPM: 가장 늦은 완료일 |
| `total_float` | INTEGER | DEFAULT NULL | 총 여유시간 (0 = Critical) |
| `is_critical` | BOOLEAN | DEFAULT FALSE | 임계 경로 여부 |
| `constraint_type` | TEXT | DEFAULT 'asap' | asap/alap/mso/mfo/snet/fnlt |
| `constraint_date` | TIMESTAMPTZ | DEFAULT NULL | 제약 날짜 |
| `version` | INTEGER | NOT NULL DEFAULT 1 | 낙관적 잠금 버전 |
| `deleted_at` | TIMESTAMPTZ | DEFAULT NULL | 소프트 삭제 시각 |
| `created_at` / `updated_at` | TIMESTAMPTZ | NOT NULL DEFAULT NOW() | 생성/수정 시각 |

> **CPM 필드 상태**: `early_start` / `late_finish` / `total_float` / `is_critical`는 **정의만 존재하며 계산 로직은 미구현**이다. 향후 `save_cpm_results` RPC가 클라이언트 Web Worker 계산 결과를 배치 저장한다. 클라이언트의 직접 쓰기는 금지하고 RPC만 허용한다(§9.2).

### 3.3 links

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| `id` | UUID | PK | 링크 ID |
| `project_id` | UUID | FK → projects(id), NOT NULL | 소속 프로젝트 |
| `source` | UUID | FK → tasks(id), NOT NULL | 선행 태스크 |
| `target` | UUID | FK → tasks(id), NOT NULL | 후행 태스크 |
| `type` | TEXT | NOT NULL | e2s(FS)/s2s(SS)/e2e(FF)/s2e(SF) |
| `lag` | INTEGER | DEFAULT 0 | Lag(양수)/Lead(음수), 일 단위 |
| `deleted_at` | TIMESTAMPTZ | DEFAULT NULL | 소프트 삭제 시각 |
| `created_at` | TIMESTAMPTZ | NOT NULL DEFAULT NOW() | 생성 시각 |

> 링크 타입은 SVAR Gantt 규약(e2s/s2s/e2e/s2e). MS Project FS/SS/FF/SF 라벨과 혼동하지 않는다(`src/lib/constants.ts`).

### 3.4 그 외 테이블 요약

- **user_profiles**: `id`(PK, FK → auth.users), `username`, `role`(CHECK owner/editor/viewer, 글로벌 역할). 1:1 연동.
- **project_members**: `id`, `project_id`(FK), `user_id`(FK), `role`(NOT NULL, CHECK owner/editor/viewer), `created_at`. 프로젝트 멤버십 + 역할.
- **project_invitations**(설계완료·미구현): `email`, `role`(editor/viewer — owner 초대 불가), `token`(UNIQUE), `status`(pending/accepted/expired/cancelled), `expires_at`(기본 7일), `UNIQUE(project_id, email, status)`.
- **shared_links**(설계완료·미구현): `token`(UNIQUE, 128비트), `scope`(project/section), `scope_task_ids`(UUID[]), `permission`(view/comment), `expires_at`, `is_active`. §8 참조.
- **baselines / baseline_tasks**: 기준선 메타(번호 0~10) + 태스크 스냅샷(start/finish/duration/cost). 현재 `baselines.data` JSONB 단일 컬럼이며 정규화 복원은 후속(상세는 03-로드맵 참조).
- **resources / assignments / costs**(스키마만): `resources`(capacity), `assignments`(required_units), `costs`(amount/rate). RCPSP/EVM 입력용.

## 4. 인덱스 및 마이그레이션

### 4.1 핵심 인덱스 7개

핵심 쿼리 경로 인덱스는 정확히 **7개**다. 누락 시 태스크 수 증가에 따라 getTasks/getLinks/cascade scheduling/RLS 서브쿼리가 Sequential Scan으로 저하된다. 특히 `idx_links_source`가 없으면 cascade scheduling 전파 시마다 `links` Full Scan이 발생한다.

| # | 인덱스 | 대상 | 목적 |
|---|---|---|---|
| ① | `idx_tasks_project_id` | tasks(project_id) | `getTasks()` WHERE project_id, RLS 서브쿼리 |
| ② | `idx_tasks_parent_id` | tasks(parent_id) WHERE NOT NULL | WBS 트리 탐색(재귀 CTE) |
| ③ | `idx_tasks_project_sort` | tasks(project_id, sort_order NULLS LAST) | 정렬 포함 Index Only Scan |
| ④ | `idx_links_project_id` | links(project_id) | `getLinks()` WHERE project_id |
| ⑤ | `idx_links_source` | links(source) | 스케줄러 후행 링크 조회 |
| ⑥ | `idx_links_target` | links(target) | CPM 역방향 패스, 선행 조회 |
| ⑦ | `idx_links_source_project` | links(source, project_id) | 스케줄러 복합 필터 Index Only Scan |

```sql
-- 핵심 7개 (CONCURRENTLY — 테이블 잠금 없음)
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_project_id     ON tasks(project_id);
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_parent_id      ON tasks(parent_id) WHERE parent_id IS NOT NULL;
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_project_sort   ON tasks(project_id, sort_order NULLS LAST);
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_project_id     ON links(project_id);
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_source         ON links(source);
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_target         ON links(target);
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_links_source_project ON links(source, project_id);
```

> **CONCURRENTLY 주의**: 트랜잭션 블록 내 실행 불가. `supabase db push`는 자동 트랜잭션을 감싸므로, 인덱스 마이그레이션은 Dashboard SQL Editor 또는 직접 psql로 실행한다.

보조 인덱스(핵심 7개와 별도): `idx_tasks_constraint`(constraint_type, constraint_date), `idx_tasks_deleted_at`/`idx_links_deleted_at`(WHERE deleted_at IS NOT NULL), `idx_shared_links_token`, `idx_project_invitations_token` 등.

### 4.2 마이그레이션 6개 (순차 실행)

실행 순서: 001 → 002 → 003 → 004 → 005 → 006.

| 번호 | 파일 | 목적 |
|---|---|---|
| 001 | `20260612000001_add_critical_indexes` | 핵심 7개 인덱스 추가 (CONCURRENTLY) |
| 002 | `20260612000002_fix_constraint_date_timestamptz` | `constraint_date` TIMESTAMP → TIMESTAMPTZ 통일 |
| 003 | `20260612000003_fix_save_cpm_results_batch` | `save_cpm_results` 배치 UPDATE + version+1 |
| 004 | `20260612000004_add_soft_delete` | `deleted_at` + RLS + soft_delete/restore RPC |
| 005 | `20260612000005_add_shared_links` | `shared_links` 테이블 + RLS |
| 006 | `20260612000006_rbac_unify_and_invitations` | 역할 통일(owner/editor/viewer) + invitations + RLS 커버리지 |

배포 주의: ① 인덱스는 CONCURRENTLY로 별도 실행. ② TIMESTAMPTZ 타입 변경은 테이블 재작성을 유발하므로 대용량 시 유지보수 윈도우. ③ 역할 변경(006)은 데이터 변환이므로 배포 전 백업 + 프론트엔드 `UserRole` 타입 동시 배포. ④ 소프트 삭제(004) 이후 `deleteTask()` 호출이 모두 RPC로 전환되어야 하므로 마이그레이션 먼저 → 코드 배포.

## 5. 낙관적 잠금 및 소프트 삭제

### 5.1 낙관적 잠금 (version + RPC)

CRDT가 처리하지 못하는 구조적 변경(Task 삭제, WBS 이동)에 `version` 컬럼 기반 낙관적 잠금을 적용한다. 현재 `src/lib/tasks.ts`의 `updateTask()`가 `version` 검증을 우회한 직접 UPDATE를 수행해 동시 수정 시 Lost Update가 발생할 수 있다. `update_task_with_version` RPC로 전환하여 버전 불일치 시 충돌 오류를 반환한다(구현은 `src/lib/optimisticLocking.ts`).

- `version` 미지정 호출(스케줄러 내부 등)은 직접 UPDATE 유지(하위 호환).
- `version` 지정 호출은 RPC 경로 → 불일치 시 `VERSION_CONFLICT`(현재/기대 버전 포함) 반환.
- 클라이언트는 충돌(409) 수신 시 최신 상태 re-fetch 후 version 갱신하여 최대 3회 재시도. 초과 시 사용자에게 충돌 안내(AC-08 UI).
- 현재 RPC는 text/start_date/duration/progress/type만 갱신하므로 `constraint_type`/`constraint_date`/`parent_id`/`sort_order` 지원을 위해 본문 확장 필요.

### 5.2 소프트 삭제 (deleted_at)

| 항목 | 내용 |
|---|---|
| 컬럼 | `deleted_at TIMESTAMPTZ DEFAULT NULL` (tasks, links) |
| 삭제 | `UPDATE ... SET deleted_at = NOW()` (물리 DELETE 없음) |
| 조회 필터 | 모든 SELECT에 `.is('deleted_at', null)` 적용 |
| RLS | SELECT 정책에 `deleted_at IS NULL` 조건 추가 (이중 보호) |
| 복구 | `restore_task(p_task_id)` RPC |
| Cascade | `soft_delete_task()` 호출 시 연결 링크도 자동 소프트 삭제 |

RLS의 `deleted_at IS NULL`(row-level 방어선)과 쿼리 필터(애플리케이션 방어선)는 의도적 중복이다.

# B. 보안

## 6. 인증 (Supabase Auth)

ZeroPM은 Supabase Auth를 인증 SSOT로 사용한다(Free 티어 월 50,000 MAU).

- **이메일/패스워드**: 이메일 확인 활성화, bcrypt 해시(Supabase 처리). 재설정은 `resetPasswordForEmail()`.
- **OAuth**: Google/GitHub 등. 콜백 `/auth/callback` → `exchangeCodeForSession()`.
- **JWT/세션**: Access token 1시간(3600초), Refresh Token Rotation 활성화, Reuse Interval 10초. 권한 변경(강등·강퇴) 시 `auth.admin.signOut(userId, 'global')`로 전 기기 세션 즉시 무효화.
- **서버 검증 원칙**: 서버 코드는 반드시 `supabase.auth.getUser()`를 사용한다. `getSession()`은 로컬 캐시를 반환하므로 위변조 토큰을 통과시킬 위험이 있어 서버 검증에 쓰지 않는다.
- **미들웨어**: `src/middleware.ts`가 보호 라우트 접근 전 `getUser()`로 검증하고 refresh token 유효 시 access token을 자동 갱신한다. 미인증 시 `/auth/login`으로 리다이렉트(`/auth`, `/share`, `/_next`는 예외).

## 7. RBAC 및 RLS

### 7.1 역할 정의 (owner / editor / viewer)

프로젝트 멤버 역할을 `owner|editor|viewer` 단일 enum으로 통일한다. 구 체계(`admin/member/observer`)는 폐기한다(`admin → owner`, `member/planner → editor`, `observer → viewer`). `role_level`로 "최소 역할" 비교를 단순화한다(owner=3, editor=2, viewer=1).

```typescript
// src/types/database.ts
export type UserRole = 'owner' | 'editor' | 'viewer';
```

### 7.2 권한 매트릭스

| 리소스 / 액션 | owner | editor | viewer |
|---|:---:|:---:|:---:|
| tasks/links SELECT | O | O | O |
| tasks/links INSERT·UPDATE·DELETE | O | O | X |
| tasks CPM 필드(is_critical/total_float/early_start/late_finish) | RPC만 | RPC만 | X |
| baselines SELECT | O | O | O |
| baselines INSERT/UPDATE | O | O | X |
| baselines/baseline_tasks DELETE | O | X | X |
| resources SELECT | O | O | O |
| costs SELECT | O | O | X |
| project_members 조회/초대 | O | O(조회) | X |
| project_members 역할변경·강퇴, 프로젝트 삭제 | O | X | X |
| shared_links CRUD | O | X | X |

> CPM 필드는 어떤 역할도 직접 쓰기 불가 — `save_cpm_results` RPC만 허용(§9.2).

### 7.3 RLS 표준 패턴

모든 테이블에 RLS를 활성화하고 `is_project_member(project_id, p_min_role)` 헬퍼로 역할 체크를 중앙화한다. 헬퍼는 `project_members` 행의 `role_level >= 요구 레벨`이거나 프로젝트 owner면 TRUE를 반환한다.

표준 4정책 패턴:
- **SELECT** = `is_project_member(project_id, 'viewer')`
- **INSERT/UPDATE/DELETE** = `is_project_member(project_id, 'editor')` (일부 owner 전용: shared_links, project_members 역할변경, baselines/costs DELETE)

```sql
-- 예: tasks 4정책 (links/resources/baselines 동일 패턴)
CREATE POLICY "tasks_select_member" ON tasks FOR SELECT
  USING (is_project_member(project_id, 'viewer'));
CREATE POLICY "tasks_insert_editor" ON tasks FOR INSERT
  WITH CHECK (is_project_member(project_id, 'editor'));
CREATE POLICY "tasks_update_editor" ON tasks FOR UPDATE
  USING (is_project_member(project_id, 'editor'))
  WITH CHECK (is_project_member(project_id, 'editor'));
CREATE POLICY "tasks_delete_editor" ON tasks FOR DELETE
  USING (is_project_member(project_id, 'editor'));
```

**전 테이블 커버리지**: tasks, links, resources, costs, baselines, project_members는 직접 정책. `project_id`를 직접 보유하지 않는 `assignments`(→ tasks 경유), `baseline_tasks`(→ baselines 경유)는 EXISTS 서브쿼리로 간접 참조한다. `costs`는 민감 정보이므로 viewer SELECT 불가(editor 이상).

**pgTAP 요점**(`supabase/tests/`): 역할별 허용/차단(viewer INSERT → 42501), 교차 프로젝트 0건, anon 공유 토큰 유효/만료, CPM 직접 쓰기 차단. 참고 `docs/RLS_TESTING.md`.

## 8. 공유 링크 보안

AC-13(프로젝트 공유 URL), AC-20(섹션 공유) 지원. `shared_links` 테이블 기반.

- **토큰**: 서버에서 `randomBytes(16).toString('base64url')`(128비트, URL-safe). 토큰 공간이 천문학적이라 열거 공격이 비현실적.
- **만료/조회 제한/활성화**: `expires_at`(기본 30일, NULL=무기한), `is_active`(false 시 토큰 삭제 없이 즉시 차단), `max_views`+`view_count`(조회 제한). 만료·비활성·초과 토큰은 RLS가 tasks 행 반환을 차단 → 403.
- **섹션 범위(AC-20)**: `scope='section'`, `scope_task_ids` UUID[]에 포함된 Summary Task만 노출. RLS `tasks.id = ANY(scope_task_ids)` 필터.
- **anon 접근**: `shared_links` 테이블 자체는 anon SELECT 불가(토큰 목록 열거 방지). 비인증 조회는 **Edge Function/API Route가 토큰 검증 후 service_role로 조회**하는 방식을 권장한다(토큰을 RLS에서 직접 검증할 수 없어 약식이 되므로). 토큰 형식 검증 + 조회 실패 응답 시간 균일화(타이밍 어택 방지) + IP당 Rate limiting 병행.
- **AC-17 팀원 초대**: 초대는 owner만. 동일 엔트로피 토큰을 `project_invitations`에 저장(7일 만료), 수락 시 `accept_invitation(token)` RPC가 토큰 검증 → `project_members`에 editor/viewer INSERT를 단일 트랜잭션으로 처리.

## 9. 서버 보안

### 9.1 보안 HTTP 헤더 (`next.config.js`)

모든 응답에 적용:

| 헤더 | 값 | 목적 |
|---|---|---|
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains; preload` | HTTPS 강제(HSTS) |
| `X-Frame-Options` | `DENY` | 클릭재킹 방지 |
| `X-Content-Type-Options` | `nosniff` | MIME 스니핑 방지 |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Referrer 최소화 |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | 불필요 기능 차단 |

CSP는 SVAR Gantt의 인라인 스타일 때문에 즉시 강제하지 않고 Report-Only(위반 수집)로 시작해 Nonce 기반 강제로 단계 전환한다(상세는 03-로드맵 참조).

### 9.2 환경변수 분류 및 Service Role 격리

**클라이언트 노출 허용(`NEXT_PUBLIC_`)**: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`(RLS가 방어선), `NEXT_PUBLIC_APP_URL`, `NEXT_PUBLIC_YJS_WS_URL`.

**서버 전용(절대 클라이언트 노출 금지)**: `SUPABASE_SERVICE_ROLE_KEY`(RLS 우회 슈퍼 권한), `SUPABASE_JWT_SECRET`, `WEBHOOK_SECRET`. Vercel에서 "Expose to browser" 해제 필수, Service Role Key는 Production만.

ESLint로 `NEXT_PUBLIC_*(SERVICE|SECRET|JWT|WEBHOOK)*` 패턴을 차단한다. `SECURITY DEFINER` RPC·service_role 작업은 Edge Function/API Route 서버 측에서만 호출한다.

**CPM 필드 직접 쓰기 방지**: `is_critical`/`total_float`/`early_start`/`late_finish`는 트리거로 비-RPC 컨텍스트의 UPDATE를 거부하고, `save_cpm_results`(`SECURITY DEFINER`)만 `set_config('request.cpm_rpc','true',true)`로 통과한다. CHECK 제약(`total_float >= 0` 등) 병행.

의존성·시크릿 스캔(npm audit + gitleaks + Dependabot, pre-commit hook)은 CI 일부로 운영한다(상세는 03-로드맵 참조).
