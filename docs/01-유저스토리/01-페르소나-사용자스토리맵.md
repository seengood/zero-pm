## 페르소나 A: 김지훈 — 프리랜서/1인 PM

### 기본 프로필

| 항목 | 내용 |
|------|------|
| 나이 | 34세 |
| 직업 | 프리랜서 IT 프로젝트 매니저 |
| 경력 | 대형 SI 업체 5년 → 독립 프리랜서 4년째 |
| 동시 관리 프로젝트 | 평균 3~5개 (스타트업 앱 개발, 중소기업 ERP 도입, 웹사이트 리뉴얼 등) |
| 현재 사용 도구 | Notion (태스크 관리), Google Sheets (간트 수기 작성), 카카오톡 (클라이언트 커뮤니케이션), MS Project (간혹, 클라이언트 요청 시) |
| 근무 환경 | 자택 오피스 + 카페. MacBook Pro 메인. Windows 노트북은 MS Project용으로 가끔 켬 |

### 목표 (Goals)

1. **납기 준수율 100%** — 여러 프로젝트를 동시에 돌리면서도 각 클라이언트와의 약속을 지키는 것이 신뢰와 직결됨
2. **PM 행정 시간 최소화** — 일정 관리에 쓰는 시간을 줄이고, 실제 PM 판단과 커뮤니케이션에 집중하고 싶음
3. **전문가처럼 보이기** — 클라이언트에게 WBS와 간트 차트를 공유할 때 "이 사람 진짜 PM이구나"라는 신뢰감을 주고 싶음

### 고통 (Pain Points)

- **Google Sheets 간트의 한계**: 태스크 날짜를 하나 바꾸면 이후 모든 행을 손으로 밀어야 함. 30개짜리 일정에서 날짜 하나가 바뀌면 30분이 날아감
- **MS Project의 진입 장벽**: 기능은 맞지만 연간 구독료($159+)가 부담스럽고, Windows에서만 돌아가서 Mac에서 쓰려면 가상머신을 켜야 함. 클라이언트에게 파일 공유도 불편함
- **Notion의 가짜 간트**: 칸반/타임라인은 있지만 의존성 자동 전파가 없음. "B 태스크는 A가 끝난 다음에 시작"이라는 관계를 설정해도 A 날짜를 바꾸면 B는 그대로임
- **도구 파편화**: 일정은 Sheets, 태스크는 Notion, 보고는 PowerPoint로 따로 관리. 데이터가 항상 어딘가서 어긋남
- **클라이언트 공유 마찰**: MS Project 파일을 보내면 클라이언트가 "저 이거 못 여는데요"라고 함

### 행동 패턴

- **아침 루틴**: 매일 오전 9시, 커피 한 잔 들고 오늘 마감 태스크와 내일 시작 태스크를 훑음. 전체 일정을 빠르게 조감하고 싶어함
- **프로젝트 킥오프**: 클라이언트 미팅 후 24시간 내에 WBS + 간트 초안을 만들어서 공유하는 것이 자신의 브랜드. 이 단계에서 속도가 중요함
- **주간 리뷰**: 매주 금요일 오후, 각 프로젝트의 진척 현황을 점검하고 다음 주 일정을 재조정함
- **모바일 확인**: 외출 중 클라이언트 문의가 오면 스마트폰으로 빠르게 일정 조회. 수정은 노트북에서 함

### 기술 수준

- SaaS 도구 적응력: 높음 (새 도구 온보딩 거부감 없음)
- 개발 지식: 중간 (Git, API 개념 이해. 코딩은 못함)
- PM 방법론: CPM/WBS 이해함. EVM은 알지만 실무에서 쓰지 않음
- 영어 UI: 불편하지 않음. 단, 한국어가 더 편함

### ZeroPM에 기대하는 것

1. **의존성 자동 재계산** — 날짜 하나 바꾸면 후속 태스크 전체가 자동으로 밀리는 것. 이것만으로도 Sheets를 버릴 수 있음
2. **클라이언트 공유 URL** — "이 링크 클릭하면 최신 일정 볼 수 있어요"라고 말할 수 있는 것. 파일 첨부 없이
3. **빠른 시작** — 로그인 후 5분 안에 첫 간트 차트가 만들어지는 온보딩

### 사용 시나리오

#### A-1. 신규 사용자 온보딩 (Flow 1 — Activity 0)

> **상황**: Google 검색 "무료 간트 도구"로 ZeroPM을 발견. 카드 등록 없이 5분 안에 첫 간트를 만들어보기로 결심.

기존 방식이라면 새 SaaS 도입마다 카드 등록, 관리자 승인, 복잡한 설정 단계를 거쳐야 했다.

ZeroPM에서는: 랜딩 페이지 [무료로 시작하기] 클릭 → 이메일+비밀번호 입력 → 인증 메일 링크 클릭 → 자동 로그인 → 빈 대시보드 → [새 프로젝트 만들기] → 이름·시작일 입력 → 빈 간트 화면 → 온보딩 툴팁 안내로 태스크 2개 추가 → 의존성 연결 안내 툴팁. 총 소요 시간: 3~5분.

##### 수용 기준

> **관련 AC:** AC-09, AC-01, AC-02

**AC-09 — 이메일 인증 후 자동 로그인**  
**Given** 신규 사용자가 이메일 인증 링크를 클릭하면  
**When** 토큰 검증이 완료되면  
**Then** 자동 로그인되어 대시보드로 진입하고, 별도 로그인 절차 없이 서비스를 바로 사용할 수 있다.

**AC-01 — 프로젝트 생성**  
**Given** 대시보드에서 [새 프로젝트 만들기]를 클릭하고 이름·시작일을 입력하면  
**When** [만들기]를 클릭하면  
**Then** 프로젝트가 목록에 즉시 표시되고 빈 간트 화면으로 진입할 수 있다.

**AC-02 — 첫 태스크 추가**  
**Given** 빈 간트 화면에서 [+ 태스크 추가]를 누르거나 빈 행을 클릭하면  
**When** 태스크명과 기간을 입력하고 Enter를 누르면  
**Then** 간트 바가 해당 날짜 범위에 즉시 렌더링된다. 0 또는 음수 기간 입력 시 오류 메시지와 함께 이전 값으로 복원된다.

#### A-2. 프리랜서 PM 일간 작업 — 일정 지연 대응 (Flow 2 — Activity 2·3)

> **상황**: 스타트업 클라이언트의 앱 개발 프로젝트 (12주, 태스크 45개). 3주차에 외주 디자이너가 UI 시안을 1주일 늦게 납품할 것 같다는 연락이 옴.

기존 방식이라면 Google Sheets를 열고, 디자인 완료 날짜를 수동으로 바꾼 뒤, 디자인에 의존하는 프론트엔드 개발 태스크 8개의 시작일을 하나씩 손으로 밀고, 전체 납기가 얼마나 밀리는지를 계산하는 데 20~30분이 소요된다.

ZeroPM에서는: 디자인 완료 태스크의 종료일을 드래그로 1주일 늘리면, FS 의존성으로 연결된 프론트엔드 태스크 8개가 자동으로 전파되어 전체 일정이 즉시 재계산된다. 크리티컬 패스가 어디인지 확인하고, 납기가 5일 밀린다는 것을 파악. 클라이언트에게 공유 링크를 그대로 보내면서 "일정 업데이트됐습니다, 링크에서 확인하세요"라고 카톡을 보낸다. 총 소요 시간 3분.

##### 수용 기준

> **관련 AC:** AC-02, AC-03, AC-05, AC-13

**AC-02 — 간트 바 드래그로 날짜 변경**  
**Given** 태스크가 간트에 표시되어 있고  
**When** 간트 바를 드래그하여 다른 날짜로 이동하면  
**Then** 시작일/종료일이 드래그 결과에 맞게 즉시 갱신되고, 연결된 후행 태스크들도 연쇄 재계산된다.

**AC-03 — FS 링크 전파**  
**Given** 태스크들이 A→B→C FS 의존성으로 연결되어 있고  
**When** 선행 태스크의 기간이 변경되면  
**Then** 모든 후행 태스크 날짜가 chain 전체에 걸쳐 자동 재계산되고, 3초 이내에 완료된다.

**AC-05 — CP 시각적 식별**  
**Given** CP 표시 토글을 활성화하면  
**When** 일정 전파가 완료되면  
**Then** Total Float = 0인 태스크 바가 강조 색으로 표시되어 납기에 직접 영향을 주는 경로를 한눈에 파악할 수 있다.

**AC-13 — 공유 링크 생성 및 비로그인 열람**  
**Given** PM이 공유 링크를 생성하면  
**When** 클라이언트가 해당 URL에 접속하면  
**Then** 로그인 없이 최신 간트 차트를 열람할 수 있고, 편집 인터랙션은 전면 차단된다.

### Quote

> "MS Project 켜려면 Windows 노트북 찾아야 하고, Notion 간트는 날짜가 안 밀리고. 그냥 두 개를 합쳐놓은 거 하나만 있으면 되는데."

## 페르소나 B: 박서연 — IT 소/중규모 팀 PM

### 기본 프로필

| 항목 | 내용 |
|------|------|
| 나이 | 38세 |
| 직업 | 핀테크 스타트업 Product Manager (팀 규모 22명) |
| 경력 | 대기업 IT PM 8년 → 스타트업 PMO 4년째 |
| 담당 범위 | 분기별 로드맵 계획, 스프린트 릴리즈 일정, 외부 파트너 연동 일정 조율 |
| 현재 사용 도구 | Jira (스프린트 관리), Confluence (문서), Monday.com (로드맵), 회사 Google Workspace |
| 근무 환경 | 사무실 + 재택 혼합. 27인치 모니터 + MacBook. 팀원들과 Slack 상시 연결 |

### 목표 (Goals)

1. **릴리즈 일정 가시성 확보** — 개발팀, 디자인팀, QA팀의 일정이 하나의 뷰에 보여서 병목을 즉시 파악하고 싶음
2. **의존성 충돌 사전 감지** — "A 기능이 늦으면 B 기능 QA가 못 시작한다"는 것을 실제 문제가 생기기 전에 알고 싶음
3. **경영진 보고 간소화** — 매주 월요일 임원 보고를 위해 PPT를 새로 만드는 것이 아니라, 최신 일정 뷰를 그대로 보여줄 수 있기를 원함

### 고통 (Pain Points)

- **도구 간 데이터 불일치**: Jira의 스프린트 상태와 Monday.com의 로드맵이 항상 어긋남. 팀원들이 두 곳에 업데이트해야 해서 한쪽은 항상 구버전임
- **Monday.com의 CPM 부재**: 타임라인 뷰는 있지만 의존성을 설정해도 Gantt 수준의 자동 재계산이 없음. 날짜 충돌이 있어도 경고가 없음
- **실시간 협업의 부재**: 팀원 3명이 동시에 일정을 수정하면 누군가의 변경이 덮어씌워짐. 버전 충돌로 인한 혼란이 주 1회 이상 발생
- **외부 파트너 일정 연동 어려움**: 외주 업체, 결제 PG사 등 외부 파트너와 일정을 공유할 때 편집 권한을 세밀하게 제어하지 못함
- **스프린트 ↔ 간트 단절**: 스프린트 단위(2주) 계획과 분기 단위 로드맵이 서로 다른 도구에 있어서 전체 그림을 보려면 머릿속에서 직접 합쳐야 함

### 행동 패턴

- **월요일 스탠드업 전**: 전체 일정 현황을 30초 안에 파악하고 이슈를 미리 파악해서 미팅에 들어감
- **스프린트 플래닝 (격주)**: 다음 스프린트 태스크를 전체 로드맵 맥락에서 잡음. 어느 태스크가 크리티컬 패스에 걸리는지 확인
- **일정 변경 발생 시**: 개발팀에서 "이번 스프린트 이 기능 못 끝낼 것 같아요"라는 말이 나오는 순간, 전체 다운스트림 영향을 즉시 분석해야 함
- **기준선 비교**: 분기 초에 확정한 일정과 현재 실제 진행을 비교해서 임원에게 보고하는 것이 정기 업무

### 기술 수준

- SaaS 도구 적응력: 매우 높음 (Jira, Confluence, Figma, Notion 모두 능숙)
- 개발 지식: 높음 (코드는 못 쓰지만 아키텍처/기술 스택 이해. 개발자와 기술 대화 가능)
- PM 방법론: Agile/Scrum + 전통적 WBS/CPM 혼합 사용. EVM 개념 알고 활용하고 싶음
- 영어 UI: 완전히 편함

### ZeroPM에 기대하는 것

1. **실시간 멀티유저 편집** — 팀원 3명이 동시에 일정을 수정해도 충돌 없이 즉시 반영되는 것
2. **크리티컬 패스 시각화** — 어떤 태스크가 전체 납기에 직접 영향을 미치는지 한눈에 보이는 것
3. **기준선 비교 뷰** — 최초 계획 대비 현재 실제 일정의 슬리피지를 시각적으로 확인하는 것

### 사용 시나리오

#### B-1. 팀 프로젝트 온보딩 — 팀원 초대 및 권한 설정 (Flow 1 + Flow 7 — Activity 0·1)

> **상황**: 3분기 릴리즈 프로젝트를 새로 시작. 개발 4명·QA 1명 팀원을 초대하고 외부 결제 파트너사에는 읽기 전용 접근을 허용해야 함.

기존 방식이라면 Monday.com 프로젝트 생성 → Jira 스프린트 보드 별도 생성 → 팀원별 권한 설정을 각 도구에서 반복해야 했다.

ZeroPM에서는: 프로젝트 생성 → 멤버 관리 패널에서 이메일+역할(Editor/Viewer) 입력 → 초대 발송 → 팀원이 링크 클릭 후 자동 진입. 외부 파트너는 공유 링크(읽기 전용)로 로그인 없이 열람 가능. 총 소요 시간: 10분.

##### 수용 기준

> **관련 AC:** AC-01, AC-17, AC-13

**AC-01 — 프로젝트 생성**  
**Given** PM이 프로젝트 이름과 시작일을 입력하면  
**When** [만들기]를 클릭하면  
**Then** 빈 간트 화면으로 즉시 진입하고, 대시보드 목록에도 새 프로젝트가 반영된다.

**AC-17 — 팀원 초대 및 역할 적용**  
**Given** Owner 권한 사용자가 이메일과 역할(Editor)을 입력하여 초대하면  
**When** 수신자가 초대 링크를 클릭하면  
**Then** 해당 역할로 프로젝트에 자동 진입하고, Viewer는 편집 인터랙션이 차단되며 RLS 수준에서도 쓰기가 거부된다.

**AC-13 — 공유 링크 생성 및 비로그인 열람**  
**Given** PM이 외부 파트너용 공유 링크를 생성하면  
**When** 외부 파트너가 해당 URL에 접속하면  
**Then** 로그인 없이 최신 간트 차트를 열람할 수 있고, 편집 인터랙션은 전면 차단된다.

#### B-2. 팀 PM 협업 — 일정 지연 대응 및 경영진 보고 (Flow 3 + Flow 8 — Activity 2·3·5)

> **상황**: 3분기 메인 릴리즈 (9월 30일 마감). 8월 중순, 결제 모듈 백엔드 개발이 2주 지연될 것 같다는 개발팀 보고.

기존 방식이라면 Monday.com에서 해당 태스크 날짜를 수동으로 바꾸고, 연결된 QA/프론트엔드 태스크를 하나씩 찾아서 확인하고, Jira에서 스프린트 일정을 별도로 조정하고, 전체 영향 범위를 텍스트로 정리해서 Slack에 공지해야 한다. 이 과정이 1~2시간 소요.

ZeroPM에서는: 백엔드 개발 태스크 종료일을 2주 연장하면 FS/SS 의존성을 따라 QA, 프론트엔드 연동, 최종 검수 태스크가 자동 전파된다. 크리티컬 패스 하이라이트를 켜서 9월 30일 마감이 깨지는지 즉시 확인. 마감이 10일 밀리는 것을 확인하고, 병렬 처리 가능한 태스크를 찾아 일정을 압축하는 방안을 팀원들과 공유 URL로 실시간 협의한다. 기준선 뷰로 최초 계획 대비 슬리피지를 캡처해서 임원 보고 자료로 활용. 총 소요 시간 15분.

##### 수용 기준

> **관련 AC:** AC-03, AC-05, AC-06, AC-07, AC-08

**AC-03 — 의존성 자동 전파**  
**Given** FS/SS 의존성이 설정된 chain에서  
**When** 선행 태스크 기간이 변경되면  
**Then** 모든 후행 태스크가 자동으로 재계산되고, 3초 이내에 완료된다.

**AC-05 — CP 동적 갱신**  
**Given** CP 표시가 활성화된 상태에서  
**When** 일정 변경으로 CP가 바뀌면  
**Then** 간트 바 색상이 즉시 갱신되어 현재 납기 위험 경로를 실시간으로 파악할 수 있다.

**AC-06 — 기준선 비교 시각화**  
**Given** 기준선이 저장된 상태에서 일부 일정이 변경된 경우  
**When** 기준선 표시 토글을 켜면  
**Then** 간트 바 아래에 기준선 바가 반투명으로 표시되어 슬리피지를 시각적으로 확인할 수 있다.

**AC-07 — 멀티유저 실시간 동기화**  
**Given** PM과 팀원이 동시에 같은 프로젝트에 접속해 있고  
**When** 어느 한쪽이 태스크를 수정하면  
**Then** 5초 이내에 다른 사람 화면에 반영되고, 변경 내용이 서로 덮어쓰지 않는다.

**AC-08 — 낙관적 잠금 (버전 충돌 처리)**  
**Given** 두 사용자가 동시에 같은 태스크를 수정할 때  
**When** 나중에 저장하는 사용자의 요청이 도달하면  
**Then** "다른 사용자가 이미 수정했습니다" 안내가 표시되고, 수정 내용은 유실되지 않으며 병합 여부를 직접 결정할 수 있다.
