# ZeroPM 핵심 User Flow

---

## Flow 1: 신규 사용자 첫 경험 (온보딩)

**트리거:** 사용자가 zeroPM.io에 처음 접속  
**페르소나:** 프리랜서 PM(A), 팀 PM(B) 모두  
**목표:** 첫 프로젝트를 만들고 간트 차트를 확인하기까지 5분 이내

---

### 1-1. 랜딩 페이지

```
[랜딩 페이지]
  - 헤드라인: "MS Project 수준의 일정 관리, 무료로"
  - 간트 차트 데모 GIF 또는 인터랙티브 데모 (읽기 전용)
  - CTA 버튼: [무료로 시작하기] / [로그인]
  - 하단: "신용카드 불필요 · 영구 무료"
```

| 단계 | 사용자 행동 | 시스템 반응 | 성공 조건 | 에러 상태 |
|------|------------|------------|----------|----------|
| 진입 | URL 접속 또는 검색 클릭 | 랜딩 페이지 로드 (< 2초) | 헤드라인과 CTA 표시 | 502: "잠시 후 다시 시도해 주세요" 배너 |
| CTA 클릭 | [무료로 시작하기] 클릭 | 회원가입 화면으로 전환 | 회원가입 폼 표시 | — |

---

### 1-2. 회원가입

```
[회원가입 화면]
  - 이메일 + 비밀번호 폼
  - Google / GitHub OAuth 버튼
  - 이미 계정이 있다면? [로그인]
```

| 단계 | 사용자 행동 | 시스템 반응 | 성공 조건 | 에러 상태 |
|------|------------|------------|----------|----------|
| 이메일 입력 | 이메일 타이핑 | 실시간 형식 검증 (blur 이벤트) | 유효한 이메일 형식 | "올바른 이메일 형식을 입력해 주세요" (인라인) |
| 비밀번호 입력 | 8자 이상 입력 | 강도 표시 바 (weak/fair/strong) | 조건 충족 시 녹색 체크 | "8자 이상, 영문+숫자 조합" (인라인) |
| 가입 제출 | [계정 만들기] 클릭 | 스피너 → 이메일 인증 안내 화면 | 인증 메일 발송 확인 메시지 | "이미 사용 중인 이메일" / 네트워크 오류 토스트 |
| OAuth | [Google로 계속하기] 클릭 | 팝업 → OAuth 완료 → 대시보드 진입 | 대시보드 랜딩 | 팝업 차단: "팝업을 허용해 주세요" 안내 |

**이메일 인증 후 경로:**
```
인증 메일 클릭
  → /auth/confirm 리다이렉트
  → 자동 로그인
  → 대시보드 (프로젝트 목록, 비어 있음)
```

---

### 1-3. 첫 프로젝트 생성

```
[대시보드 — 빈 상태]
  - 일러스트 + 문구: "아직 프로젝트가 없습니다"
  - CTA: [새 프로젝트 만들기]
  - 보조 텍스트: "샘플 프로젝트로 먼저 둘러보기" (링크)
```

| 단계 | 사용자 행동 | 시스템 반응 | 성공 조건 | 에러 상태 |
|------|------------|------------|----------|----------|
| 생성 모달 열기 | [새 프로젝트 만들기] 클릭 | 슬라이드업 모달: 프로젝트명, 시작일, 설명(선택) | 모달 표시 | — |
| 프로젝트명 입력 | 이름 타이핑 | 실시간 글자 수 표시 (최대 100자) | 1자 이상 입력 | "프로젝트 이름을 입력해 주세요" |
| 시작일 선택 | 달력 팝오버에서 날짜 선택 | 오늘 날짜 기본 선택 | 날짜 선택 완료 | — |
| 생성 완료 | [만들기] 클릭 | 스피너 → 간트 화면으로 이동 | 빈 간트 화면 진입 | DB 오류: "프로젝트를 만들지 못했습니다. 다시 시도해 주세요" 토스트 |

---

### 1-4. 첫 태스크 입력

```
[간트 화면 — 빈 상태]
  - 왼쪽 WBS 그리드: 행 없음
  - 오른쪽 차트 영역: 빈 타임라인
  - 상단 툴바: [+ 태스크 추가] 버튼 강조 (펄스 애니메이션)
  - 온보딩 툴팁: "여기서 태스크를 추가해 보세요"
```

| 단계 | 사용자 행동 | 시스템 반응 | 성공 조건 | 에러 상태 |
|------|------------|------------|----------|----------|
| 태스크 추가 | [+ 태스크 추가] 클릭 또는 빈 행 더블클릭 | 인라인 편집 모드: WBS 첫 행에 텍스트 입력 커서 | 커서 표시 | — |
| 태스크명 입력 | 이름 타이핑 후 Tab | 다음 셀(시작일)로 커서 이동 | — | — |
| 기간 입력 | 일수 입력 (예: "5") 후 Enter | 태스크 바가 간트에 즉시 렌더링 | 차트에 바 표시 | 음수 또는 0: "기간은 1일 이상이어야 합니다" |
| 추가 태스크 | Enter 또는 [+ 태스크 추가] 반복 | 다음 행 자동 생성 | 여러 태스크 표시 | — |

**Happy Path 완료 상태:**
```
태스크 2개 이상 입력
  → 간트 차트에 바 2개 이상 표시
  → 온보딩 툴팁 2단계: "태스크를 연결해 의존성을 만들어 보세요"
  → 링크 드래그 안내 오버레이 (한 번만 표시, 이후 영구 비표시)
```

## Flow 2: 프리랜서 PM의 일간 작업 흐름

**트리거:** 클라이언트로부터 "디자인 리뷰를 3일 늦춰야 할 것 같아요" 메시지 수신  
**페르소나:** 페르소나 A — 혼자 여러 프로젝트 관리, 빠른 업데이트가 핵심  
**목표:** 태스크 하나 수정 → 전체 일정 자동 재계산 → 클라이언트에게 업데이트 공유

---

### 2-1. 프로젝트 진입 및 태스크 탐색

```
[대시보드]
  프로젝트 카드 목록
  → "A사 앱 개발" 카드 클릭
  → 간트 화면 진입
```

```
[간트 화면]
  WBS 그리드 | 타임라인
  ─────────────────────────────────────
  1. 기획         ████ (완료)
  2. 디자인 리뷰  ████ ← 이것을 수정
  3. 개발 착수      ░░░░ (디자인 리뷰에 FS 의존)
  4. QA                   ░░░░
  5. 배포                     ░
```

---

### 2-2. 태스크 날짜 수정

| 단계 | 사용자 행동 | 시스템 반응 | 성공 조건 | 에러 상태 |
|------|------------|------------|----------|----------|
| 태스크 선택 | "디자인 리뷰" 바를 클릭 | 우측 디테일 패널 열림 (또는 인라인 편집 모드) | 패널에 현재 시작일/종료일 표시 | — |
| 날짜 수정 | 시작일 필드 클릭 → 달력에서 +3일 선택 | 필드 값 즉시 반영, 바 이동 애니메이션 | 간트 바가 새 위치로 이동 | 제약 조건 위반: 인라인 경고 아이콘 + "SNET 제약으로 이 날짜 이전으로 설정할 수 없습니다" |
| 또는: 드래그 | 간트 바를 오른쪽으로 드래그 | 드래그 중 날짜 툴팁 실시간 표시 ("6/15 → 6/18") | 드롭 시 새 날짜 확정 | — |
| 저장 | 자동 저장 (디바운스 500ms) | 상단바에 "저장 중..." → "저장됨" (체크 아이콘) | "저장됨" 표시 | "저장 실패 — 재시도" 버튼 포함 토스트 |

---

### 2-3. 의존성 자동 전파 확인

```
[간트 화면 — 저장 직후]

  시스템이 자동으로 후행 태스크를 재계산:

  1. 기획         ████ (변경 없음)
  2. 디자인 리뷰  ████→→→ (3일 뒤로)     ← 수정됨 (파란 하이라이트 1.5초)
  3. 개발 착수         ░░░░→→→            ← 자동 전파 (노란 하이라이트 1.5초)
  4. QA                    ░░░░→→→        ← 자동 전파 (노란 하이라이트 1.5초)
  5. 배포                      ░→→→       ← 자동 전파 (노란 하이라이트 1.5초)

  상단 알림 배너 (3초 후 자동 소멸):
  "3개 후행 태스크가 자동으로 조정되었습니다."
```

**Critical Path 변화 시:**
```
  크리티컬 패스 태스크 → 빨간 바 테두리로 표시
  전체 프로젝트 종료일이 늘어난 경우:
  → 상단 요약 칩: "프로젝트 종료일 6/30 → 7/3" (빨간 텍스트)
```

---

### 2-4. 결과 공유

```
[공유 옵션 — 우측 상단 [공유] 버튼]
  ① 링크 복사: 읽기 전용 간트 URL 생성 → 클립보드 복사
  ② 스냅샷 내보내기: PNG / PDF (미구현 — 회색 처리 + "준비 중" 배지)
```

| 단계 | 사용자 행동 | 시스템 반응 | 성공 조건 |
|------|------------|------------|----------|
| 공유 클릭 | [공유] 버튼 클릭 | 드롭다운 메뉴 표시 | — |
| 링크 복사 | [읽기 전용 링크 복사] 선택 | "링크가 복사되었습니다" 토스트 | 클립보드에 URL 저장 |
| 클라이언트 확인 | 클라이언트가 링크 접속 | 로그인 없이 간트 열람 가능 (공개 읽기) | 클라이언트가 업데이트된 일정 확인 |

## Flow 3: 팀 PM의 협업 흐름

**트리거:** 팀원 김개발로부터 Slack 메시지: "개발 착수 태스크 완료했습니다"  
**페르소나:** 페르소나 B — 5명 팀, 실시간 상태 파악과 병목 예방이 핵심  
**목표:** 진행률 업데이트 → Critical Path 재확인 → 필요 시 태스크 재배정

---

### 3-1. 실시간 상태 확인

```
[간트 화면 — PM이 이미 열어 두었음]

  팀원 프레즌스 표시 (우측 상단):
  [김개발 아바타] [이디자인 아바타] [박QA 아바타] — 현재 접속 중

  태스크 "개발 착수"에 실시간 편집 중 인디케이터:
  → 바 왼쪽에 김개발 아바타 아이콘 표시
```

---

### 3-2. 진행률 업데이트

| 단계 | 사용자 행동 | 시스템 반응 | 성공 조건 | 에러 상태 |
|------|------------|------------|----------|----------|
| 태스크 클릭 | "개발 착수" 행 클릭 | 우측 디테일 패널 오픈 | 패널에 진행률 슬라이더 표시 | — |
| 진행률 조정 | 슬라이더를 100%로 드래그 | 슬라이더 값 실시간 반영, 간트 바 내부 채움 표시 | 바 내부가 100% 채워짐 | — |
| 완료 처리 | [완료로 표시] 체크박스 클릭 | 태스크 행에 완료 아이콘, 바 색상 회색(완료) 전환 | 완료 상태 저장 | 저장 실패: 토스트 + 재시도 버튼 |
| 팀원 화면 | — | 김개발/이디자인의 화면에도 실시간 반영 (Supabase Realtime) | 타 팀원 화면 동기화 | 동기화 지연: 팀원 화면에 "동기화 중..." 스피너 |

---

### 3-3. Critical Path 변화 확인

```
[간트 화면 — 완료 처리 후]

  변화 전:
  개발 착수 ████████ (크리티컬, 빨간 테두리)
  QA        ░░░░░░░░ (크리티컬)

  변화 후:
  개발 착수 ████████ ✓ (완료, 회색)
  QA        ░░░░░░░░ (이제 크리티컬 패스에서 여유 발생? 재계산)

  상단 요약 패널 업데이트:
  전체 여유시간(Total Float): +2일 ↑
  크리티컬 태스크 수: 5 → 3
```

**Float 시각화:**
```
  각 태스크 바 우측에 점선 바로 여유시간 표시 (선택적 토글 ON/OFF)
  예: QA ░░░░░░░░ ···· (2일 여유)
```

---

### 3-4. 태스크 재배정 (미구현 기능 처리)

```
[디테일 패널 — 담당자 필드]
  현재 상태: 자원 관리 미구현
  → 담당자 필드: "준비 중" 배지 + 회색 처리
  → 툴팁: "자원 배정 기능은 다음 릴리즈에서 지원됩니다"

  현재 가능한 대안:
  → 태스크 이름에 담당자 표기 (예: "[김개발] QA 1차")
  → 댓글/메모 필드에 배정 내용 기입
```

## Flow 4: 충돌 해결 UX 흐름

**트리거:** PM A와 팀원 B가 동시에 같은 태스크("QA")를 수정  
**현재 구현:** 낙관적 잠금 (version 컬럼)  
**목표:** 비파괴적으로 충돌을 해결하고 데이터 손실 없이 최종 상태 확정

---

### 4-1. 충돌 발생 시점

```
[타임라인]
  T=0: PM A가 "QA" 태스크 디테일 패널 오픈 (version=3)
  T=5: 팀원 B도 "QA" 태스크 수정 시작 (version=3)
  T=10: 팀원 B가 기간을 5→7일로 수정, 저장 성공 → DB: version=4
  T=15: PM A가 시작일을 수정, 저장 시도 → DB 응답: 409 Conflict (version 불일치)
```

---

### 4-2. 충돌 감지 화면

```
[PM A 화면 — 모달 오버레이]

  ┌─────────────────────────────────────────┐
  │  ⚠️  다른 사용자가 이 태스크를 수정했습니다  │
  │                                         │
  │  태스크: QA                             │
  │  수정자: 팀원 B · 방금 전               │
  │                                         │
  │  변경 내용 비교:                         │
  │  ┌──────────┬──────────┬──────────┐     │
  │  │ 항목     │ 상대방   │ 내 변경  │     │
  │  ├──────────┼──────────┼──────────┤     │
  │  │ 기간     │ 7일 ★   │ 5일      │     │
  │  │ 시작일   │ 6/20    │ 6/22 ★  │     │
  │  └──────────┴──────────┴──────────┘     │
  │  (★ = 변경된 항목)                      │
  │                                         │
  │  [상대방 변경 유지]  [내 변경 적용]      │
  │  [항목별 선택...]   [나중에 결정]        │
  └─────────────────────────────────────────┘
```

---

### 4-3. 해결 선택지별 UX

**선택지 1: 상대방 변경 유지 (내 변경 포기)**
```
  클릭 → 모달 닫힘
  → 내 패널이 최신 값(팀원 B의 수정)으로 업데이트
  → 토스트: "상대방 변경 사항이 적용되었습니다"
  → 내 수정 내용은 소멸 (되돌릴 수 없음 — 단, Undo로는 가능하도록 설계 권장)
```

**선택지 2: 내 변경 적용 (상대방 변경 덮어쓰기)**
```
  클릭 → 확인 다이얼로그:
  "팀원 B의 변경이 덮어써집니다. 계속하시겠습니까?"
  [확인] → 내 version으로 강제 저장 → 토스트: "내 변경이 저장되었습니다"
  → 팀원 B 화면에 "이 태스크가 다른 사용자에 의해 수정되었습니다" 배너 표시
```

**선택지 3: 항목별 선택 (병합)**
```
  클릭 → 확장된 모달:
  각 변경 항목마다 라디오 버튼
  ○ 기간: (●) 7일 (팀원 B) / ○ 5일 (내 것)
  ○ 시작일: ○ 6/20 (팀원 B) / (●) 6/22 (내 것)
  [이 조합으로 저장] → 병합된 값으로 저장
```

**선택지 4: 나중에 결정 (보류)**
```
  클릭 → 모달 닫힘
  → 태스크 행에 경고 아이콘 지속 표시 (충돌 미해결)
  → 툴바에 "미해결 충돌 1건" 배지
  → 해당 태스크는 편집 잠금 (충돌 해결 전까지)
```

## Flow 5: Undo/Redo UX 흐름

**트리거:** 사용자가 실수로 태스크를 삭제하거나 잘못된 날짜로 이동  
**목표:** 명확한 범위와 그룹화로 실수를 빠르게 복구

---

### 5-1. Undo 가능한 행동 정의

```
Undo 대상 (세션 내 영구 스택, 최대 50개):
  ✓ 태스크 생성 / 삭제
  ✓ 태스크 날짜 이동 (드래그)
  ✓ 태스크 기간 변경 (리사이즈)
  ✓ 태스크명 / 필드 수정
  ✓ 링크(의존성) 생성 / 삭제
  ✓ 진행률 변경
  ✓ 제약 조건 변경
  ✓ 자동 전파된 후행 태스크 변경 (그룹 Undo)

Undo 불가 (서버 사이드 액션):
  ✗ 프로젝트 삭제
  ✗ 팀원 초대 / 권한 변경
  ✗ 기준선(Baseline) 저장
```

---

### 5-2. 단일 Undo vs 그룹 Undo

```
[시나리오: "디자인 리뷰" 시작일 변경]

  행동 로그 (내부 스택):
  ┌─────────────────────────────────────────┐
  │ 그룹 #5 [펼치기]                        │
  │   디자인 리뷰 시작일 6/15 → 6/18       │
  │   ↳ 개발 착수 시작일 6/20 → 6/23 (전파)│
  │   ↳ QA 시작일 6/27 → 6/30 (전파)      │
  │   ↳ 배포 시작일 7/4 → 7/7 (전파)      │
  ├─────────────────────────────────────────┤
  │ 그룹 #4  링크 생성: 기획 → 디자인 리뷰  │
  │ 그룹 #3  태스크 "배포" 생성            │
  │ ...                                     │
  └─────────────────────────────────────────┘

  기본 Undo: Ctrl+Z → 그룹 #5 전체 되돌림 (전파된 변경 포함)
  개별 Undo: 그룹 [펼치기] → 개별 행 되돌림 가능 (고급 옵션)
```

---

### 5-3. UI 위치와 키보드 단축키

```
[상단 툴바]
  ← Undo    Redo →    (버튼, 비활성 시 회색)
  Ctrl+Z / Cmd+Z    Ctrl+Y / Cmd+Shift+Z

[Undo 버튼 호버 툴팁]:
  "디자인 리뷰 날짜 변경 및 3개 후행 태스크 되돌리기 (그룹)"

[Undo 실행 후]:
  → 해당 태스크들이 이전 위치로 이동 애니메이션 (0.3초)
  → 토스트: "4개 변경이 취소되었습니다" [다시 실행] 버튼 포함
```

---

### 5-4. Undo 스택 범위 정책

```
  세션 내: 탭을 닫거나 새로고침 시 스택 초기화
  → 사용자에게 명시: "변경 내역은 현재 세션에서만 유지됩니다"

  영구 이력(History)은 미래 기능으로 구분:
  → 좌측 사이드바 [변경 이력 보기] — "준비 중" 배지로 표시
```

## Flow 6: 오프라인 → 온라인 복귀 UX

**트리거:** 사용자가 간트 편집 중 네트워크 연결 끊김  
**목표:** 오프라인 상태를 명확히 알리고, 온라인 복귀 시 데이터 손실 없이 동기화

---

### 6-1. 오프라인 진입 감지

```
[감지 즉시 — 상단 상태 바 변화]

  정상: ● 연결됨               (녹색 점)
  오프라인: ⚠ 오프라인 모드   (노란 배너, 전체 너비)

  배너 문구:
  "인터넷 연결이 끊겼습니다. 변경 사항은 로컬에 임시 저장됩니다."
```

---

### 6-2. 오프라인 중 허용 / 제한 행동

```
  허용 (로컬 대기열에 누적):
  ✓ 태스크 날짜 이동 / 리사이즈
  ✓ 태스크명 편집
  ✓ 진행률 변경
  ✓ 의존성 자동 전파 계산 (로컬에서 수행)
  ✓ Undo / Redo (로컬 스택 유지)

  제한 (시도 시 인라인 안내):
  ✗ 새 프로젝트 생성 → "온라인 상태에서만 가능합니다"
  ✗ 팀원 초대 → 동일 안내
  ✗ 기준선(Baseline) 저장 → 동일 안내
  ✗ 실시간 타 사용자 프레즌스 → 마지막 접속자 표시 유지 (페이드 처리)
```

---

### 6-3. 온라인 복귀 — 자동 동기화 시나리오

```
[네트워크 복귀 감지]

  배너 전환:
  "인터넷에 다시 연결되었습니다. 동기화 중..." (스피너)
  → "동기화 완료. 3개 변경이 저장되었습니다." (녹색, 3초 후 소멸)

  동기화 흐름:
  로컬 대기열 → Supabase에 순서대로 전송
  각 항목에 버전 체크 → 충돌 없으면 자동 저장 완료
```

---

### 6-4. 온라인 복귀 — 충돌 발생 시나리오

```
[충돌 감지 시 — 모달]

  ┌──────────────────────────────────────────┐
  │  동기화 중 충돌이 발견되었습니다          │
  │                                          │
  │  오프라인 중 수정한 2개 태스크가          │
  │  다른 사용자에 의해 변경되었습니다.       │
  │                                          │
  │  • QA (팀원 B가 기간 변경)               │
  │  • 배포 (팀원 C가 시작일 변경)           │
  │                                          │
  │  [항목별 검토하기]  [내 변경 모두 적용]  │
  │  [상대방 변경 모두 유지]                  │
  └──────────────────────────────────────────┘

  [항목별 검토하기] 선택 시:
  → Flow 4의 충돌 해결 UX와 동일한 비교 모달로 진입
  → 충돌 항목을 하나씩 순서대로 처리
  → 모든 충돌 해결 후: "동기화 완료" 토스트
```

### P2. 로딩 상태

```
[케이스별 로딩 표현]

  1. 간트 초기 로딩 (DB에서 태스크 fetch):
     → 스켈레톤 UI: WBS 행과 타임라인 바가 회색 박스로 채워짐
     → 500ms 이상 시 스켈레톤 표시 (깜빡임 방지)

     → 상단 상태 칩: "저장 중..." (스피너 아이콘)

  3. 스케줄링 재계산 중 (후행 태스크 전파):
     → 전파되는 태스크 바에 펄스 애니메이션 (0.5초)
     → 계산 완료 후 즉시 최종 위치로 이동

  4. 기준선 저장 / 대용량 연산:
     → 전체 화면 반투명 오버레이 + 중앙 스피너
     → 문구: "기준선을 저장하는 중입니다..."
```

### P3. 에러 상태

```
[에러 유형별 처리]

  1. DB 저장 실패 (네트워크 일시 오류):
     → 토스트 (우측 하단, 빨간 배경):
       "저장하지 못했습니다." [재시도] [닫기]
     → 재시도 클릭 시 즉시 재전송, 성공 시 "저장됨"으로 전환

  2. 권한 없음 (RLS 정책 위반):
     → 모달:
       "이 프로젝트를 수정할 권한이 없습니다."
       [대시보드로 돌아가기]
     → 수정 시도한 내용은 로컬에서도 롤백

  3. 제약 조건 충돌 (예: MSO로 고정된 태스크를 그 이전으로 이동):
     → 인라인 경고 (태스크 행 강조, 노란 테두리):
       "이 태스크는 [날짜] 이전으로 이동할 수 없습니다 (MSO 제약)."
     → 드래그 중이라면 드롭 취소, 바가 원래 위치로 복귀 (스냅백)

  4. 세션 만료:
     → 저장 시도 시 401 응답 감지
     → 모달: "세션이 만료되었습니다. 다시 로그인해 주세요."
       [로그인 페이지로] — 클릭 시 현재 URL을 redirect 파라미터로 유지
```

### P4. 성공 피드백

```
[피드백 위계]

  중요도 낮음 (자동 소멸, 2초):

  중요도 중간 (자동 소멸, 3초, 우측 하단 토스트):

  → 에러: 빨간색 (#dc2626) + X 아이콘
  → 정보: 파란색 (#2563eb) + i 아이콘
```

## Flow 7: 팀원 초대 및 권한 설정

### 개요

| 항목 | 내용 |
|------|------|
| **트리거** | 팀 PM이 프로젝트에 팀원을 추가하려 함 |
| **페르소나** | 페르소나 B (팀 PM, 주), 페르소나 A (클라이언트에게 Viewer 권한 부여, 부) |
| **전제 조건** | 사용자가 로그인 상태이며 해당 프로젝트의 Owner 또는 Editor 권한 보유 |
| **목표** | 팀원 초대 → 수락 → 역할별 권한 적용까지 완전한 흐름 |
| **관련 AC** | AC-17 (팀원 초대 및 권한 설정), AC-09 (인증 및 접근 제어) |

---

### 7-1: 초대 발송

#### Happy Path

```
[PM — 프로젝트 화면]
        │
        ▼
  상단 툴바 "멤버 관리" 아이콘 클릭
  (사람 + 플러스 아이콘, 현재 멤버 아바타 옆)
        │
        ▼
  ┌─────────────────────────────────────┐
  │  멤버 관리 패널 (슬라이드 오버)        │
  │                                     │
  │  [현재 멤버 목록]                    │
  │  ● 박서연 (나)          Owner  [--] │
  │  ● 이민준              Editor  [✕] │
  │                                     │
  │  ┌───────────────────────────────┐  │
  │  │ 이메일 주소 입력...            │  │
  │  └───────────────────────────────┘  │
  │                                     │
  │  역할 선택:                          │
  │  ○ Viewer   ● Editor   ○ Owner     │
  │                                     │
  │  역할 설명 (인라인 툴팁):            │
  │  Editor — 태스크 생성·편집 가능     │
  │                                     │
  │           [초대 보내기]              │
  └─────────────────────────────────────┘
        │
        ▼
  이메일 입력 + 역할 선택 후 "초대 보내기" 클릭
        │
        ▼
  DB: project_members 에 status='pending' 행 삽입
  이메일 발송: 초대 링크 포함 (토큰 기반, 48시간 유효)
        │
        ▼
  ┌──────────────────────────┐
  │  ✔ 초대 이메일을 보냈어요  │
  │  kim@example.com (Viewer) │
  └──────────────────────────┘
  (토스트 알림 3초 후 자동 닫힘)
        │
        ▼
  멤버 목록에 "대기 중" 배지와 함께 표시
  ● kim@example.com   Viewer   [대기 중] [취소]
```

#### 역할 정의표

| 역할 | 설명 | 초대 가능 여부 |
|------|------|----------------|
| **Viewer** | 읽기 전용. 간트·태스크 조회만 가능 | Owner만 가능 |
| **Editor** | 태스크·링크 생성·편집·삭제 가능 | Owner만 가능 |
| **Owner** | 멤버 관리 + 프로젝트 삭제 포함 전체 권한 | Owner만 가능 |

> Editor가 멤버 관리 패널을 열면 초대 폼 없이 현재 멤버 목록만 조회 가능 (읽기 전용).

#### Error Cases

| 케이스 | 발생 조건 | 시스템 반응 |
|--------|-----------|-------------|
| **E7-1-A** 이미 멤버인 이메일 | 입력한 이메일이 project_members에 active 상태로 존재 | "이미 이 프로젝트의 멤버예요." 인라인 에러. 초대 버튼 비활성화 |
| **E7-1-B** 대기 중 초대가 이미 존재 | 동일 이메일로 pending 초대가 남아 있음 | "이미 초대장을 보낸 이메일이에요. 재발송하시겠어요?" 확인 다이얼로그 → 재발송 or 취소 |

### 개요

| 항목 | 내용 |

### 8-3: 슬리피지 분석

#### Happy Path

```
[PM — 기준선 비교 뷰 활성화 상태]
        │
        ▼
  지연된 태스크 행 클릭
        │

  태스크 상세 패널 (우측 슬라이드 오버):
  ┌──────────────────────────────────────┐
  │  Task A — 기획 완료                   │
  │                                      │
  │  ┌─────────────────────────────────┐ │
  │  │  📊 기준선 비교   기준선 1       │ │
  │  │  ─────────────────────────────  │ │
  │  │  구분       기준선      현재     │ │
  │  │  시작일  06-01 (월)  06-01 (월) │ │
  │  │  종료일  06-07 (일)  06-10 (수) │ │
  │  │  기간       7일         10일    │ │
  │  │                                 │ │
  │  │  ⚠ 종료일 +3일 지연             │ │  ← 슬리피지 강조
  │  └─────────────────────────────────┘ │
  └──────────────────────────────────────┘
        │
        ▼
  간트 상단 요약 배너 (기준선 비교 중일 때만 표시):
  ┌────────────────────────────────────────────────────┐
  │  기준선 1 대비 — 지연 태스크: 3개  최대 슬리피지: +5일  │
  └────────────────────────────────────────────────────┘
```

#### 슬리피지 계산 규칙

| 상태 | 조건 | 표시 |
|------|------|------|
| **정시** | 현재 종료일 = 기준선 종료일 | 기준선 바와 현재 바 겹침. 강조 없음 |
| **조기 완료** | 현재 종료일 < 기준선 종료일 | 현재 바를 초록색으로 표시. 상세 패널에 "−N일 단축" |
| **지연** | 현재 종료일 > 기준선 종료일 | 현재 바를 주황색으로 표시. 상세 패널에 "+N일 지연" |
| **삭제된 태스크** | 현재 태스크 없음, 기준선에 존재 | 기준선 바만 점선으로 렌더링. 행 배경 연회색. 툴팁: "현재 일정에서 삭제된 태스크" |
| **신규 태스크** | 기준선에 없음, 현재에 존재 | 기준선 바 미표시. 태스크 행 좌측에 "NEW" 뱃지 |

### 8-4: 여러 기준선 간 전환

## Flow 11: 프로젝트 대시보드 / 멀티프로젝트 조감

### 개요

| 항목 | 내용 |
|------|------|
| **트리거** | PM이 관리 중인 여러 프로젝트의 전체 현황을 한눈에 파악하려 함 |
| **페르소나** | 페르소나 A (프리랜서 PM, 주 — 여러 클라이언트 프로젝트 조감), 페르소나 B (팀 PM, 부) |
| **전제 조건** | 사용자가 1개 이상의 프로젝트에 접근 권한 보유 |
| **목표** | 전체 프로젝트 상태 확인 → 지연 프로젝트 식별 → 개별 프로젝트 진입 |
| **관련 AC** | AC-15 (프로젝트 대시보드 요약) |
