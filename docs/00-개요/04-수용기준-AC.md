# ZeroPM 핵심 기능 수용 기준 (Acceptance Criteria)

---

## AC-01: 프로젝트 생성

**관련 페르소나:** 프리랜서 PM (1순위), IT 팀 PM (2순위)

### Happy Path

**Given** 인증된 사용자가 대시보드에 접속해 있고  
**When** "새 프로젝트" 버튼을 클릭하고 프로젝트명, 시작일을 입력한 뒤 저장하면  
**Then** 프로젝트가 목록에 즉시 표시되고, 해당 프로젝트의 빈 간트 차트 화면으로 진입할 수 있다.

### Edge Case 1 — 프로젝트명 중복

**Given** 동일 계정에 "웹사이트 리뉴얼"이라는 프로젝트가 이미 존재하고  
**When** 동일한 이름으로 새 프로젝트를 생성하려 하면  
**Then** 중복 이름을 허용하되 프로젝트 ID로 구분되므로 저장은 성공하고, 사용자는 이름 변경을 권장하는 안내 문구를 볼 수 있다.

### Error Case — 필수 입력 누락

**Given** 사용자가 프로젝트 생성 폼을 열고  
**When** 프로젝트명을 비워둔 채 저장하면  
**Then** "프로젝트명은 필수 항목입니다" 메시지가 인라인으로 표시되고, 저장은 실행되지 않는다.

## AC-02: 태스크 생성 및 기본 편집

**관련 페르소나:** 프리랜서 PM (1순위)

### Happy Path — 태스크 생성

**Given** 프로젝트 간트 화면이 열려 있고  
**When** 빈 행을 클릭하거나 "태스크 추가" 버튼을 누른 뒤 태스크명, 시작일, 기간을 입력하면  
**Then** 간트 바가 해당 날짜 범위에 즉시 렌더링되고, 입력한 종료일(시작일 + 기간 - 1)이 화면에 정확히 표시된다.

### Happy Path — 인라인 편집

**Given** 이미 생성된 태스크가 간트에 표시되어 있고  
**When** 태스크명 셀을 더블클릭하여 이름을 수정하고 Enter 또는 포커스 이탈로 확정하면  
**Then** 변경된 이름이 간트 바 레이블과 테이블 행 양쪽에 즉시 반영되고, DB에 저장된다.

### Happy Path — WBS 계층 구조

**Given** 태스크가 1개 이상 존재하고  
**When** 특정 태스크를 선택한 뒤 "하위 태스크 추가" 또는 들여쓰기(Indent) 액션을 실행하면  
**Then** 해당 태스크가 부모 태스크의 자식으로 표시되고, 부모 태스크 기간은 자식들의 범위를 포함하도록 자동 조정된다.

### Edge Case — 기간 0 또는 음수 입력

**Given** 태스크 편집 중  
**When** 기간에 0 또는 음수를 입력하면  
**Then** "기간은 1일 이상이어야 합니다" 메시지가 표시되고, 이전 유효값으로 복원된다.

### Edge Case — 간트 바 드래그로 날짜 변경

**Given** 태스크가 간트에 표시되어 있고  
**When** 간트 바를 드래그하여 다른 날짜로 이동하면  
**Then** 시작일/종료일이 드래그 결과에 맞게 즉시 갱신되고, 이 태스크에 연결된 후행 태스크들도 연쇄 재계산된다.

## AC-03: 의존성(링크) 생성 및 자동 스케줄링 전파

**관련 페르소나:** IT 팀 PM (2순위) — 이 기능이 MS Project 대비 차별점의 핵심

### Happy Path — FS 링크 전파

**Given** 태스크 A(3일), 태스크 B(2일), 태스크 C(4일)가 순서대로 A→B→C FS 의존성으로 연결되어 있고  
**When** 태스크 A의 기간을 3일에서 6일로 늘리면  
**Then** 태스크 B의 시작일이 A 종료 다음 날로 자동 이동하고, 태스크 C도 B의 새 종료일에 맞춰 연쇄 이동하여, 전체 chain이 사용자 개입 없이 재계산된다. 재계산은 3초 이내에 완료된다.

### Happy Path — Lag 포함 FS 링크

**Given** 태스크 A → 태스크 B로 FS, Lag = 2일 링크가 설정되어 있고  
**When** 태스크 A의 종료일이 변경되면  
**Then** 태스크 B의 시작일은 A 종료일 + 2일(Lag)로 계산되어 갱신된다.

### Happy Path — SS / FF / SF 링크

**Given** 태스크 A → 태스크 B로 SS(Start-to-Start) 링크가 설정되어 있고  
**When** 태스크 A의 시작일이 변경되면  
**Then** 태스크 B의 시작일이 태스크 A의 새 시작일과 동일하게(또는 Lag만큼 오프셋되어) 갱신된다.

### Happy Path — 대량 변경 전 영향 범위 확인 메시지

**Given** 태스크 A가 20개 이상의 후행 태스크를 가진 chain의 선두에 있고  
**When** 사용자가 태스크 A의 기간 또는 날짜를 변경하면  
**Then** "이 변경은 N개의 태스크에 영향을 줍니다. 계속하시겠습니까?" 확인 다이얼로그가 표시되고, [계속] 클릭 시 cascade 재계산이 실행되며, [취소] 클릭 시 변경이 롤백된다. 영향 태스크 수(N)는 다이얼로그에 정확히 표시된다.

### Edge Case — 영향 범위 1개 이하일 때 확인 생략

**Given** 변경으로 영향받는 후행 태스크가 없거나 1개뿐인 경우  
**When** 사용자가 태스크 날짜/기간을 변경하면  
**Then** 확인 다이얼로그 없이 즉시 재계산이 실행되어 사용자 경험을 방해하지 않는다.

### Edge Case — 순환 참조 시도

**Given** 태스크 A → 태스크 B FS 링크가 이미 존재하고  
**When** 태스크 B → 태스크 A로 링크를 추가하려 하면  
**Then** "순환 의존성이 감지되어 링크를 생성할 수 없습니다" 오류 메시지가 표시되고, 링크는 생성되지 않으며 기존 스케줄은 변경되지 않는다.

### Edge Case — 링크 삭제 후 재계산

**Given** A→B→C chain이 존재하고  
**When** A→B 링크를 삭제하면  
**Then** 태스크 B와 C는 의존성에서 해방되어 현재 날짜 위치를 유지하고, 더 이상 A의 변경에 영향받지 않는다.

### Edge Case — 다수 전행 태스크(Multiple Predecessors)

**Given** 태스크 C가 태스크 A와 태스크 B 모두에게 FS 의존성을 가지고  
**When** 태스크 A가 늦게 끝나도록 변경되어 A 종료일 > B 종료일이 되면  
**Then** 태스크 C의 시작일은 두 전행 중 더 늦은 종료일(A 기준)을 따라 결정된다.

## AC-04: 제약 조건 적용

**관련 페르소나:** IT 팀 PM (2순위)

### Happy Path — MSO (Must Start On)

**Given** 태스크 B에 MSO 제약이 2026-07-01로 설정되어 있고, 전행 A가 6월 말에 끝나도록 되어 있고  
**When** 전행 A의 기간이 줄어들어 A가 6월 20일에 종료되더라도  
**Then** 태스크 B의 시작일은 의존성 계산값이 아닌 MSO 날짜인 2026-07-01로 고정 유지된다.

### Happy Path — SNET (Start No Earlier Than)

**Given** 태스크 B에 SNET 2026-07-15가 설정되어 있고  
**When** 의존성 전파로 계산된 태스크 B의 시작일이 2026-07-10이면  
**Then** 태스크 B의 실제 시작일은 SNET인 2026-07-15로 조정된다. 반대로 계산값이 2026-07-20이라면 SNET을 초과했으므로 2026-07-20이 적용된다.

### Edge Case — 제약이 의존성과 충돌하여 불가능한 스케줄 생성

**Given** 태스크 B에 FNLT(Finish No Later Than) 2026-07-01이 설정되어 있고  
**When** 전행 A의 변경으로 의존성 계산 결과 태스크 B가 2026-07-10 이후에 시작해야 하면  
**Then** 태스크 B 해당 행 또는 간트 바에 "스케줄 충돌: 제약 조건을 만족할 수 없습니다" 경고 표시가 나타나고, 스케줄은 의존성 계산값으로 일단 표시되지만 경고는 사라지지 않는다.

### Edge Case — MSO 제약이 있는 태스크의 기간 리사이즈

**Given** MSO 2026-07-01이 설정된 태스크 B를 드래그로 리사이즈하면  
**Then** 시작일은 MSO로 고정된 채 기간(종료일)만 변경되고, 이 변경으로 인한 후행 태스크들도 연쇄 재계산된다.

## AC-05: Critical Path 표시

**관련 페르소나:** IT 팀 PM (2순위)

### Happy Path — CP 시각적 식별

**Given** 3개 이상의 태스크가 의존성으로 연결된 프로젝트가 존재하고  
**When** 사용자가 "Critical Path 표시" 토글을 활성화하면  
**Then** Total Float = 0인 태스크의 간트 바가 다른 색상(예: 빨간색)으로 강조 표시되고, 비 CP 태스크는 기본 색상을 유지하며, 사용자는 프로젝트 완료일에 직접 영향을 주는 경로를 한눈에 파악할 수 있다.

### Happy Path — CP 동적 갱신

**Given** CP 표시가 활성화된 상태에서  
**When** 사용자가 비 CP 태스크의 기간을 늘려 해당 태스크가 CP에 편입되는 변경을 하면  
**Then** 간트 바의 색상이 즉시 CP 색상으로 변경되고, 기존 CP 태스크 중 Float이 생긴 태스크는 기본 색상으로 복귀한다.

### Edge Case — 병렬 경로에서 동시 CP

**Given** A→C와 A→B→C 두 경로가 있고 두 경로 모두 Float = 0이면  
**Then** 두 경로의 모든 태스크가 동시에 CP 색상으로 표시된다.

### Happy Path — Total Float 값을 그리드 컬럼으로 표시

**Given** 의존성이 설정된 프로젝트에서 WBS 그리드가 표시되어 있고  
**When** 사용자가 컬럼 설정에서 "Total Float" 컬럼을 활성화하면  
**Then** 각 태스크 행에 Total Float 값이 일수(숫자) 형식으로 표시되고, CP 태스크는 0으로, 여유 있는 태스크는 양수로 표시된다. 값은 날짜 변경 시 자동 갱신된다.

### Edge Case — 단독 태스크(링크 없음)

**Given** 의존성이 전혀 없는 독립 태스크만 존재하면  
**Then** 해당 태스크는 Float = 0으로 계산되어 CP로 표시되거나, "의존성 없음" 상태로 CP 계산에서 제외되는 동작 중 하나로 일관성 있게 처리된다(두 가지 중 하나를 정책으로 결정 후 문서화).

## AC-06: 기준선(Baseline) 저장 및 비교

**관련 페르소나:** IT 팀 PM (2순위), 프리랜서 PM (1순위)
**구현 Phase:** Phase 2
**현재 상태:** 부분 구현 — `baselines.ts` CRUD(저장·목록·삭제) 완료, 기준선 비교 시각화(비교 뷰·기준선 바 표시) 미구현

### Happy Path — 기준선 저장

**Given** 프로젝트 스케줄이 확정된 시점에  
**When** 사용자가 "기준선 저장" → "Baseline 1"로 저장하면  
**Then** 현재 모든 태스크의 시작일/종료일/기간이 Baseline 1으로 스냅샷 저장되고, 이후 실제 스케줄을 변경해도 기준선 데이터는 변경되지 않는다.

### Happy Path — 기준선 비교 시각화

**Given** Baseline 1이 저장된 상태에서 일부 태스크 일정이 변경되었고  
**When** "기준선 표시" 토글을 켜면  
**Then** 간트 바 아래(또는 옆)에 기준선 바가 반투명 또는 다른 색으로 표시되어 계획 대비 현재 일정의 슬립(지연)을 시각적으로 확인할 수 있다.

### Edge Case — 최대 11개 기준선 초과 시도

**Given** 이미 11개의 기준선이 저장된 상태에서  
**When** 12번째 기준선을 저장하려 하면  
**Then** "기준선은 최대 11개까지 저장할 수 있습니다. 기존 기준선을 삭제 후 시도하세요" 메시지가 표시되고 저장은 실행되지 않는다.

### Edge Case — 기준선 저장 후 태스크 삭제

**Given** Baseline 1에 포함된 태스크 X가 이후 삭제되면  
**Then** 기준선 비교 화면에서 태스크 X는 "기준선에만 존재 (삭제됨)" 상태로 별도 표시되거나, 기준선 바만 남아 삭제된 태스크의 원래 계획을 추적할 수 있다.

## AC-07: 실시간 협업 동기화

**관련 페르소나:** IT 팀 PM (2순위)

### Happy Path — 크로스탭 실시간 반영

**Given** 동일 계정이 두 개의 브라우저 탭에서 같은 프로젝트를 열고 있고  
**When** 탭 A에서 태스크의 기간을 변경하면  
**Then** 탭 B에서 2초 이내에 해당 태스크의 간트 바가 새 기간으로 갱신되고, 탭 B 사용자는 별도 새로고침 없이 변경 사항을 인지할 수 있다.

### Happy Path — 멀티유저 동시 편집

**Given** 사용자 A(PM)와 사용자 B(팀원)가 각자의 계정으로 같은 프로젝트에 접속해 있고  
**When** 사용자 A가 태스크 1의 시작일을 변경하는 동시에 사용자 B가 태스크 2의 이름을 변경하면  
**Then** 각자의 변경이 서로에게 5초 이내에 반영되고, 두 변경은 서로 덮어쓰지 않는다.

### Edge Case — 자기 에코 방지

**Given** 사용자 A가 태스크를 변경하여 DB에 저장되고, Supabase Realtime이 변경을 브로드캐스트할 때  
**When** 변경을 발생시킨 사용자 A의 탭이 자신의 변경 이벤트를 수신하면  
**Then** 화면이 불필요하게 깜빡이거나 태스크가 원래 위치로 튀어 돌아가는 현상 없이 무시 처리된다.

### Edge Case — 동기화 지연

**Given** 네트워크 상태가 불안정하여 Realtime 이벤트가 지연되고  
**When** 지연 후 이벤트가 도착하면  
**Then** 현재 화면 상태와 비교하여 실제 변경된 경우에만 업데이트가 적용되고, 이미 최신인 경우는 무시되어 화면 깜빡임이 발생하지 않는다.

## AC-08: 충돌 해결 (낙관적 잠금)

**관련 페르소나:** IT 팀 PM (2순위)

### Happy Path — 순차 편집 (충돌 없음)

**Given** 사용자 A가 태스크를 편집하고 저장이 완료된 후  
**When** 사용자 B가 같은 태스크를 편집하면  
**Then** 사용자 B는 A의 최신 버전을 기반으로 편집하게 되고, 저장이 정상 처리된다.

### Edge Case — 동시 편집으로 버전 충돌

**Given** 사용자 A와 사용자 B가 동시에 같은 태스크(version = 5)를 열고  
**When** 사용자 A가 먼저 저장(version 5 → 6)한 뒤, 사용자 B가 구버전(version 5) 기반으로 저장을 시도하면  
**Then** 사용자 B의 화면에 "다른 사용자가 이미 이 태스크를 수정했습니다. 최신 내용을 확인 후 다시 저장하세요"라는 안내가 표시되고, B의 변경 내용은 임시로 보존되어 B가 직접 병합 여부를 결정할 수 있다. 데이터는 유실되지 않는다.

### Edge Case — 충돌 후 강제 덮어쓰기

**Given** 충돌 안내를 받은 사용자 B가  
**When** "내 변경으로 덮어쓰기" 옵션을 선택하면  
**Then** B의 변경이 최신 버전 번호와 함께 저장되고, 화면은 B의 내용을 표시한다. 사용자 A의 탭에는 실시간 동기화로 B의 최신 내용이 반영된다.

## AC-09: 인증 및 접근 제어

**관련 페르소나:** 프리랜서 PM (1순위), IT 팀 PM (2순위)

### Happy Path — 이메일 로그인

**Given** 가입된 이메일과 비밀번호를 가진 사용자가  
**When** 로그인 폼에 올바른 정보를 입력하고 제출하면  
**Then** 자신의 프로젝트 목록 대시보드로 이동하고, 타인의 프로젝트는 목록에 노출되지 않는다.

### Happy Path — 프로젝트 구성원 초대

**Given** 프로젝트 소유자가 특정 이메일 주소를 구성원으로 초대하면  
**When** 초대받은 사용자가 로그인 후 프로젝트 목록을 조회하면  
**Then** 해당 프로젝트가 목록에 나타나고 접근 권한이 부여된 기능(읽기/편집)을 사용할 수 있다.

### Error Case — 미초대 사용자 직접 URL 접근

**Given** 프로젝트에 접근 권한이 없는 사용자가  
**When** 해당 프로젝트의 URL을 직접 입력하면  
**Then** 프로젝트 내용은 표시되지 않고 "접근 권한이 없습니다" 메시지와 함께 대시보드 또는 로그인 화면으로 리다이렉트된다. DB 쿼리 수준(RLS)에서도 데이터가 반환되지 않아야 한다.

### Error Case — 만료된 세션

**Given** 로그인된 사용자의 세션이 만료된 상태에서  
**When** 프로젝트 데이터를 저장하거나 조회하려 하면  
**Then** 자동으로 로그인 페이지로 이동하고, 이동 전 작업 중이던 변경 사항은 가능한 범위 내에서 로컬에 임시 보존되거나, "세션이 만료되었습니다. 다시 로그인해 주세요" 안내가 명확하게 표시된다.

## AC-10: 오프라인 → 온라인 복귀

**관련 페르소나:** 프리랜서 PM (1순위)

### Happy Path — 짧은 오프라인 후 복귀

**Given** 사용자가 프로젝트를 편집하던 중 네트워크가 일시적으로 끊기고  
**When** 30초 이내에 네트워크가 복구되면

### Happy Path — 연결 상태 표시

**Given** 네트워크가 끊긴 상태에서  
**When** 사용자가 화면을 보면  
**Then** UI 상단 또는 눈에 띄는 위치에 "오프라인 상태 — 변경 사항이 저장되지 않을 수 있습니다" 배너 또는 인디케이터가 표시되어 사용자가 현재 상태를 인지할 수 있다.

### Edge Case — 긴 오프라인 후 복귀 시 충돌

**Given** 사용자가 오프라인 상태에서 태스크 X를 수정하는 동안, 다른 사용자가 온라인에서 같은 태스크 X를 이미 수정하고 저장했고  
**When** 첫 번째 사용자가 온라인으로 복귀하여 변경을 반영하려 하면  
**Then** AC-08의 충돌 해결 흐름과 동일하게 버전 충돌 안내가 표시되고, 사용자는 어느 버전을 적용할지 선택할 수 있으며 오프라인 중 수정 내용은 유실되지 않는다.

### Error Case — 오프라인 중 대용량 변경 후 재연결 실패

**Given** 네트워크 재연결이 반복적으로 실패하고  
**When** 사용자가 페이지를 새로고침하면  
**Then** 오프라인 중 변경 내용은 브라우저 로컬 스토리지 또는 IndexedDB에 임시 보존되어 있어 재연결 성공 후 복원 시도가 가능하다. (단, 현재 미구현이면 "네트워크 오류로 일부 변경이 저장되지 않았습니다" 명확한 경고를 표시하는 것이 최소 요건)

## AC-11: Undo/Redo (S-01)

**관련 페르소나:** 프리랜서 PM (1순위), IT 팀 PM (2순위)
**구현 Phase:** Phase 2

### Happy Path — 태스크 이동 Undo (cascade 포함)

**Given** 사용자가 태스크 A를 드래그하여 다른 날짜로 이동하였고, 이로 인해 후행 태스크 B·C가 연쇄 재계산된 상태에서  
**When** Ctrl+Z(macOS: Cmd+Z)를 누르면  
**Then** 태스크 A가 이동 이전의 원래 날짜로 복원되고, 연쇄 재계산된 태스크 B·C도 Undo 이전 날짜로 함께 복원되며, 간트 바와 DB 양쪽에 즉시 반영된다.

### Happy Path — Undo 후 Redo

**Given** 태스크 이동을 Undo하여 원래 위치로 복원된 상태에서  
**When** Ctrl+Y(macOS: Cmd+Shift+Z)를 누르면  
**Then** Undo 이전의 상태(이동 후 날짜)로 다시 전환되고, 연쇄 재계산 결과도 함께 복원된다.

### Happy Path — 링크 생성 Undo

**Given** 사용자가 태스크 A → 태스크 B로 FS 링크를 새로 생성하여, 태스크 B의 시작일이 자동 재계산된 상태에서  
**When** Ctrl+Z를 누르면  
**Then** 생성된 링크가 삭제되고, 태스크 B의 시작일은 링크 생성 이전 날짜로 복원되며, 간트 바에서 의존성 화살표도 즉시 사라진다.

### Edge Case — Undo 스택 한계 초과

**Given** 사용자가 50회 연속으로 작업을 수행하여 Undo 스택이 최대 한도(50회)에 도달하였고, Ctrl+Z를 추가로 누른 상태에서  
**When** 스택이 이미 비어 있으면  
**Then** 더 이상 Undo가 실행되지 않고, UI에 "더 이상 되돌릴 수 없습니다" 토스트 또는 Undo 버튼 비활성화로 한계를 명확히 표시한다. 화면 상태는 변경되지 않는다.

### Edge Case — Undo 후 새 액션으로 Redo 스택 초기화

**Given** 사용자가 태스크를 이동하고 Ctrl+Z로 Undo한 상태에서(Redo 스택에 1개 항목이 존재)  
**When** 새로운 태스크 기간 변경 액션을 실행하면  
**Then** Redo 스택이 완전히 초기화되어, 이후 Ctrl+Y를 눌러도 아무 동작이 일어나지 않으며 Redo 버튼이 비활성화 상태로 표시된다.

### Edge Case — DB 저장 실패 시 Undo 스택 미등재

**Given** 사용자가 태스크를 편집하였으나 네트워크 오류로 DB 저장이 실패한 상태에서  
**When** Ctrl+Z를 누르면  
**Then** 저장에 실패한 미완성 액션은 Undo 스택에 등재되어 있지 않으므로 Undo 대상이 되지 않으며, 실패 이전의 마지막 성공 저장 상태 기준으로만 Undo가 동작한다.

## AC-12: 태스크 완료율(%) 입력 및 시각화 (S-02)

**관련 페르소나:** 프리랜서 PM (1순위)
**구현 Phase:** Phase 2

### Happy Path — 진행률 슬라이더 조작 및 간트 바 채움 표시

**Given** 태스크 상세 패널 또는 인라인 편집 UI가 열려 있고  
**When** 사용자가 진행률 슬라이더를 드래그하거나 직접 숫자를 입력하여 완료율을 변경하면  
**Then** 간트 바 내부에 입력된 비율만큼 채움(fill) 표시가 즉시 갱신되고(예: 40%이면 바 길이의 40%를 다른 색으로 채움), 변경값이 DB에 저장된다.

**Given** 태스크의 진행률이 100% 미만인 상태에서

### Edge Case — 요약(Summary) 태스크의 진행률 자동 계산

**Given** 부모 요약 태스크 P 아래에 자식 태스크 A(기간 5일, 40%), B(기간 5일, 80%)가 존재할 때  
**When** 자식 태스크의 진행률 또는 기간이 변경되면  
**Then** 요약 태스크 P의 진행률은 자식 태스크의 기간 가중 평균으로 자동 재계산되어 표시되고(예: (5×40 + 5×80) / 10 = 60%), 사용자가 요약 태스크의 진행률을 직접 편집할 수 없도록 해당 입력 필드가 읽기 전용으로 비활성화된다.

## AC-13: 클라이언트 공유 URL (읽기 전용 링크)

**관련 페르소나:** 프리랜서 PM (1순위) — 핵심 기대
**구현 Phase:** Phase 2 (현재 미구현 — 선행 조건: `shared_links` 테이블 Week 2)

### Happy Path — 공유 링크 생성 및 비로그인 열람

**Given** 인증된 PM이 프로젝트 설정에서 "공유 링크 생성" 버튼을 클릭하면  
**When** 고유한 공유 토큰이 포함된 URL(예: `https://app.zeropm.io/share/{token}`)이 생성되고  
**Then** 로그인하지 않은 사용자가 해당 URL로 접속하더라도 해당 프로젝트의 간트 차트를 정상적으로 열람할 수 있으며, 로그인 화면으로 리다이렉트되지 않는다.

### Happy Path — 읽기 전용 접근 (편집 불가)

**Given** 공유 링크로 접속한 비로그인 또는 권한 없는 사용자가 간트 화면을 열람하는 상태에서  
**When** 태스크 편집, 드래그 이동, 링크 추가, 태스크 추가 등의 편집 동작을 시도하면  
**Then** 모든 편집 버튼과 인터랙티브 요소가 비활성화(disabled) 또는 숨김 처리되어 있어 어떠한 변경도 불가능하고, "읽기 전용 모드입니다" 안내가 명확히 표시된다.

### Happy Path — 프로젝트 업데이트 즉시 반영

**Given** PM이 프로젝트의 태스크 일정을 변경하고 저장한 상태에서  
**When** 공유 링크 수신자가 해당 URL에 재접속하거나 페이지를 새로고침하면  
**Then** PM이 변경한 최신 일정이 즉시 반영된 간트 화면이 표시되며, 별도의 링크 재발급 없이 항상 최신 데이터를 기반으로 열람할 수 있다.

### Edge Case — 공유 링크 비활성화

**Given** PM이 프로젝트 설정에서 기존에 발급한 공유 링크를 비활성화(또는 삭제)한 상태에서  
**When** 공유 링크 수신자가 해당 URL로 접속하면  
**Then** 간트 데이터는 표시되지 않고 "이 링크는 만료되었습니다. PM에게 새 링크를 요청하세요" 메시지가 표시된다. DB에서도 비활성화된 토큰에 대한 데이터 응답이 차단되어야 한다.

### Error Case — RLS 수준 읽기 전용 보장

**Given** 공유 토큰을 통해 인증 없이 접속하는 경우  
**When** 해당 요청이 Supabase DB에 도달하면  
**Then** RLS 정책에 의해 읽기(SELECT)만 허용되고, 삽입(INSERT)·수정(UPDATE)·삭제(DELETE)는 DB 수준에서 거부되어 클라이언트 우회 시도로도 데이터 변경이 불가능하다.

## AC-15: 프로젝트 대시보드 요약 뷰 (S-03)

**관련 페르소나:** 프리랜서 PM (1순위)

### Happy Path — 전체 프로젝트 요약 표시

**Given** 인증된 사용자가 보유한 프로젝트가 1개 이상 존재하는 상태에서  
**When** 사용자가 대시보드 화면에 접속하면  
**Then** 각 프로젝트의 이름, 전체 진행률(%), 마감일, 현재 상태(정상/지연/완료)가 카드 또는 목록 형태로 표시되고, 사용자는 개별 프로젝트로의 진입 링크를 통해 간트 화면으로 즉시 이동할 수 있다.

### Happy Path — 지연 프로젝트 강조 표시

**Given** 현재 날짜(Today)가 특정 프로젝트의 마감일을 초과하였고 해당 프로젝트의 진행률이 100% 미만인 상태에서  
**When** 사용자가 대시보드를 열람하면  
**Then** 해당 프로젝트 카드에 "지연" 배지 또는 경고 색상(예: 빨간 테두리)이 표시되어 다른 프로젝트와 시각적으로 즉시 구분되고, 지연 일수가 함께 표시된다.

### Edge Case — 프로젝트가 0개인 빈 상태

**Given** 인증된 사용자가 아직 프로젝트를 하나도 생성하지 않은 상태에서  
**When** 대시보드에 접속하면  
**Then** 빈 상태(Empty State) 일러스트 또는 안내 문구와 함께 "첫 프로젝트를 만들어 시작하세요" CTA 버튼이 표시되고, 버튼 클릭 시 프로젝트 생성 흐름으로 즉시 진입할 수 있다.

### Edge Case — 내보내기 범위 선택 옵션

**When** 내보내기 옵션 다이얼로그가 표시되면  
**Then** "현재 화면 기준(적용된 필터·줌 레벨)" 또는 "전체 프로젝트 기준(필터 무시, 전체 타임라인)"을 선택할 수 있으며, 선택에 따라 PDF에 포함되는 태스크 범위와 타임라인 스케일이 달라진다.

## AC-17: 팀원 초대 및 권한 설정 (M-07)

**관련 페르소나:** IT 팀 PM (1순위), 프리랜서 PM (클라이언트 Viewer 공유)  
**구현 Phase:** Phase 2 (현재 미구현 — 선행 조건: RBAC 통일 + `project_invitations` 테이블 Week 2)  
**관련 Flow:** Flow 7

### Happy Path — 이메일 초대 발송 및 수락

**Given** 프로젝트 Owner 또는 Editor 권한을 가진 사용자가 멤버 관리 화면을 열고  
**When** 유효한 이메일 주소와 역할(Viewer / Editor / Owner)을 선택하여 [초대 보내기]를 클릭하면  
**Then** 초대 이메일이 발송되고, 멤버 목록에 해당 이메일이 "초대 대기 중" 상태로 즉시 표시되며, 수신자가 링크를 클릭하면 프로젝트에 선택된 역할로 자동 진입한다.

### Happy Path — 역할별 권한 즉시 적용

**Given** 팀원이 초대를 수락하여 프로젝트에 진입한 상태에서  
**When** Viewer 역할로 참여한 팀원이 태스크 편집·드래그·링크 추가를 시도하면  
**Then** 모든 편집 인터랙션이 차단되고(간트 바 드래그 불가, 버튼 비활성화, "읽기 전용 모드" 안내 표시), DB 수준에서도 해당 사용자의 INSERT/UPDATE/DELETE가 RLS에 의해 거부된다.

**Given** Editor 역할로 참여한 팀원이 프로젝트에 진입했을 때  
**When** 태스크 편집, 링크 생성, 기간 변경 등의 편집 액션을 수행하면  
**Then** 모든 편집 기능이 정상 동작하고, 프로젝트 설정(멤버 관리, 프로젝트 삭제) 메뉴는 접근 불가 또는 비표시 상태이다.

### Happy Path — 멤버 제거 및 역할 변경

**Given** Owner 권한 사용자가 멤버 목록에서 특정 팀원의 역할을 변경하거나 제거하면  
**When** 변경이 저장되면  
**Then** 해당 팀원의 다음 요청부터 새 권한이 즉시 적용되고, 제거된 경우 프로젝트 데이터에 더 이상 접근할 수 없다.

### Edge Case — 이미 가입된 이메일로 중복 초대

**Given** 이미 해당 프로젝트의 멤버인 이메일로 초대를 시도할 때  
**When** [초대 보내기]를 클릭하면  
**Then** "이미 프로젝트 멤버입니다" 인라인 오류가 표시되고 초대 이메일은 발송되지 않는다.

### Edge Case — 초대 링크 만료 (7일)

**Given** 초대 이메일이 발송된 지 7일이 초과된 상태에서  
**When** 수신자가 초대 링크를 클릭하면  
**Then** "초대 링크가 만료됐습니다. 초대자에게 재발송을 요청하세요" 안내가 표시되고 프로젝트에 진입되지 않는다.

### Edge Case — 본인 탈퇴 시도 (Owner)

**Given** 프로젝트의 유일한 Owner가 본인을 제거하려 할 때  
**When** [제거] 또는 [탈퇴]를 클릭하면  
**Then** "프로젝트 소유자는 직접 탈퇴할 수 없습니다. 다른 멤버에게 Owner를 이전한 후 탈퇴하세요" 오류가 표시되고 제거가 차단된다.

### Error Case — RLS 수준 권한 강제

**Given** 클라이언트가 API를 직접 호출하거나 인증 토큰을 변조하여 Viewer 권한으로 쓰기 요청을 시도할 때  
**When** 요청이 Supabase에 도달하면  
**Then** RLS 정책에 의해 DB 수준에서 즉시 거부(403 또는 empty result)되며, 어떠한 데이터 변경도 발생하지 않는다.

## AC-18: 프로젝트 템플릿 저장 및 적용 (S-05)

**When** 해당 프로젝트에서 만든 템플릿을 새 프로젝트에 적용하면

### Edge Case — 빈 프로젝트에서 템플릿 저장 시도

**Given** 태스크가 0개인 프로젝트에서 [템플릿으로 저장]을 시도하면  
**When** 저장 버튼을 클릭하면  
**Then** "태스크가 없는 프로젝트는 템플릿으로 저장할 수 없습니다" 인라인 오류가 표시되고 저장되지 않는다.

**관련 Flow:** Flow 10

### Happy Path — CSV 내보내기

**Given** 프로젝트에 태스크가 1개 이상 존재하는 상태에서

**Given** ZeroPM에서 내보낸 CSV 또는 동일 컬럼 구조를 가진 CSV 파일이 있을 때  
**When** [가져오기 > CSV에서 가져오기]로 파일을 선택하고 미리보기를 확인한 후 [가져오기]를 클릭하면

### Edge Case — 일부 행 파싱 실패

**Given** CSV의 일부 행에 날짜 형식 오류 또는 의존성 ID 불일치가 있을 때  
**When** 가져오기를 실행하면  
**Then** 유효한 행은 정상 가져오고, 실패한 행 목록("3행: 날짜 형식 오류", "7행: 참조 태스크 ID 없음")을 가져오기 완료 후 안내창으로 표시하며, 사용자가 해당 내용을 확인하고 닫을 수 있다.

### Error Case — 필수 컬럼 누락

**Given** `name` 또는 `start_date` 컬럼이 없는 CSV 파일을 선택했을 때  
**When** 파일을 선택하면  
**Then** 미리보기 단계에서 "필수 컬럼(name, start_date)이 없습니다. CSV 형식을 확인해 주세요" 오류가 표시되고 [가져오기] 버튼이 비활성화된다.

### Error Case — 인코딩 오류 (EUC-KR 등)

**Given** EUC-KR 등 UTF-8이 아닌 인코딩 파일을 선택했을 때  
**When** 파일을 선택하면  
**Then** "UTF-8 인코딩 파일만 지원합니다. 파일을 UTF-8로 저장한 뒤 다시 시도해 주세요" 오류가 표시되고 가져오기가 차단된다.

## AC-20: 제한적 공유 링크 (특정 섹션/태스크만 열람)

**관련 페르소나:** IT 팀 PM (1순위 — 외부 파트너 접근 제어)

**Given** 프로젝트에 상위 요약 태스크(섹션) 구조가 존재하고 Owner/Editor 권한 사용자가 공유 설정을 열었을 때  
**When** "공유 범위 제한" 옵션을 선택하고 공유할 섹션(요약 태스크)을 체크박스로 지정한 뒤 [링크 생성]을 클릭하면  
**Then** 선택된 섹션과 그 하위 태스크만 표시되는 고유 URL이 생성되고, 수신자가 해당 URL로 접속 시 지정되지 않은 섹션은 보이지 않는다.

### Happy Path — 제한된 링크 수신자 열람

**Given** 섹션 A와 B 중 섹션 A만 공유하는 링크를 발급한 상태에서

**Then** 간트 화면에 섹션 A의 태스크만 표시되고, 섹션 B와 프로젝트 전체 통계는 표시되지 않으며, 읽기 전용 모드로 동작한다.

### Edge Case — 공유된 섹션 삭제 후 링크 접속

**Given** 섹션 A를 공유하는 링크가 활성화된 상태에서 섹션 A가 삭제된 경우  
**When** 수신자가 링크로 접속하면  
**Then** "공유된 데이터가 삭제됐습니다. PM에게 문의하세요" 안내 메시지가 표시되고 간트 데이터는 노출되지 않는다.

### Error Case — RLS 수준 범위 강제

**Given** 제한적 공유 링크의 토큰으로 DB에 직접 쿼리를 시도하는 경우  
**When** 허용되지 않은 섹션의 태스크에 대한 SELECT 요청이 도달하면  
**Then** RLS 정책에 의해 해당 row는 반환되지 않으며 클라이언트는 해당 데이터의 존재를 알 수 없다.

## AC-21: 스프레드시트 붙여넣기로 태스크 일괄 입력

**관련 페르소나:** IT 팀 PM (1순위 — 대규모 WBS 빠른 입력), 프리랜서 PM (2순위)

**When** 간트 WBS 그리드의 빈 셀에 포커스를 두고 Ctrl+V(macOS: Cmd+V)를 누르면  
**Then** 클립보드의 각 행이 새 태스크로 파싱되어 순서대로 WBS에 추가되고, 추가된 태스크 수를 "N개 태스크가 추가됐습니다" 토스트로 알려준다.

### Happy Path — 이름만 있는 텍스트 붙여넣기

**Given** 태스크명만 여러 줄로 복사한 상태에서(날짜·기간 없음)  
**When** WBS 그리드에 붙여넣기를 실행하면  
**Then** 각 행이 이름만 있는 태스크로 생성되고, 날짜는 프로젝트 시작일을 기준으로 기본값(기간 1일)이 자동 적용된다.

### Edge Case — 붙여넣기 도중 일부 행 파싱 실패

**Given** 붙여넣기 데이터의 일부 행에 날짜 형식 오류가 있는 경우  
**When** 붙여넣기를 실행하면  
**Then** 유효한 행은 정상 추가되고, 파싱 실패한 행은 건너뛰어진다. 완료 후 "3개 중 1개 행을 건너뜠습니다. [상세 보기]" 안내가 표시된다.

### Edge Case — 최대 행 수 초과

**Given** 300행 이상의 데이터를 붙여넣기 시도하면  
**When** 붙여넣기를 실행하면  
**Then** "한 번에 최대 200개 태스크까지 추가할 수 있습니다. 처음 200개만 추가됐습니다" 안내가 표시되고 나머지는 무시된다.
