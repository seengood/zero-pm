# ZeroPM 데이터 설계

ZeroPM의 상세 기능 목록을 구현하기 위해 **Supabase (PostgreSQL 기반)**의 데이터베이스 스키마와 **Vercel** 환경에서의 데이터 활용 및 역할 분담을 중심으로 설계했습니다.

ZeroPM은 **MS Project급의 정밀한 스케줄링(CPM/RCPSP)**, **실시간 협업(CRDT/낙관적 잠금)**, 그리고 **Zero-Cost 운영**을 목표로 하므로, 데이터베이스 설계는 관계형 데이터의 무결성(Integrity)과 계층 구조 처리(Hierarchical Data)에 중점을 두어야 합니다.

## 📊 Supabase (PostgreSQL) 데이터 설계

ZeroPM 아키텍처는 PostgreSQL의 **관계형 구조**와 **재귀 CTE**를 활용하는 **인접 리스트(Adjacency List) 모델**이 간트 차트 환경에서 쓰기 성능과 구현 난이도의 균형이 가장 우수하다고 판단했습니다.

### 1. 사용자 및 권한 관리 (Authentication & RLS Core)

사용자 인증은 Supabase Auth를 활용하며, 데이터 접근 제어는 RLS(Row Level Security)를 통해 구현됩니다.

| 테이블 (Table) | 역할 | 핵심 필드 (Columns) | 설명 / RLS 전략 |
| :--- | :--- | :--- | :--- |
| **profiles** | 사용자 기본 정보 | `id` (UUID, PK, `auth.users.id` 참조), `username`, `role` (TEXT, `admin` 또는 `member`) | Supabase Auth와 연동. `role` 필드는 **RLS 정책** 내에서 **RBAC(역할 기반 접근 제어)**를 구현하는 데 사용됩니다. |
| **projects** | 프로젝트 최상위 컨테이너 | `id` (UUID, PK), `title`, `owner_id` (UUID, `auth.users.id` 참조), `calendar_settings` (JSONB) | `calendar_settings`에는 주말 및 사용자 정의 공휴일 목록 등 CPM 엔진에 필요한 **비즈니스 캘린더 정보**가 저장됩니다. |
| **project_members** | 멤버십 관리 | `project_id` (FK), `user_id` (FK), `role` (TEXT, `admin`, `planner`, `viewer`) | **다대다(N:M) 관계** 관리. RLS 정책은 이 테이블을 조회하여 해당 `project_id`에 `user_id`가 포함된 경우에만 데이터 접근을 허용합니다. |

### 2. 핵심 프로젝트 데이터 (WBS, Scheduling & Collaboration)

#### A. Task (작업) 테이블

Task 테이블은 **WBS 계층 구조**와 **CPM 계산 결과**를 모두 포함하는 핵심 테이블입니다.

| 필드 (Column) | 데이터 유형 (Type) | 역할 및 근거 |
| :--- | :--- | :--- |
| **id** | UUID (PK) | 작업의 고유 식별자. |
| **project_id** | UUID (FK) | WBS 및 RLS의 기본 범위. |
| **parent_id** | UUID (FK, `tasks.id` 참조) | **WBS 계층 구조** 구현 (인접 리스트 모델). |
| **sort_order** | INTEGER | 같은 `parent_id` 내에서 Task의 순서를 보장. |
| **text** | TEXT | Task의 이름. |
| **type** | TEXT | `task`, `milestone`, `summary` 등. SVAR Gantt의 **Custom Task Type**을 지원합니다. |
| **version** | INTEGER/TIMESTAMP | **낙관적 잠금(Optimistic Locking)** 구현에 필수적입니다. CRDT가 담당하지 않는 구조적 변경 시 충돌을 감지하는 데 사용됩니다. |
| **start_date** | TIMESTAMPTZ | 실제 시작일 또는 현재 계획된 시작일. |
| **duration** | INTEGER | 작업 기간 (분/시간 단위로 저장 권장). |
| **progress** | FLOAT (0.0~1.0) | 완료율 (% Complete). |
| **ES** (Early Start) | TIMESTAMPTZ | **CPM 계산 결과** (가장 빠른 시작일). |
| **LF** (Late Finish) | TIMESTAMPTZ | **CPM 계산 결과** (가장 늦은 완료일). |
| **total_float** | INTEGER | 총 여유 시간. 0이면 **임계 경로(Critical Path)** 작업. |
| **is_critical** | BOOLEAN | 임계 경로 여부 (시각화용). |
| **constraint_type** | TEXT | ASAP, ALAP, MSO 등 CPM 제약 조건. |
| **constraint_date** | TIMESTAMPTZ | 제약 조건이 적용되는 날짜. |

#### B. Links (의존성) 테이블

Task 간의 관계(그래프 엣지)를 저장하는 **정규화된 연결 테이블(Junction Table)**을 사용합니다. 이는 `graphlib`를 활용한 CPM 계산의 입력 데이터가 됩니다.

| 필드 (Column) | 데이터 유형 (Type) | 역할 |
| :--- | :--- | :--- |
| **id** | UUID (PK) | 의존성 고유 ID. |
| **project_id** | UUID (FK) | RLS 및 프로젝트 범위 지정. |
| **source** | UUID (FK, `tasks.id` 참조) | 선행 작업 (`predecessor_id`). |
| **target** | UUID (FK, `tasks.id` 참조) | 후행 작업 (`successor_id`). |
| **type** | TEXT | 의존성 유형: '0'(FS), '1'(SS), '2'(FF), '3'(SF). |
| **lag** | INTEGER | 시간적 지연 (Lag) 또는 중첩 (Lead, 음수). |

### 3. 자원 및 비용 관리 데이터 (Resource & Cost)

ZeroPM은 RCPSP 충돌 감지 및 비용 추적 기능을 지원하므로, 자원 및 비용 관리를 위한 테이블이 필요합니다.

| 테이블 (Table) | 역할 | 핵심 필드 (Columns) | 설명 |
| :--- | :--- | :--- | :--- |
| **resources** | 자원 유형 정의 | `id` (UUID, PK), `project_id` (FK), `name`, `type` (TEXT, 인력/장비), **`capacity`** (FLOAT) | 자원의 최대 가용 용량($R_k$) 저장. RCPSP 충돌 감지에 사용됩니다. |
| **assignments** | 작업-자원 할당 | `id` (UUID, PK), `task_id` (FK), `resource_id` (FK), **`required_units`** (FLOAT, 할당 단위) | Task가 요구하는 자원의 양($r_{jk}$) 저장. |
| **costs** | 비용 추적 | `id` (UUID, PK), `task_id` (FK), `type` (TEXT, 고정비/시간당), **`amount`** (NUMERIC), `rate` (NUMERIC) | 비용 분석 및 비용 기준선(Cost Baseline) 계산을 위한 입력 데이터. |

### 4. 기준선 관리 데이터 (Baseline & Interim Plan)

프로젝트 추적을 위한 **11개의 기준선**과 **10개의 중간 계획** 스냅샷을 저장해야 합니다.

| 테이블 (Table) | 역할 | 핵심 필드 (Columns) | 설명 |
| :--- | :--- | :--- | :--- |
| **baselines** | 기준선 정의 | `id` (UUID, PK), `project_id` (FK), **`baseline_num`** (INTEGER, 0~10), `created_at` | 기준선 데이터셋의 메타 정보. |
| **baseline_tasks** | 기준선 Task 스냅샷 | `baseline_id` (FK), `task_id` (UUID), `snapshot_start_date` (TIMESTAMPTZ), `snapshot_finish_date` (TIMESTAMPTZ), `snapshot_duration`, `snapshot_cost` | Task의 **20개 주요 참조점** 중 일부를 스냅샷으로 저장하여 현재 계획과의 편차 분석에 사용. |
| **interim_plans** | 중간 계획 스냅샷 | `id` (UUID, PK), `project_id` (FK), **`plan_num`** (INTEGER, 1~10), `task_id` (FK), **`current_start_date`**, **`current_finish_date`** | Task의 현재 시작일과 종료일만 저장하여 Baseline 대비 진행 상황을 평가. |

### 5. 협업 상태 관리 (Realtime Presence & Sync)

Supabase Realtime과 Yjs를 활용한 협업 상태 관리는 주로 클라이언트 메모리 내에서 이루어지지만, 영속성 및 실시간 정보 교환을 위해 다음과 같은 Supabase 기능을 활용합니다.

| 기능 (Feature) | 역할 | 설명 및 근거 |
| :--- | :--- | :--- |
| **Supabase Presence** | **소프트 락 및 현재 사용자 인식** | 사용자가 현재 편집 중인 Task ID를 전파하여, 다른 클라이언트에서 해당 Task에 대한 **"소프트 락" UI**를 표시하게 합니다. Free Plan은 **200개의 동시 연결**을 지원합니다. |
| **Realtime Broadcast** | **CRDT 델타 교환** | Yjs의 CRDT 업데이트 델타(Delta) 및 커서 위치와 같은 임시 상태를 연결된 클라이언트 간에 **서버 부하 없이** 빠르게 교환하는 데 사용됩니다. |
| **Yjs Persistence Table** | **CRDT 영속성** | (선택 사항) CRDT 문서의 최종 상태나 업데이트 벡터를 `bytea` 타입의 컬럼에 저장하여 오프라인 작업 후 복구 및 영구 저장을 보장합니다. |

## 🌐 Vercel 환경에서의 데이터 활용 및 역할

Vercel은 Next.js 기반의 프론트엔드 애플리케이션을 호스팅하고, 필요한 경우 **Serverless Functions (Edge Functions)**를 통해 백엔드 로직을 처리하는 역할을 합니다.

| Vercel 구성 요소 | 역할 | 데이터 흐름 및 활용 |
| :--- | :--- | :--- |
| **Front-end Hosting (Next.js)** | **UI 및 스케줄링 엔진 실행** | **PostgreSQL/Supabase**에서 Task, Link, Resource 데이터를 읽어와 클라이언트 메모리에 로드합니다. **Web Worker**를 활용하여 로드된 데이터(DAG)에 대해 CPM 및 RCPSP 계산을 수행합니다. |
| **Client-side Storage** | **Optimistic UI 및 오프라인 지원** | 사용자 변경 사항을 즉시 **RxDB/IndexedDB**와 같은 로컬 저장소에 적용하고, **Yjs** CRDT 객체에 반영하여 UI에 즉각적인 피드백을 제공합니다. |
| **Vercel Serverless Functions** | **복잡한 트랜잭션 및 RPC 처리** | 클라이언트에서 직접 수행하기 어려운 **원자적(Atomic) 데이터 쓰기** (예: CPM 계산 결과 일괄 저장, Baseline 설정, 복잡한 RBAC 검증이 필요한 작업) 시, Supabase의 **RPC(Remote Procedure Call)** 기능을 호출하는 엔드포인트로 활용됩니다. 이는 Postgres 함수 내에서 **트랜잭션**을 보장합니다. |
| **Zero-Cost 운영** | **Vercel Hobby Plan** | Vercel의 **Hobby Plan**은 월 6,000 빌드 실행 시간, 100만 Edge 요청 등 충분한 무료 자원을 제공하여, 개인 및 소규모 팀의 Zero-Cost 운영 목표를 지원합니다. |

### 데이터 흐름 요약 비유

ZeroPM의 데이터 아키텍처는 마치 **"클라이언트 측 워크스테이션을 갖춘 중앙 도서관"**과 같습니다.

1.  **Postgres/Supabase (도서관 중앙 서버):** 프로젝트의 모든 영구적인 기록(Task, Dependency, Baseline)을 **무결성(Integrity)**을 보장하며 저장합니다. **RLS**는 누가 어떤 책(Row)을 볼 수 있는지 결정하는 출입 통제 역할을 합니다.
2.  **클라이언트 (개인 워크스테이션):** 일단 프로젝트 데이터(책)를 가져오면, **Web Worker**라는 CPU를 사용하여 **CPM/RCPSP 계산(복잡한 분석 작업)**을 중앙 서버의 부하 없이 빠르게 수행합니다.
3.  **Yjs/Presence (실시간 메모장):** 여러 사용자가 같은 Task를 편집할 때, **Yjs**는 충돌 없이 내용을 병합하며, **Presence**는 누가 현재 그 페이지를 읽고 있는지 알려주는 "소프트 락" 푯말 역할을 합니다.
4.  **Vercel Functions (도서관 사서):** 복잡하고 원자적이어야 하는 작업(예: Baseline 스냅샷 저장)이 필요할 때, 사서(Serverless Function)가 나서서 **트랜잭션**을 보장하며 안전하게 중앙 서버에 기록합니다.
